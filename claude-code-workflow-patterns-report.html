从 Subagents 到Workflow：多 Agent协作到底在设计什么？

这份报告基于原始视频截图材料，提炼多 Agent 协作的关键结构：三种协作范式、六种 workflow 编排模式，以及背后的工程设计取舍。

状态：分享版初稿 生成日期：2026-05-25 可信度：中等 口径：官方文档 + 截图转译

开场结论

Agent 是执行者，Workflow 是协作协议。

这组材料真正有价值的地方，不是强调“多开几个 Agent”，而是在讨论一个更工程化的问题：当多个 Agent 同时工作时，谁负责拆任务？谁负责并行？谁负责验证？谁负责把结果收拢成可交付产物？

从“让一个智能体临场发挥”，升级为“用可声明、可复跑、可观察的协作图来组织智能体”。

阅读边界

这份报告把原始材料当作“设计思路素材”来分析，不把所有画面文案都视作已经稳定发布的官方功能。根据当前能确认的 Claude Code 文档，Subagents、Agent Teams、Agent loop、Ultraplan 都有官方材料；更脚本化的 Workflow 形态，更适合先理解为一种实验方向或设计范式。

换句话说：重点是理解这套协作架构，而不是立即断言某个命令、环境变量或文件格式已经是稳定接口。

基础框架：三种协作范式

第一组材料的核心是三张卡片：Subagents、Agent Teams、Workflow。它们的差异不在“谁更聪明”，而在控制权、通信方式和复跑能力。

SUBAGENTS · 信使
临时派遣，
结果回汇

主 agent 临时 fork 出子 agent 去办单件事。编排逻辑藏在自然语言 prompt 里。

启动成本低、自然、灵活。但多个 subagent 之间没有强契约：输出格式、调用顺序、聚合方式基本靠主 agent 当场决定，难以复跑。

AGENT TEAMS · 工作室
长期共事，
人在中间调度

多个角色在并行会话里协作，例如 Planner / Reviewer / Implementer / Docs。

适合人和多个 agent 共同坐班：操作者可以在不同会话里观察每位 agent 各自推进。控制感强，但难以一键复跑，更像 IDE + 多个长期 chat。

WORKFLOW · 流水线
脚本编排，
阶段 · 并行 · schema

用一段 JavaScript 显式声明阶段、fan-out、聚合与结构化输出契约。

每个 agent() 都有 schema、phase、label：能复跑、能追踪、能做质量门禁。代价是要写脚本，也要审脚本。

三种范式怎么选？

范式 控制权在哪里 适合什么 主要代价
Subagents
信使 / scout		 主 Agent 的自然语言调度 一次性调研、读大量上下文、代码搜索、日志分析 协作契约弱，输出和聚合容易漂移
Agent Teams
工作室 / team		 人 + team lead + shared task list 长期任务、多人视角、跨层协作、竞争性假设调试 协作开销和 token 成本高，不适合强依赖串行任务
Workflow
流水线 / runtime		 脚本、DAG、阶段、schema、gate 可复跑的批量流程、质量门禁、结构化产物生成 要提前建模流程，调试成本更像工程系统

基础模式：六种 Workflow 编排模式

第二组材料更关键。它不是在展示 UI，而是在展示多 Agent 编排的六种基本“拓扑”。这些拓扑决定信息怎么流、控制权怎么收束、错误怎么被发现。

01Pipeline · 流水线

每个条目独立走 2-N 个串行阶段。慢条目不阻塞快条目，是批量处理任务的首选形状。

例：为 N 个 endpoint 生成 OpenAPI：extract → fragment → validate。

02Parallel + Barrier · 同步聚合

下一阶段必须等待上一阶段全部结果，用 barrier 去重、交叉验证、避免零结果过早进入结论。

提醒：不要为“代码更整洁”滥用 barrier，等待所有分支是有成本的。

03Adversarial Verify · 对抗验证

发现要触发行动的结论时，不马上相信。让多个验证者独立投票，默认拒绝，除非多数票确认。

适合：改代码、发报告、报风险。原则：幻觉先被判假，除非证据足够。

04Judge Panel · 评委制

面对开放问题，让不同角度的评委打分，再由 synthesizer 汇总成结论和折中方案。

适合：架构选型、命名、设计方案、文案。它解决的是“没有唯一正确答案”的问题。

05Loop-until-X · 累积式

不知道要跑几轮，就用数量、预算、连续无新增等停止条件收敛。重点不是循环，而是 guard。

风险：如果没有预算和停止条件，很容易从 10 个 agent 膨胀到 1000 个 agent。

06Nested Workflow · 分层正交

父任务的每个条目，本身又是一整套 multi-step workflow。适合批量且每个子任务都复杂的场景。

例：审查 10 个 PR，每个 PR 都跑“读 diff → 查风险 → 验证 → 写评论 → 汇总”。

工程补充：八种容易遗漏的编排模式

原始材料中的六种模式很适合入门，因为它们讲清楚了“多个 Agent 怎么跑”。但放到真实产品或工程系统里，还需要进一步回答四类问题：任务如何分流、失败如何恢复、状态如何共享、高风险动作如何确认。下面八种模式补足的是工程落地层面的协作边界。

分支 不是所有请求都该进入同一条流水线。
状态 多 Agent 需要共享事实、决策和证据。
失败 真实 workflow 必须设计重试、降级和退出。
人类 高风险动作不能只靠模型自己放行。
07Router · 条件路由

先判断任务类型，再进入不同路径。它解决的是“不要让所有问题都套同一条流程”。

适合：客服分流、代码/产品/资料三类请求、权限敏感任务。关键词：先分类，再执行。

08Supervisor-Worker · 主管派工

一个主控 Agent 动态派活、看进度、收结果，不够就追加 worker。它比固定并行更灵活。

适合：复杂调研、批量修复、未知工作量。关键是 supervisor 要有预算和停止条件。

09Evaluator-Optimizer · 评估改进

一个 Agent 生成，另一个 Agent 评价，失败就带着反馈回去改。它是最常见的“越做越好”闭环。

适合：写代码、写报告、生成 UI、方案迭代。它比普通 loop 更强调“带反馈修正”。

10Human Gate · 人工门禁

当动作有风险，workflow 不应该自动放行，而是停下来让人确认。

适合：删文件、发邮件、改生产配置、花钱、公开发布。原则是：高风险动作先停住。

11Retry / Fallback · 失败恢复

基础模式更像成功路径，但真实系统一定会失败。失败恢复决定 workflow 能不能稳定运行。

适合：工具不稳定、schema 不合格、网络失败。要设计 retry 上限、fallback 和失败说明。

12Blackboard · 共享工作台

多个 Agent 不只是把结果交给主控，也可以共同维护一个事实板：证据、待办、决策、风险都写在同一个地方。

适合：长任务、复杂调研、多轮设计。它解决的是“大家到底共享什么事实”。

13Speculative Race · 竞速择优

让多个 Agent 同时尝试不同路径，最后选最快、最稳、证据最好的结果。

适合：修 bug、性能优化、命名、方案探索。和评委制不同，它不是只评价，而是真的并行试做。

14Event-driven · 事件触发

不等用户主动发起，而是由事件触发 workflow。Agent 从聊天助手变成持续工作的后台系统。

适合：CI 失败分析、PR 自动审查、周期性巡检。它把 Agent 从“问答”推向“运维”。

如果说六种基础模式解决“怎么跑”，这八个补充模式解决“怎么分流、怎么失败、怎么记住、怎么让人接管”。

这套设计的本质是：控制流交给确定性结构，语义判断交给 Agent，质量收束交给 gate。

解读框架：从概念到落地

第一层，先不要被“多 Agent”这个词吸引。多 Agent 如果没有协作协议，只是多个聊天窗口。真正的设计问题是：这些 Agent 之间是什么关系。
第二层，Subagents 像临时派出去的信使，适合一次性侦察；Agent Teams 像一个工作室，适合长期共事；Workflow 则像流水线，把协作方式固化成可复跑的工程结构。
第三层，原始材料里的六种模式主要覆盖正常执行路径：流水线、并行屏障、对抗验证、评委制、循环直到条件满足、嵌套工作流。它们分别解决吞吐、聚合、幻觉、主观判断、收敛和复杂批量任务。
第四层，真正落到产品里还要补分支、失败、状态和人类介入：Router 负责分流，Blackboard 负责共享事实，Retry / Fallback 负责失败恢复，Human Gate 负责高风险动作的刹车。
最后的判断是：未来的 Agent 应用不会只靠一个大模型“想明白”，而会越来越像一个有调度、有契约、有状态、有质量门禁的执行系统。

如何迁移到 Agent 系统？

任务形态 优先选择 设计要点
一次性查资料、读代码、读日志 Subagent 限制工具权限，要求只返回摘要、证据和不确定性
多角色长期协作，例如新功能设计与评审 Agent Team 明确 lead、角色、任务看板、冲突处理方式
批量生成、批量审查、需要稳定复跑 Workflow 声明 phase、schema、parallel、barrier、预算和停止条件
高风险结论会触发真实行动 Adversarial Verify 默认拒绝，至少两个独立验证者确认后放行
开放式方案评估，没有唯一答案 Judge Panel 让不同角度独立评分，再合成 winner + grafts
请求类型差异很大，不应该走同一套链路 Router 先判断任务类型、风险等级和所需工具，再进入对应 workflow
生成物需要反复打磨，例如方案、代码、报告 Evaluator-Optimizer 把“生成”和“评价”拆开，让反馈驱动下一轮改进
多个 Agent 需要长期共享事实和证据 Blackboard 建立共享工作台，记录 facts、todos、evidence、decisions
动作可能造成真实世界影响 Human Gate 把人工确认设计成流程节点，而不是最后临时问一句

核心结论：七句话

  1. Subagent 解决的是“别污染主上下文”。
  2. Agent Team 解决的是“复杂任务需要多个独立视角互相协作”。
  3. Workflow 解决的是“多 Agent 协作要可复跑、可观察、可验证”。
  4. Schema、phase、budget、gate 是把 LLM 不确定性纳入工程系统的关键。
  5. 六种基础模式覆盖了基础执行拓扑，但还不覆盖所有工程边界。
  6. 真实 Agent 系统还要补分流、共享状态、失败恢复和人工门禁。
  7. 多 Agent 的核心不是数量，而是拓扑：谁并行、谁等待、谁验证、谁合成、谁兜底。
