当前已经具备的展示能力

文本主回答

最终 assistant message 作为主要阅读内容；流式阶段用 streamingText 临时展示，完成后补拉落库消息。

执行过程投影

runtime events 按 seqNo、createdAt 和到达顺序排序，工具、推理、产物、错误被压缩成步骤。

待交互内联

CONFIRMATION_CREATED 会进入确认 store，并在对话区和右侧待确认面板同时出现。

多场景确认卡

方案确认、权限申请、运行恢复、信息补充、测试失败、冲突、发布审批等场景统一使用决策卡呈现。

断线局部可见

浏览器 SSE error 会写入本地 runtime ERROR 事件，进入主对话过程里。

事件回放缓冲

前端保留 eventBuffer，重连时可用 afterSeq 或 Last-Event-ID 继续补齐会话事件。

现状里最影响体验的地方

  • 卡片密度偏高：多场景都展示后右栏需要滚动，优先级、分组和折叠策略还需要继续收敛。
  • 状态语义偏弱：顶部只有已连接/连接中，用户看不出最后同步点、重连状态和事件是否补齐。
  • 展示靠启发式过滤：heartbeat、tool lifecycle、snapshot delta 都由前端规则消噪，语义还不够硬。
  • 确认入口仍需排序：主对话、右栏卡片和底部输入拦截都能承接交互，下一步需要明确谁是主入口。
