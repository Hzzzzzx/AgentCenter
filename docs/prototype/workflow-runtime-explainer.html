AgentCenter 当前工作流说明

它不是一个固定流水线，而是“按顺序调用 Skill 的对话式执行器”

当前实现的目标是：工作流负责提供顺序、上下文和状态协议；真正的产出、是否需要澄清、何时完成，尽量交给 Agent 和 Skill 自己判断。 前端对话区展示的是绑定到当前工作流节点的 Agent 会话消息。

读图钥匙

工作项FE / US / TASK / BUG 等业务对象，是工作流的入口。
节点PRD / HLD / LLD 等阶段，只声明顺序和 Skill。
Skill真正执行任务的说明和能力，应优先发挥自身作用。
交互Agent 需要用户选择、确认、补充或审批时创建。

一条工作流从点击“开始处理”到对话区出现内容

模式更适合放在左下角“设置”里的“运行设置”，作为用户的默认执行偏好。

自动模式 人工确认

运行设置

自动运行工作流

开启后，Agent 判断当前节点已完成时，系统可以自动进入下一节点；关闭后，每个节点完成后都需要用户确认。

这个设置可以作为默认值；单个工作项启动时未来也可以允许临时覆盖。

AUTO

自动模式

Agent 认为当前 Skill 已完成且条件充分时，可以进入“准备推进”状态；系统可按策略自动推进，或在高风险节点前弹出确认。

MANUAL

人工确认模式

每个节点输出完成后都创建“进入下一节点 / 继续补充 / 重新执行”的确认项，由用户明确选择后再推进。

1. 用户点击开始
前端调用 POST /api/work-items/{id}/start-workflow

按钮所在位置可能是首页列表、右侧详情或会话入口。请求使用“运行设置”里的默认模式，例如 mode: AUTO 或 mode: MANUAL_CONFIRM。

2. Bridge 建立实例
创建或复用 workflow_instance 与 workflow_node_instance

每个节点保存自己的状态：PENDING、RUNNING、WAITING_CONFIRMATION、COMPLETED 等。

3. 绑定 Agent 会话
当前节点绑定 agent_session_id

对话 UI 的消息来源应是当前节点绑定的 AgentSession，而不是同一个工作项下最新的一条任意会话。

4. 写入用户输入
工作项编号、标题、描述、Skill、节点上下文作为 USER 消息展示

这就是你要求的“任务信息和 Skill 不是只展示，而是作为用户输入进入对话”。

5. Runtime 执行 Skill
OpenCode 收到三段文本：调用 Skill、用户输入上下文、节点状态协议

协议只告诉 Agent 怎么报告状态，不应该把工作流写死成僵硬脚本。

6. 前端流式展示
SSE 把 Agent 文本、工具调用、状态事件推回浏览器

对话区应该像正常聊天一样看到用户输入、Agent 输出、工具过程和最终状态。

7. 按模式决定推进
自动模式看策略，人工确认模式等用户

同样是 READY_TO_ADVANCE，自动模式可以继续跑下一节点；人工确认模式会在输入框上方固定展示推进确认。

节点状态怎么理解

状态 用户会看到什么 系统动作
IN_PROGRESS Agent 正在输出，输入框显示运行中 不推进节点，继续等 Runtime
NEEDS_USER_INPUT 输入框上方出现“当前需要交互”折叠栏 创建确认项，用户处理后把回答回灌给当前 Skill
READY_TO_ADVANCE 自动模式可继续推进；人工确认模式出现“进入下一节点 / 继续补充 / 重新执行” 按当前模式和节点风险决定是否创建推进确认项
BLOCKED 出现异常或阻塞交互 等待重试、跳过、补充信息或人工处理

当前对话 UI 映射

左侧会话

  • 普通会话
  • 任务会话
  • 当前工作项的入口

中间对话

  • USER：工作项输入、用户交互回答
  • ASSISTANT：Agent 真实回复
  • SYSTEM：状态和工作流记录

右侧详情

  • 工作项信息
  • 工作流进度
  • 待确认列表

浮层交互

  • 固定在输入框上方
  • 多个交互用 Tab 切换
  • 处理后写回 USER 消息
当前正确映射应该是：workItem.currentWorkflowInstanceId → currentNodeInstanceId → node.agentSessionId → agent_message。

现在这套设计的关键规则

  • 工作流只负责顺序和上下文，不应该强行替代 Skill 的判断。
  • 启动工作流时应明确模式：AUTO 自动推进优先，MANUAL_CONFIRM 人工确认优先。
  • 用户手动输入和交互选择都要写成 USER 消息，留在对话历史里。
  • Agent 的文本输出应该按流式进入中间对话区，工具调用作为过程事件展示。
  • 普通业务决策只提交 CHOOSE；只有 WORKFLOW_ADVANCE 确认项才能触发进入下一节点、继续补充或重跑。
  • 如果用户中途补充要求，系统应回到当前 Skill，而不是代码层直接跳到下一节点。
  • 节点是否完成主要由 Agent 输出的状态协议判断，再由用户确认或策略决定是否推进。
