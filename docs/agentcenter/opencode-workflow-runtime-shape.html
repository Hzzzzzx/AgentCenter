2. 它不是并行列表，而是 DAG 状态图

顺序、并行、分支和汇聚都可以用同一个 DAG 表达。每个节点有自己的完成契约； 下游只有在依赖节点满足契约以后才会启动。

已完成 运行中 等待用户 被依赖阻塞

3. 子 Agent 中途要交互时，运行链路怎么走

子 Agent 不应该直接把用户拖进自己的子会话。它应该把问题结构化交给 Workflow Runtime， Runtime 在主会话展示问题、记录 checkpoint，用户回答后再把答案喂回原节点继续执行。

4. Runtime 需要保存的状态

只要状态保存清楚，顺序、并行、暂停、失败重试和恢复都不会互相打架。 这也是它以后能被 Codex 或其他 Agent 复用的基础。

Run 级别

  • runId：一次 workflow 运行。
  • workflowVersion：用户固化出来的流程版本。
  • dag：节点、依赖、分支和 gate。
  • globalStatus：running、waiting、failed、succeeded。
  • checkpoint：恢复所需上下文，不依赖聊天历史。

Node 级别

  • nodeId：当前节点。
  • agent：要调用的 Agent 类型。
  • status：pending、running、waiting、succeeded、failed。
  • input/output：结构化输入输出。
  • waitReason：用户、权限、审查或外部资源。
状态 界面表现 Runtime 行为 下游能否继续
RUNNING 节点显示运行中，可展开看子 Agent 日志。 持续接收事件、更新 evidence、监听取消。 依赖它的节点不能启动。
WAITING_FOR_USER 主对话展示问题卡片和选项。 保存 checkpoint，暂停该节点，等待 resume。 只阻塞依赖它的下游。
SUCCEEDED 节点变绿，展示输出摘要和证据。 校验 output schema 和 gate，解锁下游。 通过 gate 后可以继续。
FAILED 节点变红，展示失败原因、重试或跳过按钮。 按策略 retry、fallback、ask_user 或终止 run。 默认阻塞下游，除非 DAG 有 fallback 边。

5. 我对第一版的落地判断

第一版不要急着做复杂画布编辑器。先把运行态做稳：自然语言生成 workflow、运行卡片、节点状态、 等待用户、恢复执行、证据归档。等这个链路跑通，再加可视化编辑。

这张图的重点：Workflow 的运行态不是聊天记录拼接，而是 run / node / checkpoint / evidence 共同组成的状态机。OpenCode 负责执行子 Agent，Workflow Runtime 负责秩序和恢复。
