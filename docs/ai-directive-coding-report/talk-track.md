# AI DevOps 指挥式纯 AI 编码实践讲解稿

## 1. 封面

这一页我想先讲清楚，今天分享的重点，不是某个具体功能怎么做出来，而是一种新的编码组织方式。我把它叫做“指挥式纯 AI 编码实践”。

核心不是让 AI 替我写代码，而是我来负责方向、架构、规则和验收，让 Agent 去负责实现、检查、修复和交接。也就是说，人的位置开始从“亲自写每一段代码”，上移到“组织整个工程执行”。

## 2. 背景

这件事的起点很现实。团队需要一个面向 AI DevOps 的 Agent 能力，但时间很紧。如果按传统模式推进，周期通常会比较长，可能要一两个月。

所以我一开始就没有走传统节奏，而是决定先用 AI 去快速验证方向。这个阶段里，我不是直接写代码，而是先把诉求说清楚，再让 AI 参与方案展开和快速推进，尽快把这个方向从想法变成可以讨论、可以判断的东西。

## 3. 我是怎么指挥的

这一页是我整个方法最核心的一页。我的顺序大概是这样：先讲愿景，明确我们要做的是一个 AI DevOps Agent；然后不是立刻拍板，而是先让 AI 基于诉求去给一些引导和方案选择；接下来进入设计，把它做成高保真，把多栏工作台、信息布局、交互方式先看见；然后收边界，讲清楚第一版到底做什么、不做什么、验收标准是什么；最后才进入执行，让 Agent 按照这些内容去实现。

所以我理解里的“指挥”，本质上不是一句 prompt，而是先把意图、设计、边界组织清楚，再把执行交出去。

## 4. 结果

这一页我会讲最后做到了什么程度。整个过程里，真正密集编码的窗口大概是 7 天，由我一个人主导推进，累计 169 次提交，总代码量 7 万多行，里面产品代码接近 4.9 万行，测试代码 2.3 万多行。

更重要的不是数字本身，而是最后长出来的东西已经不是一个静态 demo，而是一个可以运行的工作台，能让同学把业务 Skill 放进去试、去调、去验证。所以这次实践的价值，不只是“很快做出来”，而是把原本停留在讨论层的想法，推到了一个可以实际运行和继续收敛的状态。

## 5. AI 时代的作业模型

这一页我想表达的是，我现在对 AI 编码的一个理解。模型本身像大脑，但还不够。它还需要眼睛、地图和仪表盘，也就是感知系统和反馈系统。

左边的意图输入是人的职责，要把愿景、架构和规则讲清楚；右边的人工验收也是人的职责，要基于真实使用结果给出最后判断。中间 Agent 负责理解、实现和修复，但它不是凭空完成的，它需要代码关系、影响面、上下文这些感知工具，也需要测试、截图、日志、复盘这些反馈工具。这样它才能形成一个工程闭环，而不是一次性生成一堆代码。

## 6. 模型

这一页我会讲为什么我会特别强调模型能力。不是所有模型都能支撑这种指挥式快跑。

我的实际体验是，强模型像 `Codex + GPT-5.5`，已经能进入一个“可放手区间”，也就是目标清楚、边界清楚、可验证的任务，它能比较快地实现并闭环；但像我试过的内部 `opencode + GLM-5` 路线，在复杂任务上闭环能力还是弱很多，时间拉长了，结果不一定更好，甚至会越改越偏。

所以这套方法并不是脱离模型能力独立存在的，它有一个很现实的前提，就是模型本身要足够强。

## 7. 体会

最后这一页，我想收束成我自己的几个实践体会。

第一，生产力关系已经变了，所以以前那套围绕“人怎么写代码”的作业流程，也要跟着重构。第二，在 AI 时代，最重要的环节不再只是写代码本身，而是意图表达、设计清楚和测试前置。第三，我们要做的不是盲目相信 AI，而是围绕“怎么让 AI 发挥最大作用”去组织整套流程，包括怎么给它清楚目标、怎么帮它收边界、怎么替它兜底、怎么让它根据反馈自动闭环。

这次实践最核心的收获是：AI 编码时代，人要抓住的是意图、设计和测试，执行会越来越交给 AI，而且最终一定会走向自我闭环。
