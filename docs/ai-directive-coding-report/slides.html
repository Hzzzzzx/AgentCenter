指挥式纯 AI 编码实践

这次分享不讲某个项目有多复杂，而讲一种新的编码组织方式：人负责产品判断、边界和验收，Agent 负责实现、检查、修复和交接。

一句话主张

不是让 AI 替人写代码，而是把愿景、设计、边界和验证组织成一套 AI 可以持续执行的工程系统。

一、怎么用 AI 指挥式编码

这条路径不是一上来就堆代码，而是先把想法做成可交互原型，再经过评审、试错和模型选择，最后进入密集工程实现。

原型先行

把想法快速变成高保真

第一步不是写代码，而是用 AI 快速做出可交互高保真，让团队能围绕真实界面讨论方向。

评审收敛

内部评审后确定第一版边界

评审重点不是“好不好看”，而是第一版要承载哪些模块、流程、对话能力和验收标准。

路线选择

试过内部路线，再切到外部强模型

内部 opencode + GLM-5 尝试效果不好；时间紧、项目又是新项目且不涉及内部信息，于是改用外部 Codex + GPT-5.5 快速演进。

1

想法出现

团队需要一个面向 AI DevOps 的 Agent 能力。

2

高保真验证

用 AI 快速生成可交互原型，让抽象方向变成可感知形态。

3

内部评审

围绕界面评审功能边界、模块划分、流程节点和验收方式。

4

内部路线试错

尝试用内部工具和 GLM-5 推进，但复杂实现闭环效果不理想。

5

外部强模型快跑

在新项目、无内部信息的边界下，使用 Codex + GPT-5.5 连续实现。

6

检查和收敛

通过 code check、测试、截图和复盘，把问题继续压回闭环。

重点

指挥式编码不是盲目相信 AI，而是在“原型清楚、边界清楚、信息边界可控、模型能力足够”的条件下，把工程执行交给更强的 Agent。

指挥式编码的工作系统

强模型不是万能，但大量清楚、可拆、可验证的编码任务已经进入可用区间。

人的职责

定义愿景和验收标准；完成架构设计、模块边界和业务规则澄清；判断风险，决定什么时候拆小、暂停或接受结果。

Agent 的职责

理解目标、检索代码、执行修改、运行检查、修复问题、产出可运行系统和交接信息。

质量问题怎么闭环

传统代码问题没有消失，而是被放进 Agent 的反馈闭环：发现、定位、修复、验证、沉淀。

1

发现

typecheck、test、build、日志、截图、人工验收暴露问题。

2

定位

代码关系图、影响分析、调用链和分层排查减少盲改。

3

修复

把问题场景、边界、相关模块和验收方式交给 Agent。

4

验证

重新跑检查、测试、构建、curl 或截图，不靠感觉宣布完成。

5

沉淀

把失败模式写回 SDD、计划、审查卡片和后续提示。

问题仍存在

类型、接口、状态、UI、回归

AI 编码不会消灭工程问题，生成速度更快时，问题也会更快暴露。

闭环方式改变

从人工盯守到工具反馈

检查工具告诉 Agent 做没做对，代码关系工具告诉它影响面在哪里。

人的责任保留

方向和验收不能外包

人仍然决定目标是否正确、风险是否可接受、任务是否需要继续拆小。

质量判断

质量不是靠相信 AI，而是靠把 AI 放进可验证的工程闭环里。

二、最终成果

成果总结

这轮实践证明，指挥式纯 AI 编码可以在短周期内把想法推进到可运行、可验证、可继续收敛的工程状态。

实际效果：功能可用，验证加速

目标不是一次性做出终态，而是把想法推进到可被多人试用、可暴露真实问题、可继续收敛的系统状态。

功能可用

想要的能力已经长出来

对话、任务看板、工作流、Skill/MCP、Runtime Bridge、确认交互和设置等核心功能点已经形成可操作工作台。

可以验证

业务相关 Skill 可以放到系统里做测试、调整和联调，团队可以围绕真实操作结果判断方向。

继续收敛

问题进入后续看护
