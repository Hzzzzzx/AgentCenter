## 1. 评测方法说明

### 1.1 锚点与基线

**锚点**：Hindsight 在 LongMemEval 上实测 **91.4%**（设计阶段）。编码阶段与设计阶段的能力权重不同，因此总分不可直接横向比较。

**基线**：LongMemEval 无记忆基线约 **39%**（full-context LLM，不使用任何记忆系统）。

**估算方法**：

- 📊 **实测** = 论文发表的 benchmark 数据，高可信度
- 🔍 **架构推断** = 基于框架公开的架构设计文档、技术博客、源码分析，推断其在特定能力上的理论表现

### 1.2 编码阶段能力维度权重（10 维度）

编码阶段与设计阶段的本质差异决定了权重分布：

| 维度 | 编码阶段权重 | 设计阶段权重 | 权重差异说明 |
|------|------------|------------|-------------|
| **精确检索** | **15%** | 10% | 编码需要精确到方法签名，不能模糊 |
| **时序推理** | **15%** | 12% | 代码变更频繁，需追踪"什么时候改的" |
| 语义关联 | 12% | 15% | 编码的语义关联比设计略弱 |
| 实时索引 | **12%** | 8% | 代码变更后必须立即可查 |
| 增量更新 | **10%** | 8% | 频繁变更要求增量而非全量重建 |
| 多跳推理 | 10% | 15% | 编码的多跳比设计需求略低 |
| 冲突检测 | 8% | 10% | 编码不允许矛盾事实存在 |
| 代码结构感知 | **8%** | 3% | 编码阶段独有维度，理解 AST/调用图 |
| 上下文窗口效率 | 5% | 5% | 不变 |
| 部署复杂度 | 5% | 5% | 不变 |

### 1.3 评分含义

每个维度的分数区间含义：

| 区间 | 含义 |
|------|------|
| 90-100% | 业界领先，能精确解决该维度问题 |
| 80-89% | 优秀，该维度能力充分 |
| 70-79% | 良好，能满足大多数场景 |
| 60-69% | 中等，存在已知限制 |
| 50-59% | 勉强可用，限制明显 |
| < 50% | 不足，该维度非核心能力 |

## 4. 逐框架评分推理

### 4.1 Cortex

**加权总分：81.5**

| 维度 | 得分 | 推理依据 |
|------|------|---------|
| 精确检索 95-98% | [1] | PG+pg_trgm 提供精确三模匹配，FTS5 提供精确全文检索。对于"UserService.createUser 的参数"这类查询，trigram 能处理拼写变体同时保证精确度 |
| 语义关联 95-98% | [2] | pgvector HNSW (M=16, ef_construction=64) 是业界最强的向量索引之一。5 路信号融合（向量+全文+三模+热值+时序）确保语义召回质量 |
| 时序推理 70-80% | [3] | 有 timestamps 和 consolidation_stages (labile→early_LTP→late_LTP→consolidated)。但这是衰减模型而非 bi-temporal，不能精确回答"规则什么时候失效" |
| 多跳推理 60-70% | [4] | SQL JOIN 可以做多跳，但不如图遍历直观。对于"改了这个字段影响哪些测试"这类查询，需要多次 JOIN，不如 Cypher 直接 |
| 实时索引 90-95% | [5] | PostgreSQL triggers 可在写入时触发索引更新，延迟可控制在秒级。HNSW 的 ef_construction 参数可调，平衡写入速度和查询质量 |
| 冲突检测 85-90% | [6] | consolidation_stages 有明确的记忆合并阶段，旧记忆在合并前会被标记，不会同时存在矛盾记忆 |
| 增量更新 90-95% | [7] | PostgreSQL 的 upsert 语法支持单记录更新，不需要全量重建。plasticity/stability 字段控制哪些记忆可以更新 |
| 代码感知 30-40% | [8] | 无 AST 解析能力，不知道代码的语法结构。只能按文本内容检索 |
| 上下文效率 85-90% | [9] | 5 路信号融合在数据库端完成，减少无效 token。cross-encoder 重排提高召回质量 |
| 部署复杂度 60-70% | [10] | 需要 PostgreSQL 15+ 和 pgvector 扩展，运维复杂度中等 |

## 7. 关键发现

### 发现一：精确检索和时序推理是编码阶段最关键的差异化因素

编码阶段对精确检索的需求（15% 权重 vs 设计阶段 10%）和时序推理（15% 权重 vs 设计阶段 12%）共同决定了框架排名。Hindsight 在设计阶段凭借多跳推理领先，但在编码阶段被精确检索和时序的短板拖累。

Cortex 的精确检索（95-98%）和 total-agent-memory 的时序推理（75-85%）分别代表了编码阶段两个最重要的能力方向。

### 发现二：Tier D 工具在"代码结构感知"维度出人意料地强于多数记忆框架

| 框架 | 代码感知得分 | 说明 |
|------|------------|------|
| Claude Code | 65-75% | skills + projects 系统 |
| OpenCode | 60-70% | skills 系统 |
| Codex | 55-65% | AGENTS.md 结构 |
| Windsurf | 50-60% | Cascade 规则 |
| **total-agent-memory** | **35-45%** | 22 种 node types |
| Cortex | 30-40% | 无 AST 感知 |
| **所有记忆框架** | **< 45%** | 均弱于 Tier D |

这说明现有记忆框架普遍缺乏代码结构理解能力。Tier D 工具的 skills/projects 系统虽然原始，但确实比大多数专用记忆框架更理解代码。

### 发现三：Zep/Graphiti 的时序推理在编码阶段价值最大

编码阶段需要频繁追踪"这个接口什么时候改的、为什么改"。Zep 的 bi-temporal 模型（valid_at/invalid_at）是全场唯一能精确回答这类问题的方案。

然而，Zep 的部署复杂度（35-45%）严重拖累了综合排名。对于不需要强时序推理的场景，total-agent-memory 的 kg_at() 是一个更轻量的替代方案。

### 发现四：total-agent-memory 是编码阶段综合最优的单一存储方案

total-agent-memory 的加权总分 82.6 分来自：

- SQLite 部署最简（75-85%）
- 时序 KG 支持（75-85%）
- 图多跳能力（70-80%）
- 22 种代码 node types（35-45%）
- 无外部依赖

这使它成为 AgentCenter 编码阶段记忆系统的最佳单一选择。

### 发现五：TencentDB 的 4 层管线在编码场景下有独特价值

TencentDB-Agent-Memory 的 61% token reduction 使其在上下文效率维度（85-90%）仅次于 OpenViking。对于编码阶段的上下文窗口紧张问题，这是一个被低估的优势。

### 发现六：编码工具和记忆系统的融合是趋势

Tier D 框架（OpenCode/Claude Code）在代码感知上的优势（60-75%）远超记忆框架（< 45%），而记忆框架在语义关联和时序推理上的优势（75-95%）是 Tier D 框架的短板（10-20%）。

未来最佳方案可能是：记忆框架提供语义/时序能力，编码工具提供代码结构理解，两者通过 API 融合。

## 8. AgentCenter 编码阶段选型建议

### 8.1 短期（MVP）：基于 total-agent-memory

**理由**：SQLite 部署与 AgentCenter Bridge 技术栈一致，时序 KG 支持编码阶段核心需求，22 种代码 node types 提供基础代码感知。

**实施路径**：

1. 引入 total-agent-memory 的 schema 设计（graph_nodes + fact_assertions）
2. 基于 SQLite 实现 kg_at() 时序查询
3. 扩展 node types 覆盖 AgentCenter 代码结构

### 8.2 中期：融合 Tier D 代码感知

**理由**：记忆框架的语义/时序 + 编码工具的代码感知 = 完整方案。

**实施路径**：

1. 记忆层：PostgreSQL + pgvector（如 Cortex 的 5 信号融合）
2. 代码感知层：集成 OpenCode skills 系统
3. 时序层：引入 bi-temporal 建模（参考 Zep）

### 8.3 长期：统一记忆 + 代码理解

**目标**：单一系统同时提供语义检索 + 时序推理 + 代码结构感知。

**技术方向**：

- 基于 LSP 的代码结构索引
- 时序知识图谱（bi-temporal edges）
- 向量 + 全文 + 图混合检索
