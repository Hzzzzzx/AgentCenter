## 项目概览

Graphify 是由 neuronpedia 开发的 Claude Code 技能（skill），并非独立产品。其核心定位为**本地知识图谱构建工具**，通过多模态输入（代码、PDF、图片、Markdown）自动提取实体与关系，生成结构化的知识网络。

技术栈简明：Python 作为运行时，tree-sitter 负责代码解析，NetworkX 构建有向图，Leiden 算法（graspologic 实现）做社区检测，Claude API 处理图片理解和概念抽取，vis.js 负责图谱可视化。

区别于 Neo4j 等图数据库方案，Graphify 完全在内存中操作，无需部署外部服务，这使其非常适合个人开发者或小团队快速建立代码知识索引。

### 7阶段管线详解

**Stage 1: detect**

首阶段扫描输入文件集，识别文件类型。代码文件根据扩展名路由到 tree-sitter 解析器，PDF 走 citation mining 流程，图片调用 Claude Vision API，Markdown 则走概念抽取。这一分类决定了后续处理管道的选择。

**Stage 2: extract**

这是最核心的阶段。代码解析分两步进行：第一步用 tree-sitter 做 AST 遍历，收集 classes、functions、imports 等结构化实体；第二步做调用关系图构建，通过静态分析推断函数调用链，标记为 INFERRED 边。

PDF 处理采用引用挖掘（citation mining）策略，从学术文献中提取引用关系形成图谱边。图片处理由 Claude Vision API 完成——模型描述图片内容后，Graphify 将描述中的概念转化为图谱节点和边。Markdown 文件则提取文档标题、章节结构和内部链接，建立概念层次。

**Stage 3: build_graph**

所有实体和关系汇入 NetworkX DiGraph。每条边携带置信度标签，这是 Graphify 设计的关键洞察：

- **EXTRACTED**：直接从源码或文档中提取的关系，可信度最高
- **INFERRED**：通过静态分析推断的关系，存在假阳性可能
- **AMBIGUOUS**：置信度最低，仅在开启宽松匹配模式时保留

边的标签设计使下游分析可以按需过滤——高可信度场景只考察 EXTRACTED 边，探索性分析可包含全部类型。

**Stage 4: cluster**

Leiden 算法（grapologic 库实现）接手社区检测。Leiden 是 Louvain 的改进版，理论上保证更好的社区划分质量。对于大型代码库，Leiden 能有效将数千个节点归并为若干个内聚社区，每个社区代表一个功能模块或概念群。

**Stage 5: analyze**

图谱分析聚焦两个维度：**God nodes**（高degree节点，即与其他实体连接最多的概念）和 **Surprising connections**（通过复合评分算法挖掘的非显而易见关系）。

Surprising connections 的设计意图在于揭示架构层面的深层联系——那些在代码中物理分散但语义相关的模块，往往通过这类连接被发现。

**Stage 6: report**

分析结果写入 GRAPH_REPORT.md。报告包含社区列表及其成员、God nodes 的中心性分析、以及 ranked 的意外连接。人类可读的 Markdown 格式使非技术利益相关者也能理解图谱洞察。

**Stage 7: export**

最终导出支持多种格式：graph.json（结构化图数据）、graph.html（嵌入 vis.js 的交互式可视化）、Obsidian vault 格式（双链笔记）、以及标准 Wiki 格式。

### 多模态提取策略

Graphify 的多模态处理并非简单拼接，而是围绕知识图谱这一统一表示进行。代码提供结构知识，PDF 提供学术引用知识，图片提供视觉概念知识，Markdown 提供文档组织知识。所有模态最终都被归一化为节点和边，进入同一张图。

这种设计的优势在于：跨模态的边能够在单一图结构中自然呈现。例如一段代码中引用的论文可以通过 PDF 的引用关系和图片的视觉概念形成跨模态连接。

### 知识图谱构建

基于 NetworkX 的 DiGraph 是内存中的图数据库。不同于 Neo4j 等持久化图数据库，NetworkX 图谱随进程存在，进程结束即消失。这意味着 Graphify 的图谱天生是临时性的——每次运行从原始数据重新构建。

对于增量更新场景，Graphify 通过 SHA256 哈希缓存文件状态，仅重新处理变更文件。这在大型代码库场景下显著提升了效率。

### 社区检测与 Wiki 生成

社区检测结果直接驱动 Wiki 生成。每个社区对应一个 Markdown 文件，文件名即社区名称。Wiki 模式的核心价值在于其双链能力——通过 [[Cluster Name]] 语法，Obsidian 或其他支持双链的工具可以建立社区间的导航网络。

God node 享有独立页面，解释其为何连接如此多的实体。这实际上是一种自动化的架构文档生成——原本需要人工梳理的中心性分析，现在由算法完成。

### 增量更新机制

SHA256 缓存是 Graphify 实现增量更新的机制。每次处理前比对文件哈希，仅重新处理状态变化的文件。配合 --watch 模式，可以监听文件系统变化并触发增量重建。git hook 集成则支持在 commit 后自动同步图谱。

## 核心设计思想

### 为什么选 tree-sitter 而不是 LLM 解析代码？

这个问题触及 Graphify 最关键的设计选择。LLM 解析代码的优势在于理解语义，但其劣势同样明显：token 消耗巨大、延迟高、结果不稳定。

tree-sitter 提供确定性、线性时间的解析能力，AST 结构化程度高，适合批量处理。Graphify 在代码解析层选择 tree-sitter 保证覆盖率和高效率，仅在图片理解等 tree-sitter 无法处理的模态才调用 LLM。

这是一种**混合策略**：确定性解析交给规则系统，语义理解交给 LLM。两种能力各司其职。

### 置信度标签（EXTRACTED/INFERRED/AMBIGUOUS）的意义

边的置信度标签是 Graphify 图谱质量管理的核心机制。

在真实代码库中，静态分析推断的调用关系（如通过字符串匹配推断的反射调用）并非绝对可靠。如果没有置信度标记，所有边被同等对待，分析结果会被大量噪音污染。

通过三级标签，Graphify 让下游应用可以按需过滤。高可靠场景（如安全审查）只考察 EXTRACTED 边，探索性分析可以包含 INFERRED 边，AMBIGUOUS 边仅在需要最大覆盖率时保留。

这本质上是将「不确定性显式化」，而非试图掩盖它。

### Wiki 作为知识载体的价值

Graphify 选择 Wiki 作为主要输出格式，这一看似简单的选择其实蕴含深意。

代码知识图谱的最终消费者不一定是工程师。架构师需要理解模块边界，产品经理需要了解功能归属。这些人不会阅读 JSON 图数据，但会阅读 Wiki 页面。

Wiki 的双链语法（[[Page Name]]）天然适合表示知识网络中的导航关系。Obsidian 等工具进一步支持在图谱视图中可视化这些链接关系。这意味着 Graphify 的输出可以直接被现有知识管理工具消费，无需额外开发。

### "惊喜连接"的算法设计

Surprising connections 是 Graphify 最具价值的分析能力，但其算法细节在公开资料中披露有限。根据输出报告的结构推断，这是一个复合评分过程：综合考虑两个节点在代码库中的物理距离、共同邻居数量、边的类型权重等因素。

这种设计的价值在于：它不只告诉我们「A 和 B 有关」，更揭示了「为什么 A 和 B 有关」。通过审计追踪，每条 Surprising connection 都能回溯到具体的源码位置或文档来源。

## 优势分析

**全模态统一图谱**。Graphify 是目前少数能够将代码、PDF、图片、Markdown 纳入同一知识图谱的工具。这四种输入类型覆盖了技术团队主要的知识载体，相互之间的跨模态连接尤其有价值。

**零外部依赖**。无需 Neo4j、无需 MySQL、无需任何外部服务。所有数据在内存中的 NetworkX 图里，pip install 即可运行。这对于个人开发者和小型团队意味着极低的上手门槛。

**token 效率**。71.5 倍的 token 压缩比意味着：用传统方法将 52 个混合文件喂给 LLM 所需的 token，足够 Graphify 构建完整图谱并生成分析报告。在 LLM API 按 token 收费的背景下，这个数字直接关联成本。

**增量更新能力**。SHA256 缓存、watch 模式、git hook 集成，构成了一套完整的增量更新方案。对于大型代码库，这是必需能力。

**审计追踪**。每条边可追溯到具体源码位置，每个分析结论有原始数据支撑。这在需要向他人解释图谱洞察时尤为重要。

## 局限与不足

**非服务化设计**。Graphify 是 CLI 工具，不是 API 服务。它的输出是文件，不是可供程序消费的接口。如果要在 Bridge 中集成，需要额外的封装层。

**无持久化图数据库**。NetworkX 内存图在进程结束后消失。对于需要长期维护和查询的图谱场景，这构成根本性限制。

**社区检测是一次性的**。Graphify 不追踪社区随时间的演化。今天检测到的社区划分，与三个月后的划分无法直接对比。这对于大型项目的架构演进监控是不小的缺憾。

**多用户场景未覆盖**。没有访问控制、没有并发支持、没有事务概念。在企业环境中直接使用会有安全和管理上的问题。

**tree-sitter 语言覆盖有限**。虽然官方声称支持 Python、JS/TS、Go、Rust、Java、C、C++，但这个列表远未覆盖企业项目的全部技术栈。COBOL、PowerShell、TypeScript 的一些高级类型系统就无法处理。

**无实时更新能力**。batch 模式和 watch 模式都不是真正的实时。watch 模式本质上是文件系统监听 + 增量重建，有延迟且不适合高频变更场景。

### Wiki 导出可作为代码层知识载体

AgentCenter 的 Bridge 需要管理企业工具流程，其中涉及大量代码理解和知识沉淀。Graphify 的 Wiki 导出机制与 AgentCenter 的文档层有天然的契合点。

具体而言，AgentCenter 可以利用 Graphify 的 Wiki 输出作为代码知识的结构化载体。每个社区的 Markdown 页面包含成员实体和关系定义，可以直接作为上下文注入到 LLM prompt 中。

更进一步，Obsidian vault 导出意味着 AgentCenter 可以利用 Obsidian 的图谱可视化能力来展示代码结构。这比纯 JSON 输出的人类可读性高出一个数量级。

### 需要封装为服务才能集成到 Bridge

Graphify 当前是 CLI 工具，AgentCenter Bridge 是 Java 服务。要将两者集成，需要解决几个工程问题：

**进程调用 vs API 调用**。Bridge 需要通过 ProcessBuilder 或类似机制调用 Graphify CLI，捕获其文件输出。这不是难事，但增加了系统复杂度。

**输出格式的二次解析**。Graphify 的 graph.json 输出需要被 Bridge 消费。Bridge 需要一个 JSON 解析器将图数据读入 Java 侧的数据结构。这属于标准工程问题。

**增量接口设计**。Graphify 的 --watch 模式是文件级的，Bridge 需要一个更细粒度的事件模型来驱动增量更新。这需要额外设计。

**长期存储**。Graphify 的 NetworkX 图谱是临时的，Bridge 需要持久化存储已构建的图谱。这要求 Bridge 维护一个持久化图数据库（如 Neo4j），或使用 JSON 文件存储。

这些都不是不可逾越的技术障碍，但都需要明确的工程规划和实现投入。
