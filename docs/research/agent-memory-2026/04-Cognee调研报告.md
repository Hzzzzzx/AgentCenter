# Cognee 调研报告

> AgentCenter 记忆系统调研 · 开源项目 #4
> 调研日期：2026-05-23

---

## 项目概览

Cognee（GitHub: topoteretes/cognee）是由 topoteretes 团队开发的一个开源 cognitive data framework，目标是为 AI 应用提供从原始数据到结构化知识的完整管线处理能力。该项目在 2026 年获得 €7.5M 融资，目前 GitHub 约 12K stars，主要语言为 Python，采用 Apache 2.0 许可证开源。

Cognee 的核心定位与 Mem0、Letta 有本质区别。它并非简单地提供一组 memory API，而是将自身定义为一个完整的数据到知识的 pipeline engine。官方 tagline 称其为 "Cognitive data framework for AI applications"，这个称呼准确地反映了它的设计意图：从多种数据源摄入原始内容，通过一系列 LLM 驱动的转换步骤，最终将结构化知识注入到多种存储后端中。

从项目演进来看，Cognee 的设计思路更接近传统 ETL 工具与现代 LLM 应用的交叉点。它不试图成为一个"智能记忆管理系统"，而是将记忆管理所需的认知能力分解为可配置、可组合的管线阶段。这种设计选择既是其最大优势，也构成了其主要的使用门槛。

### 数据管线详解（classify → chunk → extract → index）

Cognee 的管线设计是其最具特色的部分。数据从摄入到知识化的过程被分解为五个阶段，每个阶段都是独立可配置的：

**第一阶段：classify（分类）**

管线首先对输入数据进行分类，确定数据的类型和处理策略。这一步决定了后续 chunking 和 extraction 阶段的具体行为。分类器通过 LLM 或规则引擎实现，为不同类型的数据分配合适的 processing path。

**第二阶段：chunk（分块）**

根据分类结果，数据被切分为适合处理的 chunk。不同的数据类型（文本、PDF、音频转录等）对应不同的 chunking 策略。Cognee 支持多种 chunking 策略，包括基于句子的、基于段落的、以及自定义规则的分块方式。

**第三阶段：extract_graph（知识图谱抽取）**

这是管线的核心阶段。LLM 被要求根据 prompt 模板从每个 chunk 中抽取知识图谱结构，输出格式为：

```
KnowledgeGraph {
  nodes: [
    Node { id, name, type, description }
  ],
  edges: [
    Edge { source, target, relationship_name }
  ]
}
```

该 prompt 的典型表述为 "extract Nodes(entities) and Edges(relationships) from text"，要求 LLM 识别文本中的实体节点和关系边，并输出结构化的图谱数据。这一步骤使得非结构化文本转化为可推理的知识结构。

**第四阶段：add_data_points（添加数据点）**

从图谱抽取的数据点被添加到索引系统中。此阶段处理节点的属性信息，包括 id、name、type、description 等字段，为后续的搜索和推理提供元数据支持。

**第五阶段：extract_fk_edges（外键关系抽取）**

最后，管线识别并建立节点之间的外键关系边。这些边构成了知识图谱的推理基础，使得基于关系的查询和图遍历成为可能。

### 多存储后端架构

Cognee 采用了 poly-store 架构设计，同时支持三类存储后端，且每一类都提供了多个实现选项：

**Graph Storage（图存储）**

支持 Neo4j、Kuzu、PostgreSQL 三种图数据库。用户可以根据性能需求、部署复杂度、运维能力选择合适的图存储后端。这种设计避免了被单一图数据库绑定，同时允许在原型阶段使用轻量级方案（如 Kuzu），在生产环境切换到成熟方案（如 Neo4j）。

**Vector Storage（向量存储）**

支持 Chroma、LanceDB、pgvector 三个向量数据库。向量存储主要用于 semantic search 和 RAG 场景，存储文本块的向量表示以支持相似度检索。

**Relational Storage（关系存储）**

支持 SQLite 和 PostgreSQL。关系存储用于存储结构化的元数据和事务性数据，补充图存储和向量存储的能力边界。

这种三层存储分离的设计哲学是：Cognee 认为 AI 应用的数据存储需求是多元的，没有任何单一数据库能够同时最优地满足图遍历、向量搜索和事务性操作的需求。因此它选择让用户根据具体场景组合使用多种存储后端，而非强行统一到一种方案。

### 搜索模式（5种）

Cognee 提供了五种独立的搜索模式，分别针对不同的检索需求：

**GRAPH_COMPLETION**

结合图遍历和 LLM 的搜索模式。系统首先在知识图谱中进行图遍历，收集相关的节点和边，然后将这些结构化信息作为上下文提供给 LLM，生成最终答案。这种模式适合需要推理和多跳查询的场景。

**RAG_COMPLETION**

传统的向量检索增强生成模式。系统通过向量相似度找到最相关的文本块，然后将这些块作为上下文提供给 LLM。这种模式适合事实性问答和基于文本内容的查询。

**CHUNKS**

原始块检索模式，直接返回与查询最相关的文本块，不经过 LLM 处理。适合需要人工审查或二次处理的场景。

**SUMMARIES**

文档摘要检索模式，返回文档的摘要信息而非完整内容。适合需要快速了解文档概览的场景。

**CYPHER**

直接图查询模式，用户可以编写 Cypher 查询语句直接遍历知识图谱。适合高级用户和需要精确图查询的场景。

这五种模式覆盖了从原始检索到 LLM 推理的完整光谱，用户可以根据场景需求选择最适合的搜索策略。

### 后台刷新管线

Cognee 提供了名为 Memify 的后台刷新机制，用于异步更新知识库中的数据点。其核心流程为：

```
get_triplet_datapoints → index_data_points
```

该管线异步执行，定期从数据源获取最新的三元组数据点，并将其索引到存储后端中。这种设计使得知识库能够保持一定的新鲜度，而无需每次都执行完整的 ETL 管线。Memify 适合用于持续更新的数据源场景，例如新闻流、社交媒体数据、或频繁变化的业务数据。

## 核心设计思想

### 为什么选择管线而非 API？

Cognee 选择管线架构的核心原因在于知识抽取过程的本质复杂性。与简单的 key-value 存储或直接的向量检索不同，从非结构化文本中抽取结构化知识涉及多个决策点：如何分块、如何抽取实体、如何建立关系、如何处理歧义。这些决策高度依赖于数据类型和应用场景，无法通过单一 API 调用完成。

管线架构将这个复杂过程分解为可配置、可观测的阶段。每个阶段都可以独立调优、替换或跳过。例如，用户可以保留 classify 和 chunk 阶段，但使用自定义的 extraction prompt 来适应特定领域的知识抽取需求。这种灵活性是单一 API 设计无法提供的。

同时，管线架构天然适合处理大规模数据。通过将处理过程分解为阶段，Cognee 可以支持增量处理、检查点重启、并行处理等工程实践，这对于生产环境至关重要。

### 为什么支持这么多存储后端？

Cognee 的多存储后端设计反映了其对 AI 应用数据需求多样性的认知。不同的存储系统擅长不同的操作：图数据库擅长多跳关系推理，向量数据库擅长语义相似度搜索，关系数据库擅长精确条件查询和事务处理。没有一个系统能够在所有维度上最优。

Cognee 的策略是提供选择权而非做选择。它将三种类型的存储后端都开放给用户，让用户根据具体应用场景的访问模式、规模需求、运维能力来做决策。这种设计也降低了迁移成本——用户可以从 SQLite/Chroma 开始，原型验证后再切换到 PostgreSQL/Neo4j，而无需修改业务代码。

### 离线优先的意义

Cognee 支持完全离线运行，通过集成 Ollama 实现本地 LLM 推理。这一设计选择对于企业应用场景至关重要：数据隐私要求敏感信息不得离开企业网络，合规要求限制了云端 API 的使用，可靠性要求降低了网络依赖。

离线优先不仅意味着隐私保护，还意味着更高的可用性和更低的运营成本。Cognee 的用户可以在没有网络连接的环境中运行完整的 pipeline，这种能力在特定行业（如医疗、金融、政府）中是刚性需求。

## 优势分析

**最灵活的数据摄入能力**

Cognee 支持 30+ 数据源连接器，覆盖文本、PDF、图片、音频转录等多种格式。这是调研系列中摄入能力最强的框架。对于需要整合多源异构数据的 AgentCenter 场景，这一能力具有直接的实用价值。

**Poly-store 架构不锁定供应商**

用户可以自由选择图数据库、向量数据库和关系数据库的组合，也可以后续切换存储后端而不修改业务代码。这种架构灵活性降低了供应商锁定风险。

**完全离线运行能力**

集成 Ollama 后，Cognee 可以完全在本地运行，不依赖任何云端服务。这对于需要数据本地化的企业场景是重要优势。

**管线架构可扩展**

每个管线阶段都可以独立配置、替换或扩展。用户可以添加自定义的 classification 策略、chunking 规则、extraction prompt，甚至引入全新的存储后端。这种架构适合需要深度定制的场景。

**同时处理结构化和非结构化数据**

Cognee 不仅能处理非结构化文本，还能处理结构化数据输入。这种多模态数据处理能力使其适合复杂的企业数据环境。

## 局限与不足

**配置复杂度较高**

Cognee 的灵活性带来的副作用是配置复杂度较高。用户需要理解管线各阶段的作用，选择合适的存储后端组合，处理版本兼容性问题。这对于简单场景可能是过度设计。

**管线延迟不适合实时场景**

数据从摄入到可搜索需要经过完整的管线过程，存在明显的延迟。对于需要实时记忆更新的对话场景，Cognee 的管线架构可能不是最优选择。

**社区规模相对较小**

相比 Mem0 和 Letta，Cognee 的社区规模和活跃度较低。这意味着更少的社区支持、更少的第三方集成、更慢的 bug 修复和问题响应。

**缺乏时间推理能力**

与 Letta 的 temporal memory 相比，Cognee 没有显式的时间推理机制。对于需要记忆时间上下文（如"上次讨论的内容"）的场景，需要用户自行设计和实现时间维度。

**API 成熟度待提升**

相比一些竞品，Cognee 的 API 设计不够统一和一致。用户可能需要花费更多时间来理解和调试 API 使用方式。

## 与 AgentCenter 的契合度

从架构特性来看，Cognee 与 AgentCenter 的契合度处于中等水平。

**高契合度场景**

AgentCenter 如果需要处理多源异构数据、进行复杂的知识抽取、建立持久的知识图谱，Cognee 的管线架构和多存储后端设计提供了强大的支持。特别是当 AgentCenter 需要整合来自文档、邮件、数据库等多种来源的信息时，Cognee 的 30+ 数据源连接器可以显著加速开发。

**中契合度场景**

对于 AgentCenter 的工作流编排功能，Cognee 的异步管线模型与工作流的批处理特性天然契合。Memify 的后台刷新机制也可以用于定期更新知识库。

**低契合度场景**

如果 AgentCenter 需要支持实时对话中的记忆更新，Cognee 的管线延迟是一个障碍。同样，如果 AgentCenter 的主要交互模式是简单的 key-value 记忆存取，Cognee 的重量级设计会造成不必要的复杂度。

**建议**

对于 AgentCenter，建议将 Cognee 作为知识管理模块的核心组件，用于处理文档理解、知识图谱构建、多源数据整合等后台任务。同时，应该评估是否需要额外的实时记忆层来满足对话场景的低延迟需求。
