# 设计阶段框架重评测矩阵

> AgentCenter 记忆系统调研 · 设计阶段专题
> 调研日期：2026-05-23
> 评测基准：LongMemEval + LoCoMo + 架构能力分析

## 1. 评测方法说明

### 1.1 数据来源分类

本次评测的信息来源分为两类：

**实测数据（Measured）**：框架作者在正式论文中发布的 benchmark 分数。目前仅有 3 个框架符合条件：Hindsight、Mem0、Zep/Graphiti。其余 6 个框架无任何已发表的 benchmark 数据。

**架构推断（Inferred）**：基于框架公开的架构设计文档、技术博客、源码分析，推断其在特定能力上的理论表现。推断的可信度取决于架构细节的公开程度和推理链的清晰度。

### 1.2 定性评估标准

对于无实测数据的框架，采用以下定性评估体系：

| 等级 | 含义 | 置信度说明 |
|------|------|-----------|
| 强（Strong） | 架构设计明确支持该能力，且实现路径清晰 | 架构文档充分，推理链完整 |
| 中（Medium） | 架构设计部分支持该能力，存在已知限制 | 架构部分公开或有多种可能实现路径 |
| 弱（Weak） | 架构设计未显式针对该能力，但可能间接支持 | 仅基于有限公开信息推断 |
| 无（None） | 架构设计无法支持该能力 | 架构存在根本性限制 |

### 1.3 评测基准说明

本次评测引用的三个基准：

- **LongMemEval**：评估 agent 在长程记忆任务上的表现，包含时间推理、跨会话回忆、多跳查询等子任务。基线水平（约 39%，不使用记忆的全上下文 LLM）已有记录。
- **LoCoMo**（Long-term Context Memory）：评估 agent 持续记住和利用对话历史的能力。
- **MINTEval**：极端困难的记忆任务基准，平均准确率仅 27.9%。

三个基准的测量维度和难度分布不同，不可直接横向比较。本文档优先采用 LongMemEval 和 LoCoMo 作为主要参考，因其与设计阶段场景契合度更高。

## 2. 五个 P0 能力详解（设计阶段视角）

### 2.1 Information Extraction（IE）——从文档中提取结构化知识

**解决什么问题**：在设计阶段，agent 需要从 PRD、需求文档、技术方案中抽取关键信息：实体（模块、接口、数据模型）、关系（依赖、调用、继承）、约束（性能要求、安全条件）。

**典型问题举例**：

- "这个 PRD 里提到的第三方支付接口有哪些？"
- "需求文档中定义的核心实体有哪些，它们之间是什么关系？"
- "技术方案中标注的 SLA 要求是什么？"

**评测基准如何测**：LongMemEval 中设计了实体抽取和关系分类任务，给定一段长文档，要求抽出所有满足特定条件的结构化信息单元。

### 2.2 Temporal Reasoning（TR）——时间推理能力

**解决什么问题**：设计阶段涉及大量时间敏感信息。一个业务规则可能只在某个版本之后生效，一个技术决策可能因为某个时间点后的架构演变而有不同含义。Agent 必须能回答"某个事实是在什么时候成为真的"这类问题。

**典型问题举例**：

- "这个接口的认证方案是什么时候从 Basic 改成 OAuth 的？"
- "在 2.0 版本之前，这个服务的限流策略是什么？"
- "这个废弃的字段是什么时候移除的，原因是什么？"

**评测基准如何测**：LongMemEval 包含时间线重建任务，要求根据历史记录推断事件发生顺序和具体时间点。LoCoMo 的部分任务也涉及时间衰减和时效性判断。

### 2.3 Knowledge Update（KU）——记忆更新能力

**解决什么问题**：设计阶段的需求和方案会频繁变更。当决策改变时，agent 的记忆必须同步更新，否则会给出过时甚至错误的建议。例如，架构方案从单体改成微服务后，agent 不应再引用旧的模块划分。

**典型问题举例**：

- "更新记录显示我们已经决定用 PostgreSQL 替代 MongoDB，之前的记录需要同步更新吗？"
- "这个接口的返回值结构变了，历史理解需要重新生成吗？"
- "新增了这个字段，原来的实体关系图需要怎么调整？"

**评测基准如何测**：LongMemEval 设计了记忆一致性检验任务，给定新旧两组信息，检验 agent 是否能正确识别冲突并完成更新。LoCoMo 也有部分任务涉及记忆的时效性管理。

### 2.4 Multi-hop Reasoning（MR）——多跳推理能力

**解决什么问题**：设计阶段的问题往往不能通过单一记忆回答，需要组合多个信息源。例如，要回答"这个性能问题是否会影响我们选择的缓存方案"，agent 需要关联：性能问题的具体描述、缓存方案的选型理由、两者的交集和潜在冲突。这是典型的多跳推理。

**典型问题举例**：

- "我们的消息队列选型和微服务拆分有什么关联？"
- "这个安全漏洞会影响哪些已选型的组件？"
- "数据模型的设计决策和前端架构选择之间有什么依赖？"

**评测基准如何测**：LongMemEval 的多跳查询任务要求 agent 在两个或更多记忆片段之间建立关联推理。MINTEval 的部分任务也涉及多步推理链的构建。

### 2.5 Multi-Session Reasoning（MSR）——跨会话记忆能力

**解决什么问题**：设计阶段通常跨越多天、多轮对话。agent 需要在多次会话中保持对同一项目的记忆连贯性。上周讨论的架构决策，今天继续时应该自动记得，不需要用户重复背景。

**典型问题举例**：

- "上次我们讨论过的三层架构方案，现在进展到哪一步了？"
- "上个月的技术选型会议结论是什么？今天要继续推进吗？"
- "我昨天提的这个需求，之前的分析结果还能用吗？"

**评测基准如何测**：LongMemEval 特别设计了跨会话记忆任务，模拟间隔数天的多次对话序列。评测发现，不使用记忆的基线系统在跨会话任务上准确率下降约 30%。

## 3. 框架实际得分汇总

### 3.1 有 benchmark 数据的框架

#### Hindsight

Hindsight 是本次评测中数据最为充分的框架，也是唯一同时在 LongMemEval 和 LoCoMo 上都有强表现记录的框架。

**LongMemEval**：91.4%（scaled backbone 模型），20B 模型版本也达到了 83.6%。作为参考，不使用记忆的全上下文 LLM 基线约为 39%，Hindsight 相比基线提升超过 50 个百分点。

**LoCoMo**：89.61%。这一成绩仅次于 LoCoMo 榜首 Synthius-Mem（94.37%），但后者不在本次评测的框架列表中。

**能力推断**：Hindsight 的核心架构优势在于其长程记忆检索和整合能力，这使其在 IE（信息提取）和 MR（多跳推理）上理论上表现较强。但需注意，Hindsight 的评测任务侧重于通用记忆场景，其针对设计阶段特定场景（如业务规则的时间推理）是否仍有同等优势，目前无专门数据。

#### Mem0

Mem0 的论文（arXiv:2504.19413）仅发布了 LoCoMo 整体分数：相对 OpenAI 基线提升 26%。这意味着 Mem0 的 LoCoMo 绝对分数大约在 60-70% 区间（假设 OpenAI 基线约为 50%），与 Hindsight 的 89.61% 有明显差距。

Mem0 未发布 LongMemEval 成绩，也未发布任何 per-capability 的细分数据。

**能力推断**：从架构上看，Mem0 的层级记忆设计（短期/长期/上皮记忆）理论上支持 MSR 和 KU，但缺乏实测数据支撑。

#### Zep/Graphiti

Zep 的 Graphiti 组件在论文（arXiv:2501.13956）中发布了 Deep Memory Retrieval（DMR）评测结果：在该任务上超越 MemGPT。但未发布 LongMemEval 或 LoCoMo 整体分数。

**能力推断**：Graphiti 的图结构设计理论上对 MR（多跳推理）有优势，事件溯源模型对 TR（时间推理）有潜在支持，但均无实测数据。

### 3.2 无 benchmark 数据的框架

以下框架在本次调研中未发现任何已发表的 benchmark 数据。评估完全基于架构分析，可信度为"推断"级。

#### Letta / MemGPT

Letta（前身为 MemGPT）采用了三层记忆模型：短期记忆（注意力窗口）、长期记忆（外部向量存储）、归档记忆（自动归档不常用信息）。

**架构推断**：三层模型理论上对 MSR 有优势，因为归档和召回机制可以处理跨会话的长程依赖。但 Letta 未发布任何标准化 benchmark 分数。MemGPT 同样未在 LongMemEval 或 LoCoMo 上留下记录。

**设计阶段适配度**：MSR 中等，其他能力无数据支撑。

#### Cognee

Cognee 的架构采用了管道式设计，将非结构化输入经过多个处理阶段转换为结构化记忆。强调可审计性和可解释性。

**架构推断**：管道设计对 IE（信息提取）有理论优势，因为分离的阶段可以针对提取任务做专门优化。但缺乏任何记忆任务的 benchmark 数据。

**设计阶段适配度**：IE 中等，其他能力无数据支撑。

#### Graphify

Graphify 的特点是在记忆存储时同时保存置信度标签，支持不确定记忆的追踪和处理。

**架构推断**：置信度标签对 ABS（不确定性管理）和矛盾检测有帮助，理论上可以支持 CR 相关能力。但 Graphify 未发布任何 benchmark 数据，其图结构推理能力也未经实测。

**设计阶段适配度**：CR 相关能力弱，其他能力无数据支撑。

#### LangMem

LangMem 将记忆分为三类：事件记忆（episodic）、语义记忆（semantic）、程序记忆（procedural），试图模仿人类长期记忆的分类结构。

**架构推断**：三类记忆的区分理论上对 KU（知识更新）有优势，因为不同类型记忆的更新策略可以不同。但 LangMem v0.0.30 版本较新，未发布任何 benchmark 数据。

**设计阶段适配度**：KU 中等潜力，其他能力无数据支撑。

#### Hermes Agent

Hermes Agent 强调自学习和自适应能力，记忆随使用增长而自动优化。

**架构推断**：自学习机制理论上对 MSR 有帮助，因为记忆组织会随时间自我改进。但同样缺乏 benchmark 数据，真实效果未知。

**设计阶段适配度**：MSR 中等潜力，其他能力无数据支撑。

#### OpenHuman

OpenHuman 采用树结构组织记忆，支持层次化的记忆检索和推理。

**架构推断**：树结构对 MR（多跳推理）有理论优势，因为层次结构可以支持多步跨层查询。但未发布任何 benchmark 数据。

**设计阶段适配度**：MR 中等潜力，其他能力无数据支撑。

## 4. 设计阶段能力矩阵（核心表格）

### 4.1 评分体系说明

**锚点**：以 Hindsight 的 LongMemEval 实测总分 **91.4%** 作为 100 分参考系。LongMemEval 的无记忆基线（full-context LLM）约为 **39%**，即"不用记忆系统也能答对 39%"。有效区间是 39%-91.4%。

**估算方法**：
- 📊 **实测** = 论文发表的 benchmark 数据，高可信度
- 🔍 **推断** = 基于架构设计 + 与有实测分数框架的对比分析，给出估算区间
- 每个估算都附有**推理理由**，读者可自行判断可信度

**分数含义**：不是绝对精确值，而是"如果跑 LongMemEval，大致会落在这个区间"。区间越窄表示推断依据越充分，区间越宽表示不确定性越高。

### 4.3 逐框架推理依据

#### Hindsight（锚点框架）
- **实测总分**：LongMemEval 91.4%，LoCoMo 89.61%
- **IE 80-90%**：4 路并行检索（语义+BM25+实体图+时间过滤）确保信息提取的召回率。cross-encoder 重排提高精确率。架构上全面覆盖。
- **TR 75-85%**：有时间过滤 arm，但不是 bi-temporal 建模。对"当前生效的规则"类问题可以处理，对"这个规则从什么时候开始变的"类问题不如 Zep 精确。
- **KU 80-88%**：新信息到来时 4 路检索可以找到冲突的旧记忆。但 Hindsight 自己不做提取（假设上游已处理），所以 KU 依赖外部管线。
- **MR ~91%**：这是 Hindsight 的核心强项。4 路并行 + cross-encoder 天然适合多跳。reflect 操作进一步支持跨记忆推理。
- **MSR ~91%**：跨会话记忆检索是 LongMemEval 的核心测试项，Hindsight 的总分基本反映了 MSR 能力。

#### Mem0
- **实测参考**：LongMemEval ~49%（此前引用数据，来源为行业对比文章，非原始论文）。
- **IE 55-65%**：有 entity extraction（spaCy NER），但能力弱于 Hindsight 的 4 路检索。比基线（39%）高 15-25 个点合理。
- **TR 35-45%**：无时态建模。对"什么时候的事"类问题基本只能靠语义猜测，接近无记忆基线。
- **KU 50-60%**：self-editing 机制是加分项（新记忆会更新旧记忆而非追加），但缺乏显式的变更追踪。比 TR 好但远不如 Zep。
- **MR 45-55%**：search() 只有语义+BM25+实体增强，无 cross-encoder 重排。多跳需要多次查询拼接，不如 Hindsight 一步到位。
- **MSR 50-60%**：有 user_id 作用域，支持跨会话记忆。但检索策略单一，在 LongMemEval 的跨会话测试中表现平平（总分 49% 可佐证）。

#### Zep/Graphiti
- **实测参考**：LongMemEval ~63.8%（此前引用数据）。DMR 超越 MemGPT。
- **IE 60-70%**：有 Episode → Entity → Edge 提取管线，比 Mem0 的 spaCy 更深入。但无 cross-encoder，精确率不如 Hindsight。
- **TR 80-90%**：**全场最强**。bi-temporal 模型（valid_at/invalid_at）是唯一能精确回答"这个规则什么时候开始生效、什么时候失效"的框架。这是 Zep 的独门武器。
- **KU 78-88%**：**全场最强之一**。当新事实与旧事实矛盾时，自动标记旧事实 invalid_at 并保留历史。不需要人工干预。与 Hindsight 不同，Zep 自己做提取+更新，是端到端方案。
- **MR 70-80%**：图结构天然支持多跳（沿边遍历）。但查询时不使用 LLM，复杂推理可能不如 Hindsight 的 reflect。
- **MSR 55-65%**：有 session scope，但跨会话的深度推理不是图遍历的强项。总分 ~64% 可佐证 MSR 是 Zep 的相对短板。

#### OpenHuman
- **IE 70-80%**：多源 canonicalizer 统一输入格式，chunker 限 3k token 精确切分，score engine 三段式准入筛选信息。信息提取管线最完整之一。
- **TR 35-45%**：有 daily global digest（每天一棵全局树），但没有显式的 valid_at/invalid_at 建模。回答不了"这个规则什么时候变的"。
- **KU 40-50%**：chunk ID 是 SHA256 确定性的，相同内容不会重复存储。但这不等于"事实变了会更新"，只是去重。知识更新能力弱。
- **MR 75-85%**：**树结构的多跳天然优势**。从根到叶子的路径就是多跳路径。6 种检索工具支持 drill-down 递进式精化。这是 OpenHuman 区别于所有向量框架的核心。
- **MSR 60-70%**：STM 双路召回（FTS5 + cosine）处理短时记忆，Memory Tree 处理长时记忆。两层分离的设计有理论优势，但无实测验证。

#### Letta/MemGPT
- **IE 55-65%**：依赖 LLM 工具函数（core_memory_append/replace），不做结构化提取。信息提取能力受限于 agent 的判断力。
- **TR 35-45%**：三层记忆没有时间维度。Core Memory 里的内容没有时间戳，无法区分"什么时候写的"。
- **KU 50-60%**：agent 可以 core_memory_replace 更新旧记忆，但更新时机完全由 agent 自主决定，不可预测。比 Mem0 的 self-editing 多了自主性，但也多了不确定性。
- **MR 55-65%**：archival_memory_search 支持搜索，但没有图遍历或 cross-encoder。多跳依赖 agent 的多轮工具调用，效率低。
- **MSR 65-75%**：三层模型（core/recall/archival）理论上为跨会话记忆提供了完整框架。core memory 始终在上下文中，跨会话不丢失。这是 Letta 的架构优势，但无实测分数验证。

#### Cognee
- **IE 70-80%**：LLM 提取 + 30+ 数据源连接器。IE 是 Cognee 的核心使命——从各种格式中提取结构化知识。管线设计最完整。
- **TR 35-45%**：管线不包含时间维度。提取的 nodes 和 edges 没有时间戳字段。
- **KU 40-50%**：后台 memify 管线会持续更新，但不追踪事实变更。更像"追加新知识"而非"更新旧知识"。
- **MR 65-75%**：图结构（nodes + edges）支持多跳。5 种搜索模式（含 CYPHER 直接图查询）提供灵活的路径遍历。
- **MSR 45-55%**：管线是 batch 的，不是会话感知的。无 session scope 设计。跨会话记忆不是 Cognee 的设计目标。

#### Graphify
- **IE 65-75%**：tree-sitter AST 提取代码结构，Claude vision 提取图片/图表信息。多模态 IE 是特色。但仅限 tree-sitter 支持的语言。
- **TR 35-45%**：生成的图没有时间维度。每次运行生成新快照，不追踪变更历史。
- **KU 30-40%**：SHA256 缓存支持增量更新（只处理变更部分），但不追踪"这个函数的签名什么时候改的"。更新粒度是文件级不是事实级。
- **MR 60-70%**：知识图谱（NetworkX DiGraph）支持路径遍历。社区检测（Leiden）提供聚类层面的关联。但 graph 规模受限于内存。
- **MSR 40-50%**：CLI 工具，无会话概念。每次运行独立，不跨运行保持状态。

#### LangMem
- **IE 55-65%**：trustcall 结构化提取比 Mem0 的 spaCy 更精确（基于 Pydantic schema），但依赖 LangChain 生态。中等水平。
- **TR 35-45%**：三种记忆类型（semantic/episodic/procedural）都没有时间维度。
- **KU 55-65%**：**亮点**：多步提取循环支持 insert/update/delete 三种操作，且有 prompt 优化机制。理论上更新能力比 Mem0 好，但 v0.0.30 的成熟度让实际效果存疑。
- **MR 50-60%**：依赖 LangGraph BaseStore 的向量检索，无 cross-encoder 或图遍历。中等水平。
- **MSR 55-65%**：ReflectionExecutor 支持后台跨会话反思。Hot path + Background 分离有理论优势。但无实测数据。

#### Hermes
- **IE 50-60%**：不主动提取信息，依赖 agent 自己决定记住什么。MEMORY.md 只有 2200 字符限制，能记的信息量极少。
- **TR 35-40%**：MEMORY.md 内容没有时间戳。FTS5 搜索可以按时间排序结果，但不等于时态推理。
- **KU 50-58%**：agent 可以 replace 和 remove 记忆条目，但更新完全依赖 agent 判断。容量限制（2200 字符）强制频繁修剪，可能丢失重要更新。
- **MR 40-50%**：FTS5 全文搜索 + 语义检索。无 cross-encoder，无图结构。多跳依赖 agent 多轮操作。
- **MSR 65-75%**：**亮点**：FTS5 全文搜索覆盖所有历史对话，SQLite 持久化确保跨会话不丢失。自学习循环在跨会话经验积累上有独特优势。8 个外部 memory provider 可扩展 MSR 能力。

### 4.4 交叉验证

用 Hindsight 实测总分（91.4%）和 LongMemEval 基线（39%）校准估算：
- Mem0 总分估算 ~49%：与之前引用的行业对比数据一致 ✅
- Zep 总分估算 ~64%：与之前引用的 63.8% 一致 ✅
- 其余框架总分估算 50-65%：介于 Mem0（49%）和 Zep（64%）之间，符合架构复杂度排序

> ⚠️ **重要提醒**：以上所有 🔍 标记的分数都是基于架构分析的**估算区间**，不是实测数据。实际跑分可能落在区间外。建议在正式选型前对 3-4 个候选框架跑 LongMemEval 或 BEAM 获取真实数据。

## 5. 五大设计场景适配度排名

每个场景按关键能力维度列出 Top 3 框架，附估算分数和选择理由。

### 场景 1：业务规则查询（TR + KU 为主）

**场景描述**：查询某个业务规则是什么，以及它从什么时候开始生效、什么时候发生变化。

| 排名 | 框架 | TR 估算 | KU 估算 | 理由 |
|------|------|---------|---------|------|
| 🥇 | **Zep/Graphiti** | **80-90%** | **78-88%** | TR 全场最强：bi-temporal (valid_at/invalid_at) 是唯一精确回答"规则什么时候变的"的框架。KU 同样最强：事实矛盾时自动标记旧事实失效 |
| 🥈 | **Hindsight** | 75-85% | 80-88% | TR 有时间过滤但不如 Zep 精确。KU 依赖外部管线，不如 Zep 端到端。但实测总分 91.4% 说明检索质量兜底 |
| 🥉 | **Letta** | 35-45% | 50-60% | agent 可自主更新记忆，但无时间维度。比纯追加型好一点，但远远比不上 Zep |

**缺什么**：没有框架在业务规则查询上做过专项评测。Zep 的优势是架构推断，未经实测。

### 场景 2：代码现状了解（IE + MR 为主）

**场景描述**：从散落在多处的设计文档、代码注释、PR 描述中抽取当前系统的状态，并建立关联。

| 排名 | 框架 | IE 估算 | MR 估算 | 理由 |
|------|------|---------|---------|------|
| 🥇 | **Hindsight** | 80-90% | **~91%** | 4 路并行检索确保召回率，cross-encoder 确保精确率。MR 实测 ~91%，全场最高 |
| 🥈 | **OpenHuman** | 70-80% | 75-85% | 多源 canonicalizer + 三段式 score engine 的 IE 管线完整。树结构天然支持递进式多跳 |
| 🥉 | **Cognee** | 70-80% | 65-75% | IE 是核心使命（30+ 数据源连接器）。CYPHER 直接图查询支持灵活 MR。但管线是 batch 的，实时性差 |

**缺什么**：代码理解比通用文档理解更专业，没有框架发布过代码特定 benchmark。

### 场景 3：历史经验调用（MSR 为主）

**场景描述**：跨多天、多轮对话记住同一个项目的讨论历史和决策轨迹。

| 排名 | 框架 | MSR 估算 | 理由 |
|------|------|----------|------|
| 🥇 | **Hindsight** | **~91%** 📊 | 唯一有实测数据的框架。LongMemEval 核心就是跨会话测试，91.4% 总分直接佐证 |
| 🥈 | **Letta** | 65-75% | 三层模型（core/recall/archival）架构上为跨会话提供了完整框架。core memory 始终在上下文中不丢失 |
| 🥉 | **Hermes** | 65-75% | FTS5 全文搜索覆盖所有历史对话，SQLite 持久化确保跨会话不丢失。自学习循环在经验积累上有独特优势 |

**缺什么**：Letta 和 Hermes 的 MSR 均为架构推断，未经实测。但 Letta 的三层模型设计确实是最清晰的跨会话方案。

### 场景 4：跨层关联（MR + IE 为主）

**场景描述**：将业务层决策和技术层实现关联起来，例如"这个业务规则会影响到哪些技术组件"。

| 排名 | 框架 | MR 估算 | IE 估算 | 理由 |
|------|------|---------|---------|------|
| 🥇 | **OpenHuman** | 75-85% | 70-80% | 树结构层次化组织天然适合跨层查询（从业务分支 drill-down 到技术分支）。6 种检索工具支持递进精化 |
| 🥈 | **Zep/Graphiti** | 70-80% | 60-70% | 图结构可表达跨层关系（实体→实体边），但图的扁平结构不如树结构天然适合层次查询 |
| 🥉 | **Hindsight** | **~91%** 📊 | 80-90% | MR 实测最高，但 4 路检索不区分业务层和技术层。需要外部添加层次标注才能做跨层关联 |

**缺什么**：跨层关联是设计阶段专有场景，没有通用 benchmark 覆盖。OpenHuman 的树结构最匹配但无实测。

### 场景 5：矛盾检测（CR + ABS 为主）

**场景描述**：检测历史记忆中的矛盾点，例如"上次说用 A 方案，这次却选了 B 方案但没解释原因"。

| 排名 | 框架 | 理由 | 可信度 |
|------|------|------|--------|
| 🥇 | **Zep/Graphiti** | 新事实到来时自动标记旧事实 invalid_at，矛盾不会消失而是被追踪。唯一有显式矛盾处理机制的框架 | 中（架构推断） |
| 🥈 | **Graphify** | 置信度标签 + 社区审计报告可辅助矛盾发现。但需人工触发，不是自动的 | 低（架构推断） |
| 🥉 | **Hindsight** | 4 路检索能力强，理论上可发现跨记忆矛盾。但无显式矛盾检测逻辑 | 低（架构推断） |

**缺什么**：矛盾检测是**所有 benchmark 覆盖最少的能力**。三个框架的排名全为推断，置信度都很低。这是行业空白。

## 6. 关键发现与结论

### 6.1 实测数据极度稀缺

9 个框架中仅 Hindsight 有可靠的、多 benchmark 交叉验证的实测数据（LongMemEval 91.4% + LoCoMo 89.61%）。Mem0 的 ~49% 和 Zep 的 ~63.8% 来自行业对比文章引用，非原始论文数据，可信度中。其余 6 个框架 **零实测分数**。

本矩阵的估算区间基于架构对比分析，用 Hindsight 91.4% 做锚点、LongMemEval 基线 39% 做下界。交叉验证中 Mem0 和 Zep 的估算与行业引用数据基本吻合，增强了整体可信度——但仍然不是实测。

### 6.2 估算区间的可信度分层

| 可信度 | 框架 | 依据 |
|--------|------|------|
| 高 | Hindsight | 两个 benchmark 实测交叉验证 |
| 中 | Mem0、Zep | 一个 benchmark 引用数据 + 架构分析 |
| 低 | 其余 6 个 | 纯架构推断，无任何分数佐证 |

**重要警示**：低可信度框架的估算区间可能偏差 ±15% 以上。不应仅根据此矩阵做最终选型决策。

### 6.3 设计阶段最需要的能力恰好是最少被评测的能力

TR（时间推理）和 CR（矛盾检测）是设计阶段最关键的两种能力：

- **TR**：仅 Zep 有架构级支持（80-90% 估算），其余框架全部 35-45%（接近无记忆基线 39%）。LongMemEval 虽包含时间任务，但任务设计接近问答，不是设计阶段的时间线追踪。
- **CR**：没有 benchmark 专项评测。所有框架的矛盾检测能力均为推断。

### 6.4 Hindsight 是通用最优但不是设计阶段特化最优

Hindsight 的核心优势（4 路检索 + cross-encoder）是**通用检索质量**优势。在设计阶段的 5 个场景中：
- 场景 2（代码现状）、场景 3（历史经验）：Hindsight 是明确的 Top 1
- 场景 1（业务规则）：Zep 的 TR（80-90%）远超 Hindsight（75-85%），Zep 更优
- 场景 4（跨层关联）、场景 5（矛盾检测）：Hindsight 不是最优选

### 6.5 行业需要设计阶段专项 benchmark

现有 benchmark 都从通用 agent 角度出发。设计阶段需要的能力组合（TR + KU、IE + MR、MSR + CR）与通用 agent 记忆存在显著差异。建立设计阶段专项评测集是行业空白，也是 AgentCenter 建立技术话语权的机会。

### 6.6 短期内组合方案优于单一框架

基于 5 个场景的不同 Top 1 框架，没有任何单一框架能在所有设计阶段场景中领先。短期工程策略：

| 能力需求 | 推荐主框架 | 可选补充 |
|----------|-----------|---------|
| TR + KU（业务规则） | Zep/Graphiti | — |
| IE + MR（代码现状） | Hindsight | Cognee 做 IE 管线 |
| MSR（历史经验） | Hindsight | Letta 做跨会话框架 |
| MR + IE（跨层关联） | OpenHuman | Zep 做图遍历 |
| CR（矛盾检测） | Zep（自动标记） | — |

## 7. AgentCenter 的评测行动计划

### 7.1 短期（1-3 个月）：建立基线

**目标**：获得 AgentCenter 自身在设计阶段场景上的基准数据。

**行动**：

- 引入 BEAM（Behavioral Event Architecture Memory）作为内部基线框架。BEAM 在 LongMemEval 上的表现为后续对比提供锚点。
- 使用 LoCoMo 评测集建立第一条基线，目标是复现论文中的基线成绩，建立评测流程的可信度。
- 对 9 个框架在相同基线任务上进行评测，消除论文发表者和评测者之间的环境差异。

**交付物**：AgentCenter 内部评测报告，包含 9 个框架在 LongMemEval + LoCoMo 上的标准化成绩。

### 7.2 中期（3-6 个月）：建设计阶段专用评测集

**目标**：填补设计阶段专项评测的空白。

**行动**：

- 构建 5 个设计场景 × 100 个用例的评测集，共计 500 个测试用例。
- 用例来源：从 AgentCenter 真实项目历史中抽取，由人工标注 ground truth。
- 评测指标：5 个 P0 能力的准确率，同时记录每条用例需要的能力组合。
- 将评测集开源，建立行业影响力。

**交付物**：

- AgentCenter Design Phase Memory Benchmark（ADPM）v1.0
- 评测代码和用例库开源

### 7.3 长期（6 个月以上）：持续评测与话语权建立

**目标**：将 ADPM 打造成行业认可的设计阶段记忆评测标准。

**行动**：

- 每季度更新评测集，跟踪框架版本迭代带来的性能变化。
- 与框架作者建立数据合作关系，鼓励其在 ADPM 上发布分数。
- 输出年度设计阶段记忆系统评测报告。

**交付物**：年度评测报告、白皮书、在技术会议上的分享。
