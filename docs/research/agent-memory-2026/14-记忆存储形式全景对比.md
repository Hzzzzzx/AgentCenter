## 1. 六大存储介质对比

| 维度 | 向量数据库 (Vector DB) | 图数据库 (Graph DB) | 关系型数据库 (Relational DB) | 文件系统 (File-based) | 内存图 (In-Memory Graph) | 混合存储 (Hybrid) |
|------|------------------------|---------------------|-----------------------------|----------------------|------------------------|------------------|
| **代表框架** | Mem0 (Qdrant), Engram (FAISS/Qdrant), agentmemory (SQLite+HNSW) | Zep/Graphiti (Neo4j), Cognee, MemMachine (Neo4j) | Letta (PostgreSQL), Cortex (PostgreSQL+pgvector), total-agent-memory (SQLite), agentmemory (SQLite) | Synthius-Mem (域文件), OpenCode (AGENTS.md), Claude Code (CLAUDE.md), Codex (AGENTS.md+memories/), Windsurf (.windsurf/rules/) | Graphify (NetworkX DiGraph) | Mem0, Hindsight, LangMem, total-agent-memory, OpenViking, MemMachine |
| **优点** | 语义相似度搜索优秀; HNSW/IVF 原生支持; 高效 k-NN 查询; 可扩展 | 关系建模优秀; 原生多跳遍历; 支持 bi-temporal 边; 表达力强的查询语言 (Cypher/GQL) | 成熟技术; ACID 事务; 灵活查询; 丰富索引; SQL 表达力强; 加 pgvector 后可做向量搜索 | 简单; 人类可读; 版本控制友好; 无需 DB 部署 | 快速遍历; 灵活; 易于原型开发 | 取各存储之长; 不同查询类型优化; 灵活性高 |
| **缺点** | 查询表达力有限; 复杂关系查询差; 频繁更新效率低; 大规模时内存密集 | 运维复杂度高; 学习曲线陡; 扩展有挑战; 调优复杂 | Schema 刚性; 原生无语义搜索需扩展; 向量支持需 pgvector 等扩展 | 无原生索引; 扩展性差; 查询表达力有限; 无并发访问 | 无持久化; 规模受限; 单节点 | 运维复杂; 数据一致性挑战; 多系统维护 |
| **最佳场景** | 语义搜索、RAG、记忆检索、embedding 相似度 | 知识图谱、关系密集型数据、时间推理、实体链接 | 结构化数据、事务型记忆、profile/preference 存储 | Agent 提示词/偏好、简单记忆、静态配置 | 原型开发、小数据集、临时图 | 复杂记忆需求、多样化检索模式 |
| **最差场景** | 复杂 JOIN、事务更新、层次查询、时间推理 | 简单键值访问、高吞吐写入、扁平文档存储 | 非结构化文本、纯语义搜索（无扩展时） | 高容量数据、并发访问、复杂查询 | 生产环境、大规模、持久记忆 | 简单场景、资源有限 |
| **时序推理** | 有限——通常只有时间戳过滤 | **优秀**——bi-temporal 边 (Graphiti 的 valid_at/invalid_at) | 好——timestamp 列、temporal tables | 仅文件时间戳 | 有限 | 因组件而异 |
| **多跳推理** | 差——需要多次向量查询拼接 | **优秀**——原生图遍历 | 中等——SQL JOIN 可用但繁琐 | 无 | 优秀 | 因组件而异 |
| **查询延迟** | ~1-50ms (HNSW，取决于规模) | ~5-100ms（取决于遍历深度） | ~1-20ms（索引查询） | N/A（文件 I/O） | 亚毫秒（内存中） | 通常 10-100ms 聚合 |
| **运维复杂度** | 中等（有托管云选项） | 高（需要图数据库专业知识） | 低-中等 | 非常低 | 低 | 高 |

## 3. 重点框架源码级存储分析

### 3.1 Cortex — PostgreSQL + pgvector 深度分析

Cortex 是所有框架中存储设计最精密的。它的 memories 表有 35+ 个字段：

```sql
CREATE TABLE memories (
    id              SERIAL PRIMARY KEY,
    content         TEXT NOT NULL,
    embedding       vector(384),           -- pgvector
    content_tsv     tsvector GENERATED ALWAYS AS (to_tsvector('english', content)) STORED,
    tags            JSONB DEFAULT '[]'::jsonb,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    heat_base       REAL NOT NULL DEFAULT 1.0,
    heat_base_set_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    surprise_score  REAL DEFAULT 0.0,
    importance      REAL DEFAULT 0.5,
    emotional_valence REAL DEFAULT 0.0,
    confidence      REAL DEFAULT 1.0,
    store_type      TEXT DEFAULT 'episodic',
    consolidation_stage TEXT DEFAULT 'labile',
    plasticity      REAL DEFAULT 1.0,
    stability       REAL DEFAULT 0.0,
    is_protected    BOOLEAN DEFAULT FALSE,
    hippocampal_dependency REAL DEFAULT 1.0
    -- ... 更多字段
);
```

**索引配置**:

```sql
CREATE INDEX idx_embedding ON memories USING hnsw (embedding vector_cosine_ops) WITH (m=16, ef_construction=64);
CREATE INDEX idx_content_tsv ON memories USING gin (content_tsv);
CREATE INDEX idx_content_trgm ON memories USING gin (content gin_trgm_ops);
CREATE INDEX idx_heat ON memories (heat_base);
```

**5 路信号融合** (server-side PL/pgSQL):

1. 向量余弦相似度 (pgvector)
2. 全文搜索 (tsvector/tsquery)
3. 三元组模糊匹配 (pg_trgm)
4. 热值衰减 (thermodynamic heat)
5. 时序衰减 (exponential recency decay)

**记忆生命周期**: labile → early_LTP → late_LTP → consolidated → reconsolidating

还有 Wiki schema (claim_events, concepts, pages) 用于知识提炼。

### 3.2 Engram — FAISS + SQLite

```sql
CREATE TABLE documents (
    id INTEGER PRIMARY KEY,
    text TEXT NOT NULL,
    metadata JSON DEFAULT '{}',
    session_id TEXT,
    role TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

FAISS 索引: IndexFlatIP (L2归一化后的内积=余弦相似度)
Embedding: bge-large-en-v1.5, 1024维

### 3.3 agentmemory — SQLite + 纯 Python HNSW

```sql
CREATE TABLE memories (
    id INTEGER PRIMARY KEY,
    content TEXT NOT NULL,
    kind TEXT DEFAULT 'episodic',
    importance REAL DEFAULT 0.5,
    confidence REAL DEFAULT 1.0,
    created_at TIMESTAMP,
    event_time TIMESTAMP,
    access_count INTEGER DEFAULT 0,
    accessed_at TIMESTAMP,
    embedding BLOB  -- float32 packed
);
```

HNSW: M=16, ef_construction=100, ef_search=50, seed=42 (确定性)
6 信号加权: semantic(0.30) + BM25(0.12) + activation(0.18) + graph(0.18) + importance(0.10) + temporal(0.12)

### 3.4 total-agent-memory — SQLite + FTS5 + 时序知识图谱

```sql
-- 核心图节点
CREATE TABLE graph_nodes (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,       -- 22种类型
    name TEXT,
    content TEXT,
    properties JSON DEFAULT '{}',
    importance REAL DEFAULT 0.0,
    first_seen_at TIMESTAMP,
    last_seen_at TIMESTAMP,
    mention_count INTEGER DEFAULT 0
);

-- 时序事实断言
CREATE TABLE fact_assertions (
    id TEXT PRIMARY KEY,
    subject_id TEXT REFERENCES graph_nodes(id),
    predicate TEXT NOT NULL,
    object_id TEXT REFERENCES graph_nodes(id),
    valid_from TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    valid_to TIMESTAMP,
    superseded_by TEXT REFERENCES fact_assertions(id)
);
```

**kg_at()**: 查询"在任何时间点什么为真"——这是除了 Zep 之外唯一在 SQLite 中实现时序知识图谱的框架。

## 4. 存储选型决策矩阵

| 需求 | 推荐存储 | 可选替代 | 不推荐 |
|------|---------|---------|--------|
| 语义搜索 | 向量DB (Qdrant/FAISS) | PG+pgvector | 文件系统 |
| 时间推理 | 图DB (Neo4j) | SQLite+fact_assertions | 向量DB |
| 多跳推理 | 图DB (Neo4j) | 树结构 (OpenHuman) | 文件系统 |
| 事实更新 | 图DB (bi-temporal) | PG (temporal tables) | 向量DB |
| 企业合规 | PG (ACID) | Neo4j | 文件系统 |
| 低延迟检索 | 向量DB (HNSW) | PG+pgvector | 图DB (深度遍历) |
| 简单部署 | SQLite | 文件系统 | 图DB + 向量DB |
| 成本敏感 | SQLite+FTS5 | Engram (零LLM查询) | 混合存储 |

## 5. AgentCenter 存储方向建议

基于以上分析，AgentCenter 记忆存储的演进路径如下：

### 5.1 短期（MVP）

**SQLite + FTS5**

- 和现有 AgentCenter Bridge 技术栈一致，降低迁移成本
- 参考 total-agent-memory 的 schema 设计（graph_nodes + fact_assertions）
- 可在 SQLite 层面实现基础的时序知识图谱（kg_at 查询）
- FTS5 提供全文搜索能力，弥补纯向量检索的表达力不足

### 5.2 中期

**PostgreSQL + pgvector**

- 引入 Cortex 的 5 信号融合思路（向量 + 全文 + 模糊 + 热值 + 时序）
- 利用 pgvector 的 HNSW 实现低延迟语义检索
- 通过 pg_trgm 实现拼写纠错和模糊匹配
- 设计记忆生命周期（labile → LTP → consolidated）

### 5.3 长期

**PostgreSQL + pgvector + 图扩展**

- 如果时间推理需求强烈，考虑 Apache AGE（PostgreSQL 图扩展）或独立 Neo4j 实例
- Neo4j 的 bi-temporal 模型是唯一精确回答"规则什么时候变"的方案
- 也可以保持纯 PG 路线，通过 temporal tables + fact_assertions 实现大多数时序需求

## 6. 关键发现

### 发现一：PostgreSQL + pgvector 是性价比最高的单一存储

Cortex 证明了一个 PostgreSQL 实例就能同时完成：

- 向量搜索（pgvector HNSW）
- 全文搜索（tsvector/tsquery）
- 模糊匹配（pg_trgm）
- 时序衰减（热力学模型）
- Cross-encoder 重排

这比维护独立的向量数据库 + 图数据库 + 搜索引擎要经济得多。

### 发现二：SQLite 也能做很多

total-agent-memory 在 SQLite 上实现了：

- 时序知识图谱（fact_assertions 表）
- 多种检索信号融合（BM25 + Dense + Fuzzy + Graph + CrossEncoder）
- RRF (Reciprocal Rank Fusion) 融合排序

这说明存储复杂度不一定是功能完整度的前提。

### 发现三：图数据库的时序推理不可替代

Zep 的 bi-temporal 模型是唯一能精确回答以下问题的方案：

- "这条规则在哪个时间段内有效？"
- "什么时候这条规则被新规则取代了？"
- "在特定历史时间点，哪些事实为真？"

如果 AgentCenter 需要强时序推理能力（如合规审计、规则变更追踪），Neo4j 或其等价方案是必要的。

### 发现四：文件系统是记忆的基线不是解决方案

所有主流编码工具（OpenCode、Claude Code、Codex、Windsurf）都停在文件系统层面：

- 人类可读、可版本控制
- 但无原生索引、无并发访问、无查询表达能力

这是 AgentCenter 需要超越的基线。文件系统可以作为偏好和提示词的补充存储，但不能作为主记忆系统。
