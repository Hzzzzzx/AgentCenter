# Zep/Graphiti 调研报告

> AgentCenter 记忆系统调研 · 开源项目 #3
> 调研日期：2026-05-23

## 项目概览

Zep/Graphiti 是为企业级 AI 应用设计的记忆框架，其核心定位是「时序知识图谱引擎」。与 Mem0 的向量优先策略不同，Graphiti 将时间维度作为一等公民，将每个事实建模为图谱中有生命周期约束的节点和边。这使得它在需要「回忆某个时间点之前发生了什么」的场景中具有独特优势。

Zep 整体是一个平台产品，包含持久化存储、API 服务和前端管理层；Graphiti 是其底层知识图谱引擎，负责从对话中提取事实、构建图谱并执行时序推理。两者关系可以类比为 Postgres 与 SQLAlchemy ORM：Graphiti 是底层引擎，Zep 是包装层。

从开源角度看，Graphiti 仓库约 24K GitHub stars，采用 Apache 2.0 许可证（开源版本）+ 企业云商业模式。技术栈以 Python 为主，支持 Neo4j 和 FalkorDB 作为图数据库后端。在合规性方面，Zep 已通过 SOC 2 Type II 和 HIPAA BAA 认证，这对医疗、金融等强监管行业是重要加分项。

性能数据方面，Graphiti 在 LongMemEval 基准上取得 63.8% 的得分，领先 Mem0 约 15 个百分点，但低于 Hindsight。其最强能力集中在时序推理类任务——这正是它与其他框架形成差异化的战场。

### 时序知识图谱核心模型

Graphiti 的数据模型是其最核心的设计决策。每个事实（fact）不再是一条扁平的向量相似度记录，而是图谱中的一个节点或边，拥有明确的有效时间窗口。

以「张明是项目负责人」这条事实为例，它在图谱中表现为：

```
Node: 张明 (entity_id: entity_123)
Node: 项目负责人 (entity_id: entity_456)
Edge: 张明 ──[角色: 项目负责人]──▶ 项目
     valid_at: 2024-01-01
     invalid_at: null  ← 表示当前有效
     expired_at: null
     reference_time: 2024-03-15  ← 系统何时学到这个事实
```

关键字段的含义：

- `valid_at`：事实开始生效的时间点
- `invalid_at`：事实失效的时间点（用于处理「换岗」「离职」等场景）
- `expired_at`：知识本身的保真期限（用于处理「密码已更换」等必然过期信息）
- `reference_time`：系统何时学习到这个事实（bi-temporal 模型的核心）

bi-temporal 模型是 Graphiti 区别于其他框架的关键。它同时追踪两条时间线：一是「事件发生的时间」（valid_at/invalid_at），二是「系统得知事件的时间」（reference_time）。这意味着即使 AI 今天才知道「张明2023年曾是项目负责人」，这条记录也会被正确建模，而不会与「张明2024年已离职」这条新事实产生混淆。

### Episode处理管线

Graphiti 的输入处理管线遵循「Episode → Nodes → Entities → Edges → Contradictions」的顺序：

**Step 1: Episode 接收**
原始对话输入被打包为一个 Episode，包含原始文本、时间戳和对话上下文。

**Step 2: Node 提取**
LLM 从 Episode 中抽取实体节点。这一步调用 LLM，但只用于离线知识抽取，不在查询路径上。

**Step 3: Entity 解析**
将抽取的实体名称映射到图谱中已存在的实体。这一步采用三层策略（详见下一节）。

**Step 4: Edge 提取**
基于实体间关系抽取边，并附带 `valid_at` 时间戳。

**Step 5: 矛盾检测**
当新边与已有边产生矛盾时，触发矛盾解析。最典型的处理方式是前向失效（forward invalidation）：旧边的 `invalid_at` 被设置为新边的 `valid_at`，而非物理删除。这确保历史记录完整可查。

### 实体消歧三层策略

Graphiti 的实体解析采用成本递增的三层策略，这是在精度和性能之间取得的务实平衡：

**第一层：精确匹配**
基于字符串比对或已有 ID 进行匹配。成本最低，适用于结构化数据源或已有实体 ID 的场景。

**第二层：模糊匹配**
使用 MinHash + LSH（Locality Sensitive Hashing）算法进行近似匹配。MinHash 是一种快速估计集合 Jaccard 相似度的技术，LSH 则将其转化为近邻搜索能力。这一层能处理拼写错误、轻微变体等场景。

**第三层：LLM 回退**
当模糊匹配也无法确定时，调用 LLM 进行语义理解。这一层成本最高，因此被设计为最后手段。

这套分层策略的哲学是：用最小计算成本解决问题的阶段绝不浪费更昂贵的手段。

### 检索架构

Graphiti 的检索是纯离线计算、在线快速查询的典型设计。

**Query 时（Online）**：
1. BM25 关键词搜索：捕捉精确词项匹配
2. 向量语义搜索：捕捉意图相似性
3. 时间窗口过滤：根据 `SearchFilters` 中的 `valid_at`/`invalid_at` 约束筛选
4. 图谱遍历：从种子节点出发沿边扩展，聚合子图信息

**特点**：
- 全程无 LLM 调用，延迟可控
- P95 延迟在云版本中小于 300ms
- 支持按时间范围检索：「2024年3月的项目负责人是谁？」
- 支持事实生命周期查询：「这个决策是什么时候做的？现在还有效吗？」

## 核心设计思想

### 为什么用时序图谱？

传统 RAG 和简单向量记忆的本质缺陷在于：它们将知识建模为「快照」，而非「历史」。当用户问「这个项目之前是谁负责的」，向量系统只能找到「最相似」的片段，无法区分「现在是张明负责」和「之前是李四负责」这两个互相矛盾但都正确的陈述。

时序图谱解决了这个问题。每个事实带有时间边界，系统可以精确回答「在某个时间窗口内，哪些事实成立」。这对企业场景尤为重要：合规审计需要知道「当时是基于什么信息做的决策」，项目管理需要知道「需求是什么时候变更的、谁批准的」。

Graphiti 选择图谱而非简单的时间戳标量，原因是图谱能表达实体间的复杂关系网络，且遍历效率高于向量邻域搜索。

### invalid_at 的哲学含义

`invalid_at` 字段的设计体现了一种「不删除」哲学。在传统数据库中，当事实发生变化时，我们通常用 UPDATE 覆盖旧值。这丢失了历史轨迹。

Graphiti 选择 UPDATE → 前向失效：当新事实到来时，将旧事实的 `invalid_at` 设置为新事实的 `valid_at`，然后插入新事实节点。这样图谱中同时存在「旧事实（已失效）」和「新事实（当前有效）」两条记录。

这种设计有几个好处：
- 历史可追溯：任何时间点的事实状态都可重建
- 矛盾可审计：可以追溯矛盾的演变过程
- 并发友好：新事实无需等待旧事实的所有引用方确认

代价是图谱会不断膨胀。Graphiti 通过 `expired_at` 提供主动清理机制，对必然过期的信息设置 TTL。

### 为什么不在查询时用 LLM？

这是 Graphiti 与 Mem0 另一个重要分歧点。Mem0 在查询时使用 LLM 进行重排序和意图推理；Graphiti 坚持在查询路径上不调用 LLM。

原因有几个层面：

**延迟**：LLM 推理在 500ms-2s 之间，向量+图谱遍历在 50-300ms。在高频对话场景中，这个差距会被放大。

**成本**：每次查询都过 LLM 的成本在大规模应用中不可忽视。

**可靠性**：LLM 输出有不确定性，在需要精确答案（如「某人的任期截止到哪天」）的场景中，概率性输出是风险。

Graphiti 的方案是：将 LLM 能力前移到离线抽取阶段，在知识入库时充分使用 LLM 的理解能力；查询时则用确定性的图谱算法保真度。

## 优势分析

**时序推理能力独一无二**：这是 Graphiti 最强的差异化能力。没有其他开源记忆框架能原生支持 bi-temporal 模型和时序范围查询。对于需要「回忆历史上某刻状态」的场景，Graphiti 是唯一选择。

**知识图谱结构优于平面向量**：图谱能表达实体间多跳关系，可以回答「张明的同事的项目的客户是谁」这类多跳问题。向量系统处理这类问题需要暴力枚举，效果差。

**查询效率高**：无 LLM 查询路径确保了低延迟，这对实时对话场景至关重要。

**企业合规性完整**：SOC 2 Type II + HIPAA BAA 认证在同类开源框架中罕见。对医疗、金融等行业的 AI 应用，这是硬性门槛。

**开源可审计**：Apache 2.0 许可证允许企业内部分叉和定制，不存在供应商锁定风险。

## 局限与不足

**运维复杂度高**：需要维护 Neo4j 或 FalkorDB 图数据库，相比 SQLite/Postgres 的 Mem0，部署和运维成本显著更高。对于中小型团队，这可能是不小的负担。

**实体消歧成本**：三层策略中最贵的 LLM 回退在实体名称歧义度高时会频繁触发。需要提前评估目标领域的实体密度和命名规范性。

**非工程信号感知**：Graphiti 依然以对话为核心设计场景，不感知代码变更、PR 合并、CI 失败等工程信号。与 AgentCenter 的 Runtime 事件体系缺乏原生集成点。

**社区规模**：24K stars 相比 Mem0 的 26K 略少，文档和示例的丰富程度还有提升空间。遇到问题时社区支持力度可能不足。

**学习曲线**：时序图谱的概念和查询语法比简单向量存储更复杂，团队需要投入时间学习。

## 与 AgentCenter 的契合度

AgentCenter 的核心场景是企业工具流程编排，记忆系统需要支持工作流状态跟踪、多轮会话上下文、确认项历史等复杂需求。Graphiti 的时序推理能力在这些场景中有直接价值：

- **工作流演进追踪**：工作流的节点状态随时间变化，Graphiti 可以精确重建「某个时间点工作流处于什么状态」
- **确认项历史**：ConfirmationRequest 的审批历史天然带有时间维度
- **多跳推理**：「谁批了这个需求」→「需求对应哪个工作流」→「工作流当前状态」这类问题需要图谱能力

但 AgentCenter 还需要感知 Runtime 信号（代码变更、构建结果、测试状态），Graphiti 对这类非结构化工程数据的处理不如 Mem0 灵活。

**结论**：Graphiti 适合作为 AgentCenter 记忆系统的时间维度组件，与现有向量记忆形成互补。纯时序查询场景用 Graphiti，工程信号积累场景用 Mem0 或其他方案。两者组合使用可能是最优解，而非单一框架替代。
