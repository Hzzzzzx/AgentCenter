# LangMem 调研报告

> AgentCenter 记忆系统调研 · 开源项目 #7
> 调研日期：2026-05-23

## 项目概览

LangMem 是 LangChain/LangGraph 生态下的一个年轻记忆框架，GitHub 仓库为 langchain-ai/langmem，当前版本 0.0.30，获得 1,464 颗星（截至 2026 年 5 月）。项目采用 MIT 许可证，以 Python 为主要实现语言，其核心设计理念从人类记忆系统中汲取灵感，将 agent 的记忆划分为语义记忆（Semantic Memory）、情景记忆（Episodic Memory）和程序记忆（Procedural Memory）三种类型。

从项目定位来看，LangMem 处于 Memory Framework for LLMs 的交叉地带：它既不像 LangChain 内置的简单 memory 那样轻量，也不像 LongMemEval 等学术基准所衡量的超长上下文方案那样厚重。LangMem 的差异化在于将记忆管理拆解为一套结构化的提取管线，并额外提供了 Prompt 优化和短时摘要能力。这两件事在同类开源方案中并不常见——大多数框架关注的是"记住什么"，而 LangMem 同时关注"如何让 agent 高效地记住"和"如何让 prompt 本身学会记住"。

### 三类记忆模型（语义/情景/程序）

LangMem 采用了认知科学中经典的三分法来组织 agent 记忆，这一设计并非营销概念，而是有其工程层面的对应实现。

**语义记忆（Semantic Memory）** 对应的是事实和知识层面的信息，例如"Python 是一种编程语言"或"用户的项目使用 Vue 3 + Pinia"。这类记忆以结构化的 Profile 或 Collection 形式存储，在检索时不需要回忆具体事件，只需要知道某个事实是否存在。在 AgentCenter 场景下，语义记忆最适合存储用户偏好、系统配置和企业流程规范等相对稳定的信息。

**情景记忆（Episodic Memory）** 存储的是过去经历的时间线片段，例如"agent 在 2026-05-20 帮助用户修复了一个认证 bug"。情景记忆的核心价值在于支持 agent 回顾类似场景的历史决策，从而在面对相似任务时能够借鉴过往经验。存储形式为 Collection，每条记录通常包含时间戳、情境描述和结果。

**程序记忆（Procedural Memory）** 则对应系统行为模式，即知道"如何做"而非知道"是什么"。在 LangMem 中，程序记忆以 Prompt rules 的形式编码，直接影响 agent 的行为策略。例如一条程序记忆可以是"当用户提到性能问题时，先检查数据库查询日志而非直接修改代码"。这种记忆类型的提取和更新最为困难，因为它涉及行为模式的归纳而非事实的陈述。

### 多步提取管线（MemoryManager）

LangMem 的核心提取逻辑封装在 MemoryManager 类中，它采用多步循环的方式从对话上下文里提取和更新记忆。这一设计值得深入分析，因为它直接影响记忆的质量和提取的可靠性。

MemoryManager 的提取流程分为以下步骤：

```
1. Retrieve
   └─ 从 BaseStore 拉取已有记忆（用于比较和去重）

2. Prepare
   └─ 将已有记忆 + 原始 messages 组装为带上下文的提示词

3. Multi-Step Extraction (trustcall)
   ├─ Insert:  添加新记忆
   ├─ Update:  修改已有记忆
   └─ Delete:  删除过时记忆

4. Filter
   └─ 区分 internal memory ID 和 external memory ID，过滤响应
```

步骤 3 中的 Insert/Update/Delete 三元操作由 `trustcall` 库驱动，该库基于 Pydantic 模型做结构化提取，确保提取出的记忆条目具有类型安全和schema约束。这种设计的好处在于：每次提取的结果可以直接落盘，不需要二次清洗。相比之下，纯提示词工程驱动的记忆提取方案往往面临"提取格式不稳定"的问题，需要额外的后处理。

值得注意的是，多步提取的迭代次数由 `max_steps` 参数控制。对于简单对话，一到两步足以完成记忆更新；对于复杂的讨论场景，可能需要更多步骤来做多轮比较和推理。

### 两种记忆形成策略（热路径 vs 后台）

LangMem 提供了两种截然不同的记忆形成策略，分别对应不同的实时性要求和计算资源分配。

**热路径（Hot Path）** 是一种主动策略，agent 在对话过程中实时调用记忆工具。LangMem 为此提供了 `create_manage_memory_tool()` 和 `create_search_memory_tool()` 两个工厂函数，生成的工具可以直接注入到 agent 的工具集中。热路径的优势在于记忆更新零延迟，agent 在当前对话轮次就能利用最新提取的记忆。然而代价也显而易见：每次记忆操作都会引入额外的 LLM 调用延迟，对于延迟敏感的生产环境这是一个需要权衡的因素。

**后台（Background）** 策略则由 ReflectionExecutor 驱动，它在对话结束后异步处理记忆提取。这种方式的 trade-off 与热路径正好互补：没有延迟影响、可以进行更深度的分析（因为对话已结束，不受时间压力）、但记忆更新是延迟的。ReflectionExecutor 特别适合那些不需要立即响应、但需要系统性整理大量历史交互的场景。

从工程实现角度看，热路径将记忆操作变成了对话流的同步一部分，而后台策略则将记忆管理下沉为独立的后处理管线。两者并非互斥——对于同一个 agent，完全可以同时启用热路径（处理即时需求）和后台策略（处理系统性复盘）。

### 短时记忆摘要

除了三种长期记忆类型，LangMem 还内置了短时记忆摘要能力，专门处理对话窗口内的消息压缩问题。这在实践中非常重要，因为即使有了长期记忆，对话过程中的上下文窗口仍然是有限的。

短时记忆摘要的核心特性包括：

- **Token 感知的压缩**：基于 token 计数而非固定字符数来做截断，压缩比更可控
- **Running summary**：维护一个跨越多轮的累积摘要，而非每次都从零开始
- **系统消息保留**：确保系统提示词等关键上下文不被摘要掉
- **工具调用处理**：对 tool_calls 等结构化内容有专门的保留策略

这套机制对于需要在有限上下文窗口内维持长时间对话的 agent 尤为关键。相比直接将历史消息截断，摘要方案保留了更多语义信息。

## 核心设计思想

### 为什么用人类记忆类比？

LangMem 选择将认知科学中的人类记忆三分法（语义、情景、程序）作为核心抽象，并非单纯的概念包装。这三种记忆类型在工程层面有明确的对应关系：

语义记忆对应的事实型存储，最适合用键值或文档数据库实现，检索时强调匹配而非推理。情景记忆对应的事件型存储，需要时间索引和上下文关联能力。程序记忆则必须编码为可直接执行的规则或提示词片段，因为它的消费者是推理引擎而非人类。

这套类比的价值在于为开发者提供了直觉上的锚点：当需要决定"某条信息应该存在哪里"时，可以借助人类记忆的直观分类来思考。同时，三种记忆类型的更新策略差异，也为工程实现提供了明确的分野——不同类型的记忆，其提取频率、更新逻辑和检索模式本就不同。

### trustcall 结构化提取的意义

trustcall 是 LangMem 记忆提取管线的底层驱动库，它基于 Pydantic 模型做结构化输出。在传统的记忆提取方案中，LLM 返回的是自然语言文本，需要后处理解析才能结构化。trustcall 的核心创新在于将 Pydantic schema 作为约束条件，直接引导 LLM 输出符合 schema 的 JSON 对象。

这一设计解决了几个实际问题：

**格式稳定性**：基于 schema 的约束大幅减少了提取结果的结构散射，使得记忆存储的 schema 演进更容易管理。

**类型安全**：提取结果可以直接反序列化为 Python 对象，不需要手动解析。

**可测试性**：Pydantic 模型本身可以附带验证逻辑，提取结果天然经过验证。

对于 AgentCenter 这类企业级平台而言，trustcall 的结构化提取意味着记忆系统产出的数据可以被下游系统更可靠地消费，不需要额外的适配层。

### Prompt 优化为什么重要？

Prompt optimization 是 LangMem 最独特的功能点之一，它解决了一个长期被忽视的问题：大多数 agent 的 system prompt 是静态的，而记忆系统的效能很大程度上取决于 prompt 能否有效地驱动记忆提取和检索。

LangMem 提供了三种优化策略，复杂度递增：

| 策略 | LLM 调用次数 | 适用场景 |
|------|-------------|---------|
| prompt_memory | 1 次 | 快速迭代，对质量要求不极端 |
| metaprompt | 2-5 次 | 平衡场景，需要比 prompt_memory 更系统性的优化 |
| gradient | 4-10 次 | 深度优化，分离关注点最彻底 |

prompt_memory 策略只做一次 LLM 调用来生成优化后的 prompt，适用于开发阶段快速试错。metaprompt 策略会多轮迭代，逐步精化 prompt 的结构和指令。gradient 策略则将 prompt 的不同方面（格式指令、上下文策略、检索触发条件等）分离处理，每方面单独优化后组合。

这套梯度式设计的工程意义在于：团队可以根据实际资源预算和优化需求选择合适的策略层级，而不是要么不用、要么全套上。

### 存储无关设计的好处

LangMem 的核心记忆功能完全基于 LangGraph 的 BaseStore 接口，这是一个存储无关（storage-agnostic）的抽象层。BaseStore 定义了一套简洁的键值操作 API，底层可以对接 InMemory（进程内缓存）、PostgreSQL（持久化存储）或其他支持键值语义的存储引擎。

存储无关设计的直接好处是避免了供应商锁定。在企业环境中，数据的持久化需求差异巨大：有些场景适合用轻量的内存缓存，有些场景则必须落库并支持 SQL 查询。LangMem 的 BaseStore 抽象让同一个记忆系统可以灵活适配不同的存储后端，而不需要修改业务逻辑层代码。

另一个附带好处是测试便利性：使用 InMemory 后端可以零依赖地跑通所有记忆逻辑，CI 流水线中不需要额外部署数据库服务。

## 优势分析

**Prompt 优化内置**：在同类开源框架中，LangMem 是少数将 prompt 优化作为一等公民功能的方案。这对于需要持续优化 agent 表现的团队而言是实质性的价值。

**存储无关核心**：BaseStore 的抽象让 LangMem 可以在不修改业务逻辑的情况下切换存储后端，这对于需要兼顾开发和生产环境的团队非常重要。

**结构化提取管线**：trustcall 驱动的多步提取相比纯提示词工程更加可靠，产出的记忆数据可直接消费。

**LLM 驱动的记忆整合**：不同于简单的向量相似度检索，LangMem 的记忆整合由 LLM 驱动，可以做语义层面的比较和推理，而非仅做关键词匹配。

**短时摘要内置**：对话窗口内的消息压缩方案已经包含在框架内，不需要额外集成。

**LangChain 生态整合**：作为 LangChain 生态的一部分，LangMem 可以与 LangGraph 的 agent 编排能力无缝衔接，降低了 LangChain 用户的接入成本。

## 局限与不足

**项目成熟度偏低**：当前版本 0.0.30 是一个相当早期的版本号，意味着 API 尚未稳定，breaking changes 的风险较高。对于需要生产级稳定性的企业项目，这是一个需要认真评估的因素。

**LangGraph 依赖**：虽然核心功能可以脱离 LangGraph 运行，但高级特性（如 ReflectionExecutor）的集成度更高。已深度使拥其他 agent 框架（如 CrewAI、AutoGen）的团队，迁移成本不可忽视。

**缺乏基准测试数据**：目前没有公开的 LongMemEval 等标准 benchmarks 数据来衡量 LangMem 在长程记忆任务上的表现，难以与其他方案做客观对比。

**社区规模有限**：1,464 颗星的项目在 GitHub 生态中属于小众项目，第三方插件、工具和社区支持都相对匮乏。

**无知识图谱或时间推理**：与一些更前沿的记忆方案相比，LangMem 不支持知识图谱构建或时间线推理等高级能力，其记忆模型相对扁平。

**无托管服务**：作为一个开源库而非 SaaS，LangMem 不提供开箱即用的托管服务，企业需要自行运维。

## 与 AgentCenter 的契合度

从架构层面看，LangMem 与 AgentCenter 的契合度中等偏高，主要加分项在于其存储无关设计和 prompt 优化能力。AgentCenter 作为企业级 agent 编排平台，记忆系统的存储灵活性是刚需——不同企业可能已有不同的存储基础设施，BaseStore 抽象可以有效适配这一需求。

Prompt 优化能力则是一个差异化亮点。AgentCenter 的目标客群中包含大量需要持续优化 agent prompt 的企业团队，内置的梯度式 prompt 优化可以降低这部分工作的复杂度。

然而需要正视的挑战包括：LangMem 的年轻程度意味着与 AgentCenter 集成时需要承担较高的依赖风险。此外，LangMem 目前缺少对多 agent 共享记忆场景的明确支持——在 AgentCenter 的编排模型中，多个 agent 可能需要共享同一份记忆库，这方面的设计文档尚不清晰。

综合评估，LangMem 适合作为 AgentCenter 记忆模块的技术选型候选之一，但建议以实验性引入为主，在受控环境中验证其稳定性和性能表现后再决定是否投入生产。建议重点考察的方向包括：BaseStore 与 AgentCenter 现有存储层的适配成本、ReflectionExecutor 在大规模历史数据上的性能表现，以及多 agent 共享记忆场景下的并发安全性。
