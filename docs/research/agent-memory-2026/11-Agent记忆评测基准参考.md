# Agent 记忆评测基准参考

> AgentCenter 记忆系统调研 · 评测基准篇
> 调研日期：2026-05-23

## 1. 为什么需要评测基准

在构建企业级 Agent 记忆系统的过程中，我们不可避免地会遇到一个根本性问题：如何判断我们的记忆系统到底"好不好"？这个问题如果不回答清楚，后续所有的技术选型、架构设计和产品迭代都会失去锚点。

没有评测基准就没有量化依据。在没有基准的情况下，团队对" Mem0 比 Zep 好"或者" Synthius-Mem 达到了 94%"这样的判断就只能凭感觉或者引用第三方结论。评测基准把"好"和"不好"变成了可测量、可对比、可复现的数字。

我们的调研报告需要可信的数据支撑。AgentCenter 的定位是企业智能编排平台，记忆系统是这个平台的核心能力之一。当我们需要向客户、技术团队或者管理层解释"为什么选择当前的记忆架构"时，最有力的证据就是：我们在这套 benchmark 上验证过，量化指标优于备选方案。

评测基准是技术决策的证据基础。从更宏观的视角看，记忆系统的评测基准还决定了我们如何定义"记忆"本身——评测框架会显式化我们对 Agent 记忆能力的期待：是只考查阅历史信息，还是要能处理事实更新？是要支持多跳推理，还是要能识别并解决矛盾记忆？不同的 benchmark 反映了不同的技术假设，而选择哪些 benchmark 就等于选择了我们认同哪些技术假设。

## 2. 评测基准全景图

根据调研，我们将目前主流的 Agent 记忆评测基准分为四个层级。

### Tier 1：核心记忆基准

**LongMemEval**（ICLR 2025）是我们在调研中引用最多的基准之一。全称 Long-Term Memory Evaluation for LLM Agents，核心测试场景包括跨会话记忆、知识更新、多跳推理、时序推理和干扰抵抗。数据集规模最大到 1M token 上下文。在 LongMemEval-S（500 题）上，Hindsight 得分 91.4%，Zep 63.8%，Mem0 49.0%。LongMemEval V2（arXiv:2605.12493）扩展到 451 题，采用"有经验同事"的评测场景，最高得分是 AgentRunbook-C 的 72.5%。

**LoCoMo**（ACL 2024）是目前业界采用最广泛的全称 Long Context Memory benchmark。测试能力涵盖信息提取、多跳推理、偏好遵循、时序推理和摘要生成。标准数据集 LoCoMo-10 包含 10 个多会话对话和 1,813 道题目，LoCoMo-10K 提供了更大规模的测试集。在 ACL 2026 上，Synthius-Mem 刷出了 94.37% 的 SOTA 分数，True Memory Pro 93.0%，MemMachine 91.69%。Causal-LoCoMo 扩展（arXiv:2605.17641）在原版基础上增加了因果干预测试能力。

**BEAM**（ICLR 2026，全称 Beyond a Million Tokens: Benchmarking and Enhancing Long-Term Memory in LLMs）是第一个将测试规模推到 10M token 的基准。包含 100 个对话、2000+ 题目，覆盖 128K/500K/1M/10M 四个规模档位。BEAM 的最大贡献是完整覆盖了全部 10 种记忆能力，并且在 HuggingFace（数据集 Mohammadta/BEAM）和 GitHub（mohammadtavakoli78/BEAM）上都已开源。

### Tier 2：专项记忆基准

**MINTEval**（arXiv:2605.18565）专注于多目标干扰下的记忆性能测试，即当多个干扰目标同时存在时，系统能否准确提取和保持目标记忆。数据集包含 15.6k Q&A 对，平均上下文 138.8k token，最长达 1.8M token。涵盖状态追踪、多轮对话、维基百科修订和 GitHub 提交四个领域。关键发现：所有系统在该基准上的平均准确率仅有 27.9%，说明这个方向极具挑战性。项目 GitHub 为 amynji/MINTEval。

**MemConflict**（arXiv:2605.20926）研究记忆系统在面对矛盾记忆时的表现。定义了三种冲突类型：动态冲突（事实随时间变化）、静态冲突（矛盾事实在同一时间并存）和条件冲突（事实在特定条件下才为真）。核心发现是答案正确性和记忆检索与排序的一致性之间存在明显偏差。

**MemGym**（arXiv:2605.20833）提出了一个长时域记忆环境，测试记忆在真实 Agent 任务中的表现，包括工具使用、深度研究、编码和计算机会话。其创新点在于提出了 Memory-isolated scores，即把记忆性能从推理、检索和工具使用能力中解耦出来单独评分，从而更精确地定位记忆模块的短板。MemGym 还包含一个 MemRM reward model，基于 Qwen3-1.7B 微调得到。

**EvoMemBench**（arXiv:2605.18421）从一个自我演化的视角来评测 Agent 记忆，对比了 15 种记忆方法与强力的长上下文基线。关键结论：没有一种单一的记忆形式在所有场景下都持续有效。GitHub 为 DSAIL-Memory/EvoMemBench。

### Tier 3：RAG 评测基准

这一层虽然不是专门为 Agent 记忆设计，但与记忆的检索能力直接相关。

**RAGAS**（Retrieval Augmented Generation Assessment）是目前最广泛使用的 RAG 评测框架，测试响应相关性、事实正确性和流畅度。**CRUD-RAG** 专注于 RAG 系统的增删改查操作评测。**ARES**（Automated RAG Evaluation System）提供了自动化的 RAG 评测系统。

### Tier 4：行业基准

**Memstate Benchmark**（GitHub: memstate-ai/memstate-benchmark）支持跨 Memstate、Mem0 和基于 MCP 的系统进行对比评测，并支持任意 OpenRouter 模型。**ATANT**（GitHub: Kenotic-Labs/ATANT）是一个 AI 连续性评估框架，包含 250 个叙事测试、1835 道验证题和 10 个检查点。其特点是评估流程中不使用 LLM，避免了 LLM 自我评判引入的偏差。**OpenAgentBench** 测试工具选择最优性、状态转换正确性、记忆卫生、权限安全、恢复行为和多 Agent 协调能力。

## 3. 核心基准详解

### 3.1 LongMemEval：调研引用的分数之源

LongMemEval 是我们在技术调研报告中引用最频繁的基准之一，原因在于它最接近企业 Agent 的真实使用场景。LongMemEval-S 包含 500 道题，LongMemEval-KU 包含 78 道 Oracle 题目，专门测试知识更新能力。

LongMemEval 的设计理念是把 Agent 放在一个需要"跨长周期记忆信息"的场景中。典型问题包括：用户在第三会话提到的偏好，在第十会话时还能被正确回忆吗？用户曾经纠正过的错误信息，在新会话中还会被错误地复述吗？这些问题的本质是测试记忆系统在时间跨度上的保持能力和更新能力。

LongMemEval V2 引入了一个非常有价值的评测范式——"有经验同事"场景。在这个场景下，Agent 被视为一个与用户长期共事的同事，需要展现出对用户历史偏好、工作方式和之前讨论过的事项的记忆能力。这个范式更接近企业 Agent 的实际使用场景，也因此得到了更高的行业关注度。V2 最高分 AgentRunbook-C 达到 72.5%，说明当前最优系统在企业场景下仍有明显提升空间。

### 3.2 LoCoMo：业界采用最广的基准

LoCoMo（Long Context Memory benchmark）之所以成为业界引用最多的基准，关键在于它构建了一套系统化的能力分类体系。LoCoMo 将 Agent 记忆能力分解为五大维度：

信息提取（Information Extraction）考察系统能否从长上下文中准确找到并提取特定信息。多跳推理（Multi-hop Reasoning）要求系统综合多个来源的信息才能得出正确答案。偏好遵循（Preference Following）测试系统是否记得并遵循用户在历史对话中表达的偏好。时序推理（Temporal Reasoning）验证系统对事件发生顺序和时间线的理解。摘要生成（Summarization）评估系统压缩长上下文为核心信息的能力。

LoCoMo-10 数据集包含 10 个多会话对话和 1813 道题目，覆盖了日常个人助理场景中最典型的时间跨度。Synthius-Mem 在 ACL 2026 刷出 94.37% 的分数，True Memory Pro 93.0%，说明在特定评测集上头部系统已经接近人类水平。但需要注意，94% 的分数不代表在所有场景下都达到了人类水平——评测集的设计边界决定了分数的解读范围。

### 3.3 BEAM：覆盖最全、规模最大

BEAM 的核心贡献有两个：第一是将评测规模首次推到 10M token 量级，第二是完整覆盖了全部 10 种记忆能力。

BEAM 测试的 10 种能力分别是：Abstention（知道什么时候说"我不知道"）、Contradiction Resolution（矛盾解决）、Event Ordering（事件排序）、Information Extraction（信息提取）、Instruction Following（指令遵循）、Knowledge Update（知识更新）、Multi-Session Reasoning（多会话推理）、Preference Following（偏好遵循）、Summarization（摘要）和 Temporal Reasoning（时序推理）。

这 10 种能力的完整覆盖是 BEAM 区别于其他基准的核心价值。在 BEAM 之前，没有任何单一基准能够评估 Agent 记忆系统的全部能力维度。这意味着如果想全面评估一个记忆系统，在此之前需要跑多个不同的基准，BEAM 大幅降低了评测复杂度。

BEAM 还引入了 LIGHT 框架，这是一个三层记忆架构：Episodic Memory（情景记忆）、Working Memory（工作记忆）和 Scratchpad Memory（草稿板记忆）。这个框架本身就是一个记忆系统的设计参考。

BEAM 的四个规模档位（128K/500K/1M/10M）让我们能够系统性地观察记忆系统在不同上下文长度下的性能衰减曲线。这对 AgentCenter 来说尤为重要，因为我们的企业用户会处理不同复杂度的工作任务，需要的记忆跨度也不同。

## 4. 能力维度对照表

理解 10 种记忆能力维度是正确使用评测基准的前提。下面逐一解释每个维度，并说明 AgentCenter 的 5 层需求如何对应这些能力。

**信息提取（IE）**是最基础的记忆能力：指能否从存储的历史信息中准确定位并提取出需要的内容。这是所有记忆系统必须具备的核心能力，对应 AgentCenter 的工作记忆层和情景记忆层。

**多跳推理（MR）**要求系统综合多个不连续信息源才能得出答案。例如用户问"上次我让你找的那个供应商，价格比三月份谈的还低吗"，系统需要从至少两个时间点的记忆中提取数据进行比较。对应 AgentCenter 的情景记忆层和分析层。

**知识更新（KU）**测试系统能否正确更新已存储的过时信息。用户说"我们的供应商换了，从 A 换成 B"，系统需要将旧信息标记为过时并用新信息替代，而不是两者都保留。这是企业场景中极为关键的能力，对应 AgentCenter 的情景记忆层和知识层。

**时序推理（TR）**要求理解事件的时间顺序。"你记得我们先讨论了 X 然后才决定 Y 吗"这类问题测试的是系统对时间线的维护能力。对应 AgentCenter 的情景记忆层。

** Abstention（ABS）**指系统知道自己不知道什么。当问题超出了记忆范围时，正确拒绝回答比编造一个答案更有价值。这是一个被很多评测忽略但对企业场景极其重要的能力——在客户服务场景中，虚构信息的危害远大于说"我不记得了"。对应 AgentCenter 的知识层。

**矛盾解决（CR）**处理记忆中的矛盾信息。当系统存储了"A 公司是最好的供应商"和"A 公司交付总是延期"这两条矛盾信息时，面对"A 公司到底怎么样"的问题应该给出平衡的答案。这个能力在企业场景中直接影响回答的可信度，对应 AgentCenter 的知识层。

**事件排序（EO）**要求对多个事件按时间或因果顺序排列。"请按顺序列出我们这次项目中的五个关键节点"需要系统维护完整的事件时间线。对应 AgentCenter 的情景记忆层。

**指令遵循（IF）**测试系统是否记得并遵循用户在历史对话中给出的具体指令。"按照上次我们约定的格式来生成报告"需要系统记住并在当前会话中应用历史指令。对应 AgentCenter 的工作记忆层。

**偏好遵循（PF）**与指令遵循类似但更关注隐式偏好。用户说"以后这种东西不要发给我"，系统需要记住并在后续自动应用这个偏好。对应 AgentCenter 的用户画像层。

**摘要（SUM）**评估系统压缩长上下文为简洁信息的能力。当对话历史非常长时，系统能否提炼出关键信息供当前会话使用。对应 AgentCenter 的工作记忆层和分析层。

MINTEval 的发现尤其值得注意：所有系统在该基准上的平均准确率仅 27.9%。这意味着当记忆系统面对"多目标干扰"的场景时，现有任何方案都无法可靠工作。对 AgentCenter 来说，这意味着我们不能假设记忆系统在复杂多任务场景下能正确运作，必须为这类场景设计额外的保护机制。

## 6. AgentCenter 应该怎么用这些基准

### 6.1 建议采用的评测基准组合

基于我们的调研，推荐 AgentCenter 采用以下基准组合来验证记忆系统：

**BEAM**作为主基准。它完整覆盖 10 种能力，四个规模档位足以覆盖从简单查询到超长会话的各种场景。BEAM-1M 应该是我们的核心评测点，因为企业 Agent 的实际上下文长度通常在这个量级。

**LoCoMo**作为辅助基准。尽管 BEAM 已经覆盖了 LoCoMo 测试的所有能力，但 LoCoMo 的题目设计更成熟、行业认可度更高，可以作为交叉验证的基准。

**LongMemEval-S**用来重点测试知识更新能力。这是企业场景中最关键的痛点之一，LongMemEval-S 的知识更新专项测试（LongMemEval-KU）值得单独运行。

**MINTEval**作为"压力测试"。我们需要在 27.9% 的平均分基础上明确自己的位置，并设定提升目标。

### 6.2 如何建立自己的评测集

公开基准的局限在于：无法覆盖 AgentCenter 的特定业务场景。因此我们需要建立自己的评测集，具体方法如下：

从生产日志中提取典型记忆使用场景。分析 AgentCenter 实际运行中的对话日志，识别出最频繁出现的记忆相关问题类型，例如"查找用户上次提到的项目名称"、"确认用户偏好的报告格式"等。基于这些典型场景设计评测题目，每个场景至少 20 道题，确保统计显著性。

标注自己的 ground truth。评测集中的每个问题必须有明确的正确答案，这个答案应该来自实际业务逻辑而不是人工猜测。可以邀请产品团队和客户成功团队共同参与标注，确保答案符合业务预期。

维护评测集版本。随着 AgentCenter 功能迭代和用户场景扩展，评测集也需要持续更新。建议每季度进行一次评测集审查，移除过时的题目并补充新场景。

### 6.3 评测流程设计

评测流程应该分为三个层级：

**离线基准评测**。每当我们考虑升级记忆系统组件（比如从 Mem0 切换到 Zep，或者自研记忆层），先用 BEAM、LoCoMo 和 LongMemEval 进行离线基准评测。记录各能力维度的分项得分，重点关注 KU（知识更新）和 CR（矛盾解决），因为这两个维度在企业场景中最具业务价值。

**在线 A/B 测试**。在离线评测通过后，将新方案与当前方案在真实用户中进行 A/B 测试。评测指标应该包括：用户显式满意度（"这个问题回答得准确吗"）、隐式行为指标（用户是否需要重复提供已经说过的信息）、以及记忆相关工单率（用户投诉"Agent 不记得上次说的"的比例）。

**定期回归评测**。即使没有主动升级记忆系统，也应该每季度运行一次完整基准评测，监控记忆系统在各项能力上的漂移情况。记忆系统不同于其他模块，它的状态会随时间累积而变化，回归评测能帮我们及时发现性能退化。
