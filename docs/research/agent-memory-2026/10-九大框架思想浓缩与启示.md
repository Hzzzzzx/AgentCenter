# 九大记忆框架思想浓缩与启示

> 本文档是「记忆框架速查卡」，供项目负责人在 5 分钟内向团队成员、干系人或技术负责人解释每个框架的核心思想。每张卡片的重点是「这个框架相信什么」而非「它暴露什么 API」。

---

## 1. Mem0

> 记忆只需要两个 API：add() 和 search()。

**关键词**：simplicity wins, self-editing memory, parallel retrieval

### 它的核心主张

Mem0 的哲学是**极简主义**——不要过度设计。记忆本质上就是「从对话中提取事实」然后「之后能找回来」两件事。API 简单，概念简单，上手快。但简单不等于简陋：它的 self-editing 机制说明它在后台做了大量工作，只是没有把复杂性暴露给用户。

### 它怎么做的（最简版）

- add() 管道走 8 个阶段，但对外只暴露一个调用
- 新记忆不是简单追加，而是 self-edit：对比已有记忆，避免重复，保留最准确的版本
- search() 三路并行：语义向量 + BM25 关键词 + 实体增强，同时跑，取最优
- 不维护知识图谱，纯靠向量检索 + 关键词
- 多语言 SDK，部署门槛低

### 精妙之处

**self-editing 而不是 append**。这是整个框架最聪明的设计——聊天机器人式的「不断追加」会导致记忆膨胀和噪声累积。Mem0 选择让新记忆「编辑」旧记忆，类似于人类会不断修正自己的认知而不是无限堆砌信息。对企业场景来说，这个方向是对的。

### 局限性

- 最低的 LongMemEval 得分（49%），说明极简有天花板
- 无时间推理能力，无法回答「今年 1 月的情况是什么」
- 无知识图谱，只能做相似度匹配，不能做关系推理
- 对复杂企业场景（多项目、时变性业务规则）支持弱

### 给 AgentCenter 的启示

**记忆需要自我修剪，而不是无限增长**。这个原则应该贯彻在我们整个记忆系统的设计中。但 49% 的得分提醒我们：简单是起点，不是终点——我们必须在保持简洁的同时加入 Mem0 没有的能力（时态推理、知识图谱）。

## 2. Letta / MemGPT

> 把 LLM 当操作系统，记忆就是它的 RAM、Cache 和磁盘。

**关键词**：three-tier memory, agent-autonomous, context window calculator

### 它的核心主张

Letta 相信** agent 应该自主管理自己的记忆**，就像操作系统管理 RAM 和磁盘一样。把记忆分层（热/温/冷），agent 自己决定什么放核心记忆、什么放召回记忆、什么归档。这个主张的底层信念是：人类是最好的记忆管理者，agent 也应该有能力做到。

### 它怎么做的（最简版）

- 三层记忆架构：Core Memory（始终在上下文，agent 直接读写）→ Recall Memory（对话历史，可搜索）→ Archival Memory（无限外部存储）
- Agent 拥有工具函数：core_memory_append/replace, memory_rethink, archival_memory_insert/search
- 上下文窗口计算器动态追踪 token 使用量，防止溢出
- 有配套的 Web UI 和 API 服务，不只是库

### 精妙之处

**上下文窗口计算器**。这是整个框架最值得「偷」的想法——实时追踪 agent 可用的 token 预算，在接近上限时主动触发记忆操作。对我们构建企业级记忆服务来说，这个能力是不可妥协的：多个 agent 共享上下文窗口，必须有精确的资源调度。

### 局限性

- 完整运行时环境，不是轻量库，部署复杂
- agent 自主管理记忆对企业场景有风险——它可能忘记重要的事情或记住错误的事情
- per-agent 隔离模型不适配共享企业记忆池
- 研究导向，有 peer-reviewed 论文，但工程化程度不如商业产品

### 给 AgentCenter 的启示

**三层抽象（热/温/冷）是正确的分层方向**。但「agent 自主决定」这个模型对企业不适用——我们需要人工介入的机制（human-in-the-loop）。context window calculator 必须有，但调度的控制权要在系统层面，不在 agent 层面。

## 3. Zep / Graphiti

> 记忆中的每个事实都有「保质期」——「张三是项目经理」在一月是真的，现在未必。

**关键词**：temporal reasoning, bi-temporal graph, invalid_at

### 它的核心主张

Zep 的核心信念是**世界是变化的，记忆必须反映变化**。「李四负责这个项目」这个事实，可能在 3 月之前是真的，4 月之后王五接手了。传统的向量数据库会把这两个矛盾的事实混在一起，而 Zep 用时间戳显式建模：每个图边有 valid_at 和 invalid_at，新的矛盾事实不会覆盖旧的，而是让旧的过期。

### 它怎么做的（最简版）

- 数据模型：每条事实是图的一条边，带 valid_at（何时生效）和 invalid_at（何时失效）时间戳
- 新事实与旧事实矛盾时：old.invalid_at = new.valid_at，历史完整保留
- 实体消解：精确匹配 → 模糊匹配（MinHash/LSH）→ LLM fallback
- Bi-temporal：同时追踪事件发生时间和系统知晓时间
- 查询时不需要 LLM，P95 < 300ms

### 精妙之处

**invalid_at 不是细节，是核心特性**。大多数框架把记忆当作静态存储，而 Zep 把记忆当作时间序列——任何事实都有时效，过期不等于错误，只是被新事实替代了。这对业务场景是刚需：供应商关系、组织架构、审批流程，都在不断变化。

### 局限性

- 需要图数据库（Neo4j/Kuzu），运维复杂度增加
- 比 Mem0 复杂一个数量级
- 仍然以对话为中心，对结构化业务知识的建模能力有限
- 实体消解的 LLM fallback 成本不可忽视

### 给 AgentCenter 的启示

**业务规则知识必须有时态**。在我们的「业务领域知识层」，供应商关系、审批流程、人员职责都是随时间变化的。Zep 的 bi-temporal 模型是唯一认真对待这个问题的方案。这不是可选项——对企业记忆系统来说，时态推理是基本要求。

## 4. Cognee

> 不是记忆 API，是数据到知识的管道——把所有东西都吃进去，吐出一个知识图谱。

**关键词**：poly-store, pipeline, offline-capable

### 它的核心主张

Cognee 相信**企业记忆的原材料不只是对话，而是everything**：文档、API 响应、数据库记录、PDF 报表。它不只是一个 RAG 库，而是一个 ETL 管道，把各种来源的数据转换成可查询的知识图谱。接入 source 数量和格式的广度是它最核心的差异化。

### 它怎么做的（最简版）

- 管道流程：classify → chunk → extract_graph(LLM) → add_data_points → extract_fk_edges
- Poly-store 架构：图（Neo4j/Kuzu/PG向量版）、向量（Chroma/LanceDB/pgvector）、关系型（SQLite/PG）可自由选择
- 30+ 数据源连接器：Slack、Notion、PDF、数据库等
- 完全离线运行，支持 Ollama

### 精妙之处

**Poly-store 架构——不锁定数据库**。这是企业架构的真实需求：不同类型的数据适合不同的存储引擎，强行统一到一个数据库是反模式。Cognee 的做法是让用户自己选择合适的存储组合，同时保持统一的查询接口。这个思路我们应该采纳。

### 局限性

- 管道延迟高，不是实时系统（batch 导向）
- 无时间推理能力
- 设置复杂，30+ 连接器意味着大量的配置工作
- 成熟度不如 Mem0/Zep，polish 程度较低

### 给 AgentCenter 的启示

**企业记忆的来源是多元的**——代码、文档、工单、审批记录、会议记录，我们的系统必须能 ingesting 所有这些。但 Cognee 是 batch 的，我们要 real-time。Cognee 的 poly-store 思路是架构层面的正确答案，但执行节奏我们要自己把握。

---

## 5. Graphify

> 任何输入都能变成知识图谱，既能机器查询，也能人类浏览——知识不应该只有 AI 能读。

**关键词**：confidence labels, Wiki output, multimodal

### 它的核心主张

Graphify 相信**知识要同时服务于机器和人类**。大多数框架把记忆设计成纯 API 可访问的格式，Graphify 多走了一步——输出人类可读的 Wiki。这意味着你既能写程序查询「哪些模块依赖 X」，也能让人直接打开 Wiki 浏览器看看「这个系统的结构是什么样的」。

### 它怎么做的（最简版）

- 管道：detect → extract(tree-sitter/Claude vision) → build_graph → cluster(Leiden) → analyze → export(Wiki/HTML/Obsidian)
- 每条边标注来源：EXTRACTED（直接提取）/ INFERRED（推断）/ AMBIGUOUS（不确定）——对置信度诚实
- 52 个文件语料库压缩了 71.5 倍 token
- 支持增量更新（SHA256 缓存）
- 无需外部数据库，本地文件即可

### 精妙之处

**置信度标签**。Graphify 对每条关系都标注了「我是怎么知道的」——是从代码直接提取的，还是 LLM 推断的，还是我自己也不太确定。这在企业场景中极其重要：决策者需要知道 AI 的判断有多可靠，不能把推断当事实用。

### 局限性

- CLI 工具，不是服务，集成成本高
- 批量处理，非实时
- 无时间推理能力
- 无持久化存储（SHA256 缓存只管增量，不保存图谱）

### 给 AgentCenter 的启示

**两个想法值得偷**：第一，置信度标签——我们的记忆系统应该对每条知识标注来源和可信度。第二，Wiki 输出——记忆要 human-browsable，不能只是黑盒 API。用户要能看到 agent「知道」了什么，这是企业信任的基础。

## 6. Hindsight

> 检索质量决定一切——用 4 种策略并行搜，用 cross-encoder 重排，用 reflect 做跨记忆推理。

**关键词**：91.4% LongMemEval, cross-encoder, reflect reasoning

### 它的核心主张

Hindsight 的信念是**「找到对的记忆」比「存储记忆」更重要**。不管你用多精妙的方式提取和存储记忆，如果检索时找不对，一切白搭。所以它把投资压倒性地放在检索端：4 路并行召回 + cross-encoder 重排 + reflect 操作（跨记忆推理）。91.4% 的 LongMemEval 得分遥遥领先证明了这条路线的有效性。

### 它怎么做的（最简版）

- 4 路并行检索策略：语义向量 + BM25 + 实体图 + 时间过滤，同时跑
- Cross-encoder rerank：对并行结果做精细化的相关性重排
- Reflect 操作：给定多条记忆，LLM 做跨记忆推理，不只是返回「哪条相关」，而是「这些记忆共同说明了什么」
- 专注检索层，不做记忆提取（假设上游已经有记忆了）

### 精妙之处

**Reflect——跨记忆推理而不是单条召回**。这是 Hindsight 与其他框架最大的区别。大多数 RAG 系统回答「根据这条文档，X 是对的」，Hindsight 回答「综合这 5 条记录，X 的判断依据是 A、B、C，且它们在逻辑上是一致的」。这是从「检索」到「洞察」的飞跃。

### 局限性

- Cross-encoder 引入额外延迟和成本
- 无知识图谱（纯检索系统）
- 依赖上游的记忆提取质量（自己不做提取）
- 还没有被广泛验证的生产级部署案例

### 给 AgentCenter 的启示

**91.4% vs 49% 的差距告诉我们：检索策略比存储设计更重要**。我们应该把最多的工程投入放在检索质量上，而不是存储格式。同时，reflect 操作——跨记忆推理——是我们系统必须支持的。agent 不应该只是「找到了相关文档」，应该是「综合分析了这些记录后，形成了洞察」。

## 7. LangMem

> 人类记忆有三种类型：语义记忆（事实）、情景记忆（经历）、程序记忆（技能）——对它们区别对待。

**关键词**：semantic/episodic/procedural, hot/background formation, prompt optimization

### 它的核心主张

LangMem 相信**不同类型的记忆需要不同的处理方式**。「Python 是编程语言」是语义记忆，「我第一次调试线上故障」是情景记忆，「如何做 code review」是程序记忆。它们形成方式不同、检索模式不同、更新频率不同，用同一种机制处理它们是认知上的懒惰。

### 它怎么做的（最简版）

- 三种记忆类型：Semantic（事实/知识，相对稳定）、Episodic（经历/事件，特定上下文）、Procedural（技能/流程，可复用）
- 记忆形成两个路径：Hot path（agent 对话中调用工具，同步形成）和 Background（对话后 ReflectionExecutor 异步处理）
- 内置 prompt 优化：3 种策略（fast/balanced/thorough）
- 存储层解耦，不依赖 LangGraph

### 精妙之处

**Hot path vs Background 的分离**——这本质上是在模拟「意识」和「潜意识」的分工。Agent 在对话中即时形成的记忆走热路径（快速但不深度），对话结束后的反思走冷路径（慢速但彻底）。这个模型映射到我们系统：实时代码补全需要热路径（低延迟），而知识库构建应该走冷路径（高质量）。

### 局限性

- 极其早期（v0.0.30），1.4K stars，社区小
- 无 benchmark 数据，无法与其他框架比较
- 无知识图谱，三种记忆的区分只在 prompt 层面
- 成熟度远低于其他框架

### 给 AgentCenter 的启示

**三种记忆类型的区分映射到我们的需求**很自然：业务规则是语义记忆，事故调查是情景记忆，调试流程是程序记忆。Hot/Background 分离是工程上的正确抽象——实时性要求和深度加工需求必须分别满足，不能用同一个管道。

## 8. Hermes Agent

> Agent 从经验中学习——复杂任务完成后，把「怎么解决的」存成技能，这是最高价值的记忆。

**关键词**：self-learning loop, bounded memory, 70+ tools

### 它的核心主张

Hermes 的信念是**最有价值的记忆不是事实，而是「上次我怎么解决这个问题」**。它不花太多精力在「记住什么是对的」上面，而是强调「记住我怎么解决的」。每次完成复杂任务，agent 自动提取经验教训，保存为可复用的技能（skills）。这个自学习循环才是终极记忆系统。

### 它怎么做的（最简版）

- 内置记忆：MEMORY.md（agent 笔记，2200 字符）和 USER.md（用户画像，1375 字符）——故意设置上限
- 8 个外部记忆 provider 作为插件（Mem0、Hindsight、Honcho 等），同一时间只能激活一个
- 全量对话历史的 FTS5 全文搜索
- 70+ 工具，28 个工具集，15+ 消息平台
- 自学习循环：任务完成 → 提取经验 → 存为技能

### 精妙之处

**自学习循环——任务完成后的自动经验提取**。这是 Hermes 区别于所有其他框架的核心。绝大多数记忆系统是被动的（你告诉它记住什么），Hermes 是主动的（任务完成后自动反思和保存）。这才是真正的「从经验中学习」。

### 局限性

- 记忆上限过于激进（2200+1375 字符），企业场景完全不够用
- 完整 agent 框架，不是轻量库
- 外部记忆 provider 只能选一个，多 provider 并行不支持
- 无知识图谱

### 给 AgentCenter 的启示

**两个想法值得偷**：第一，**自学习循环**——每个 workflow 执行完成后，系统应该自动提取「什么起了作用，什么没有」，并保存为经验。这正是程序记忆的来源。第二，**有界记忆强制修剪**——你不能无限堆积，必须决定什么值得记住。我们的三层记忆模型应该有明确的容量边界。

## 9. OpenHuman

> 记忆是一棵树，不是一个列表——结构化检索优于扁平相似度搜索。

**关键词**：memory tree, deterministic, three scopes, Obsidian

### 它的核心主张

OpenHuman 的信念是**向量数据库返回扁平结果，而树能保留层次**。一个列表里放着「设计原则、API 网关、缓存策略、订单模块」，你只能做相似度搜索。但一棵树可以按项目分层，可以按业务域分层，可以 drill-down——结构本身就是信息。这不只是存储格式的差异，而是认知模型的差异。

### 它怎么做的（最简版）

- 记忆树：确定性结构，三种作用域（source/topic/global）
- 三段式准入：≤0.15 直接丢弃，0.15-0.85 LLM 判断，≥0.85 直接接受——最小化 LLM 调用
- 6 个检索工具：query_source/topic/global, search_entities, drill_down, fetch_leaves——渐进式精化
- STM 双路径：FTS5 + cosine similarity 混合
- Obsidian 集成——记忆树直接导出为人类可读的笔记

### 精妙之处

**树结构保留语义层次**。代码有层次（package → class → method），组织有层次（公司 → 部门 → 团队），知识有层次（领域 → 子域 → 概念）。把这些扁平化成向量，丢失的恰恰是最重要的结构信息。OpenHuman 选择保留层次，这是一个正确但反主流的设计决策。

### 局限性

- GPL3 许可证，商业使用受限
- 桌面端应用，不是服务端
- Rust 技术栈贡献门槛高
- 无时间推理能力
- 早期 beta，稳定性存疑

### 给 AgentCenter 的启示

**三个想法值得偷**：第一，**树 > 扁平**——我们的代码上下文（package → class → method）和业务知识（领域 → 子域 → 概念）天然是树形结构，不应该扁平化成向量。第二，**三种作用域**（source/topic/global）对应我们的需求：按项目 / 按业务实体 / 跨项目全局。第三，**确定性处理**——相同输入产生相同树结构，这使得系统可审计，对企业合规至关重要。

## 跨框架启示总结

### 五个共识（大家都同意的）

1. **记忆需要主动修剪，不能无限堆积**。Mem0 的 self-edit、OpenHuman 的三段式准入、Hermes 的有界记忆，都指向同一个结论：存储容量有限，必须有淘汰机制。
2. **检索质量比存储格式更重要**。Hindsight 91.4% vs Mem0 49% 的差距证明，同样的记忆内容，不同的检索策略可以带来 40+ 个百分点的效果提升。
3. **多模态 ingestion 是必须的**。企业记忆的原材料是代码、文档、工单、会议记录、邮件——单一来源不够用。
4. **存储层要解耦，不要锁定数据库**。Cognee 的 poly-store 是架构层面的正确抽象，不是过度设计。
5. **记忆系统必须可审计**。Graphify 的置信度标签、OpenHuman 的确定性处理，都指向同一个企业级需求：用户需要知道系统「怎么知道的」。

### 四个分歧（大家吵的）

1. **Agent 自主管理 vs 系统强制管理**。Letta/Hermes 偏向 agent 自主决定记忆内容，其他框架偏向系统主导。企业场景更认可后者。
2. **时态推理是必须还是可选**。只有 Zep 认真对待，其他框架要么忽略要么还没做。这是当前最大的能力缺口。
3. **向量检索 vs 结构化检索**。Mem0/Hindsight 押注向量，OpenHuman 押注树形结构。这场争论还没定论。
4. **实时 vs Batch**。Cognee/Graphify 是 batch，Mem0/Letta 强调实时性。对企业工作流来说，两者都需要。

### 三个空白（没人做的，我们的机会）

1. **工程信号实时捕获**。9 个框架没有一个追踪 real-time engineering signals——代码提交、PR review、部署记录、故障事件。这是巨大的蓝海。AgentCenter 的记忆系统如果能从 IDE、CI/CD、监控系统直接拉取这些信号，将彻底改变 agent 的上下文质量。

2. **共享企业记忆池**。所有框架都是 per-user 或 per-agent 隔离模型，没有一个为「多个 agent 服务同一个组织」的场景设计。共享记忆、权限控制、版本管理、一致性保证——这些都是企业记忆系统的标配能力，目前是空白。

3. **Human-browsable 的记忆编辑界面**。只有 Graphify 和 OpenHuman 做了 Wiki/Obsidian 输出，大多数框架还停留在 API 层。对企业用户来说，能看到、审查、修改 AI 记忆的内容，是建立信任的前提。这块体验设计几乎没人做。
