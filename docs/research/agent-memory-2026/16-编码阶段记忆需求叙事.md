# 编码阶段记忆需求叙事

> AgentCenter 记忆系统调研 · 编码阶段专题
> 调研日期：2026-05-23
> 定位：从编码阶段的真实工作流出发，推导记忆系统需要什么能力

---

## 1. 编码阶段的核心记忆问题

编码阶段是软件生命周期中**高频率、高密度**的智力活动。与设计阶段相比，编码阶段的问题更具体、更琐碎，但每一决策的上下文丢失都会造成重复劳动和认知负担。编码阶段的开发者在本质上在三件事：

1. **写代码** — "这个方法怎么实现？参数是什么？返回值是什么？"
2. **改代码** — "为什么要改？改了什么影响？测试覆盖了吗？"
3. **查代码** — "这段逻辑在哪？谁写的？为什么这样写？"

这三件事的每一件都依赖记忆系统支撑，但现有工具链在这些记忆需求上存在明显空白。

### 问题一：上下文断裂 (Context Fracture)

**现象描述**：开发者切换任务后回来，Agent 忘记之前的代码决策；多人协作时，Agent 不知道别人改了什么；跨 session 的 debug 过程无法追踪。

**典型场景**：

```
开发者 A：
  "上周你帮我修的那个并发 bug，用的是读写锁还是 CAS？"
  → Agent 查不到上周的对话记录，查不到当时的决策 reasoning

开发者 B：
  "张工昨天改了 UserService 的 equals 方法，我今天动了一下为什么跑不过？"
  → Agent 不知道昨天发生了什么变更

跨 session：
  "这个 race condition 我调了两天，最后怎么修的来着？"
  → 之前的调试过程、尝试过的方案、最终解法全部丢失
```

**根本原因**：现有 Agent Session 之间是隔离的，代码变更通过 git 管理但 Agent 记忆不与 git 同步，调试过程中的隐性知识（tacit knowledge）没有记录机制。

**影响**：开发者被迫在内部记忆和外部载体（笔记、文档、Slack）之间手动同步，造成认知负担和信息丢失。

### 问题二：知识沉淀缺失 (Knowledge Evaporation)

**现象描述**：踩过的坑、调试经验没有沉淀；性能调优的决策依据丢失；架构权衡的 reasoning 没有记录。

**典型场景**：

```
"为什么这里用了 Redis 而不是本地缓存？当时怎么考虑的？"
  → 代码里只有 result，没有 reasoning
  → 三个月后连当事人都不记得了

"这个接口为什么要加分布式锁？不加会怎样？"
  → 没有记录当时的 race condition 复现过程
  → 后来者以为这是过度设计

"这个 batch size 为什么设成 500？"
  → 没有记录是从 1000 降到 500 的血泪教训
  → 后来者可能直接改成更大的值引发 OOM
```

**根本原因**：代码只记录"是什么"，不记录"为什么"和"怎么来的"。代码审查（Code Review）只关注正确性，不关注决策过程。CI/CD 记录构建结果，不记录构建过程中的临时方案和放弃的路径。

**影响**：团队在同一个坑里反复摔，每次踩坑都是新的踩坑，没有形成组织记忆。

### 问题三：个性化不足 (Personalization Deficit)

**现象描述**：Agent 不了解团队的代码风格偏好；不知道项目的命名约定、错误处理模式；不了解测试覆盖率要求、review 标准。

**典型场景**：

```
"我们团队 prefer 早期 return 而不是嵌套 if"
  → Agent 生成的代码充满嵌套 if-else，分支层级深

"我们用 Result<T> 而不是抛异常"
  → Agent 写的代码到处是 throw new RuntimeException

"review 时我们要求必须有 @since 注解"
  → Agent 不知道这个团队规范

"我们不喜欢用 Optional，嫌啰嗦"
  → Agent 到处给你包装 Optional
```

**根本原因**：Agent 的默认行为来自预训练，缺乏对特定团队规范的感知。代码风格配置（.editorconfig、lint 规则）解决了格式化问题，但解决不了更高层次的约定（设计模式偏好、错误处理策略、测试覆盖率目标）。

**影响**：Agent 生成的代码需要大量人工 review 和修改才能符合团队标准，开发者需要反复纠正 Agent 的行为，效率提升有限。

## 2. 编码阶段记忆类型需求

编码阶段涉及的记忆类型比设计阶段更细粒度、更高频。下表按记忆类型分类，给出对应的编码场景、具体示例和优先级。

### 记忆类型与编码场景映射表

| 记忆类型 | 编码场景 | 具体示例 | 优先级 |
|---------|---------|---------|--------|
| **事实 (Fact)** | API 契约、接口定义、database schema、配置项 | "User 表有 15 个字段，主键是 ULID，不是自增 ID" | **P0** |
| **事件 (Event)** | Bug 修复历史、deployment 记录、incident 处理 | "2026-05-20 修复了 #123 并发 bug，根因是 race condition，用 CAS 替换 synchronized" | **P0** |
| **过程 (Procedure)** | 调试步骤、构建流程、环境配置步骤 | "复现 race condition: 1. 启动 3 个 worker 2. 并发写 shared_state 3. 检查 ERROR log" | **P1** |
| **偏好 (Preference)** | 代码风格、review 标准、测试要求 | "团队使用早期 return 避免嵌套 if；错误处理用 Result<T> 而非抛异常" | **P1** |
| **规则 (Rule)** | 编码规范、合规要求、安全要求 | "所有 public API 必须有 @RateLimiter 注解；金额计算超过 1000 元需二级审批" | **P0** |

### 记忆类型详细说明

#### 事实 (Fact) — P0

编码阶段的事实记忆是最硬的记忆，不容模糊。API 签名、schema 定义、配置项值——这些必须是精确的。

**典型事实记忆内容**：

- 接口签名（方法名、参数类型、返回值类型）
- Database schema（表名、字段名、字段类型、约束）
- 配置值（超时时间、重试次数、batch size）
- 依赖版本（Spring Boot 3.4.5, MyBatis 3.0.4）
- 环境变量名称和用途

**质量标准**：精确匹配，不接受模糊或近似。

#### 事件 (Event) — P0

编码阶段的事件记忆是线性的，需要按时间顺序组织。每个事件有时间戳、参与者、变更内容、变更原因。

**典型事件记忆内容**：

- Commit 记录（谁在什么时候改了什么，为什么改）
- Bug 修复记录（什么时候发现、什么时候修好、怎么修的）
- Deployment 记录（什么时候部署了什么版本，有什么问题）
- Incident 记录（什么时候出了什么故障，怎么恢复的）
- API 变更记录（接口什么时候加了参数、改了返回值语义）

**质量标准**：可溯源、可关联。任何一个事件应该能追溯到具体的 commit、PR 或 incident ticket。

#### 过程 (Procedure) — P1

编码阶段的过程记忆是操作性的，记录"怎么做"。调试步骤、环境配置步骤、构建步骤都是典型的过程记忆。

**典型过程记忆内容**：

- 调试步骤（如何复现一个 bug，分几步）
- 环境配置步骤（本地环境怎么配，涉及几个配置文件）
- 构建步骤（如何打一个可部署的包，要过哪些检查）
- Release 步骤（如何发布一个新版本，涉及哪些系统和审批）

**质量标准**：可重现。其他开发者按照过程记忆能够复现同样的结果。

#### 偏好 (Preference) — P1

编码阶段的偏好记忆是动态的，随团队和项目变化。代码风格偏好、review 标准偏好、测试覆盖率偏好都需要记忆。

**典型偏好记忆内容**：

- 代码风格（命名约定、格式化规则、注释要求）
- 错误处理策略（用 Result 还是异常、日志规范）
- 测试要求（覆盖率阈值、必须有的测试类型）
- Review 标准（什么必须 review、什么可以 waive）
- 文档要求（什么时候要写 javadoc、README 要求多详细）

**质量标准**：可应用到新代码生成。Agent 知道了这个偏好后，生成的新代码应该符合要求，不需要每次都纠正。

#### 规则 (Rule) — P0

编码阶段的规则记忆是强制性的，违反规则会导致合规问题或生产故障。编码规范、安全要求、合规要求都属于规则记忆。

**典型规则记忆内容**：

- 安全规则（不能明文存储密码、API 必须鉴权）
- 合规规则（数据保留周期、审计日志要求）
- 编码规范（所有 public API 必须有文档、所有 mutation 必须有测试）
- 架构规则（不能跨层直接调用、数据库访问必须通过 Repository）

**质量标准**：可检查、可拦截。在 Agent 生成代码时能够发现违反规则的地方并主动提示或阻止。

## 3. 编码阶段检索模式

编码阶段的检索模式分为三种，各有不同的时间跨度和精确度要求。

### 模式一：精确查找 (Exact Lookup)

**查询特征**：查询内容明确，期望精确答案，不接受模糊匹配。

**典型查询**：

```
"UserService 的 create 方法在哪？签名是什么？"
"OrderMapper.xml 里 updateOrderStatus 的 SQL 是什么？"
"这个字段在哪些地方被引用了？"
"lastModifiedBy 这个字段是什么时候加的？"
```

**检索需求**：

- 精确关键词匹配
- 代码结构索引（package → class → method）
- 完整的引用链追踪
- 变更历史（什么时候改的、谁改的）

**现有工具**：LSP (Language Server Protocol)、grep、AST search、GitNexus code intelligence

**记忆补充**：仅靠现有工具不够——它们能找到代码在哪，但无法回答"为什么这样写"和"当时考虑了哪些替代方案"。记忆系统需要补充决策 reasoning 和上下文。

**检索质量标准**：

| 维度 | 标准 |
|------|------|
| 召回率 | 100%，不能遗漏任何匹配项 |
| 精确率 | 必须精确到具体的行/方法/signature |
| 延迟 | 必须毫秒级响应，不能超过 500ms |
| 上下文 | 返回结果时需要包含周围 5 行以内的上下文 |

### 模式二：语义关联 (Semantic Association)

**查询特征**：查询内容不明确，依赖语义相似度和概念关联。

**典型查询**：

```
"之前类似的并发问题是怎么解决的？"
"有没有用过类似的消息队列消费模式？"
"这个地方之前出过什么 bug？"
"有没有其他地方用了类似的缓存策略？"
```

**检索需求**：

- 语义相似度计算
- 跨文件、跨模块关联
- 概念图谱（哪些概念相关联）
- 历史经验召回（调试经验、最佳实践、踩坑记录）

**现有工具**：向量搜索、code embedding（如 CodeBERT、GraphCodeBERT）

**记忆补充**：现有 embedding 工具擅长代码相似度，但不理解"这个代码是为了解决什么问题"。记忆系统需要补充问题的语义描述和解决方案的 reasoning。

**检索质量标准**：

| 维度 | 标准 |
|------|------|
| 召回率 | 前 5 个结果中应包含相关经验 |
| 排序 | 最相关的经验排在最前面 |
| 延迟 | 应在 1 秒以内返回结果 |
| 可解释性 | 返回结果需要附带匹配原因，便于开发者判断相关性 |

### 模式三：时序推理 (Temporal Reasoning)

**查询特征**：查询内容涉及时间线和因果链，需要追踪演变过程。

**典型查询**：

```
"这个模块的架构是怎么演变的？"
"为什么从单例改成了依赖注入？"
"这个接口的第一个版本是什么样的？"
"这个字段加进来的时候有没有引起什么问题？"
```

**检索需求**：

- 时间线追踪（按时间顺序组织事件）
- 变更链（每个变更的前因后果）
- 因果推理（为什么做了这个决定）
- 对比能力（版本 A vs 版本 B 的差异）

**现有工具**：git log、git blame、git diff

**记忆补充**：git 记录了变更的事实，但不记录决策的 reasoning。"从单例改成 DI"这个 commit 记录了 what，但没有记录 why（为什么要改？是因为测试困难还是扩展性需求？）。

**检索质量标准**：

| 维度 | 标准 |
|------|------|
| 完整性 | 必须包含完整的变更链，不能断链 |
| 准确性 | 每个变更的原因必须有据可查 |
| 可视化 | 需要支持以时间线方式展示 |
| 关联性 | 每个变更需要关联到具体的业务背景或技术债务 |

## 4. 编码阶段 vs 设计阶段对比

编码阶段和设计阶段虽然都属于软件生命周期的前半段，但记忆需求有本质差异。

### 核心维度对比

| 维度 | 设计阶段 | 编码阶段 |
|------|---------|---------|
| **事实粒度** | 需求、业务规则、流程定义 | API 签名、schema、配置项 |
| **事件粒度** | 评审会、决策会议、需求变更 | commit、PR、deploy、incident |
| **检索频率** | 低频高价值（每天几次但每次重要） | 高频高价值（每天数十次） |
| **时序跨度** | 周/月级（设计周期长） | 天/周级（代码变更频繁） |
| **精确度要求** | 模糊可接受（设计是草稿） | 必须精确（代码是确定行为） |
| **实时性** | 低，可异步（设计可以缓一缓） | 高，需要即时上下文（写代码时不能等） |
| **知识来源** | 文档、会议纪要、需求列表 | 代码、git、CI/CD、PR review |
| **最关键的记忆类型** | 规则 > 事实 > 事件 | 事实 > 事件 > 过程 > 规则 |

### 记忆消费者对比

| 维度 | 设计阶段 | 编码阶段 |
|------|---------|---------|
| **主要消费者** | 设计师、架构师 | 开发者、reviewer |
| **消费者数量** | 少（架构团队） | 多（全团队） |
| **消费者专业度** | 高（需要架构思维） | 中等（需要工程思维） |
| **记忆查询复杂度** | 低（关键词+简单语义） | 高（精确+语义+时序混合） |
| **记忆更新频率** | 低（设计稳定） | 高（代码持续变更） |

### 记忆来源对比

| 维度 | 设计阶段 | 编码阶段 |
|------|---------|---------|
| **主要来源** | PRD、会议纪要、文档 | 代码、git、CI/CD |
| **来源格式** | 文本为主、结构化程度低 | 代码为主、结构化程度高 |
| **来源可靠性** | 中（文档可能过时） | 高（代码是事实） |
| **来源可解析性** | 低（自然语言难解析） | 高（代码有明确语法树） |
| **自动提取可行性** | 低（需要 NLP + 推理） | 高（AST + git log 可自动提取） |

### 记忆系统能力需求对比

| 能力 | 设计阶段需求 | 编码阶段需求 |
|------|------------|------------|
| **精确检索** | 中 | **高**（必须精确到方法签名） |
| **时序推理** | 中 | **高**（需要知道什么时候改了什么） |
| **多跳推理** | 高 | **中高**（影响链分析） |
| **实时索引** | 低 | **高**（代码变更后立即可查） |
| **冲突检测** | 中 | **高**（不能有矛盾事实） |
| **上下文窗口** | 中 | **大**（编码涉及长文件、多模块） |
| **增量更新** | 低 | **必须**（代码频繁变更） |

## 5. 编码阶段 5 个 P0 记忆需求

基于以上分析，编码阶段有 5 个 P0 级别的记忆需求，这些需求如果不满足，Agent 在编码阶段的表现将严重受限。

### 需求一：代码决策记忆 (Code Decision Memory)

**定义**：记录"为什么这样写"的 reasoning，支持后续查询。

**解决的问题**：当开发者问"为什么这里用 CAS 而不是 synchronized"时，系统能够给出当时的决策依据。

**记忆内容**：

- 决策时间、决策者
- 考虑过的替代方案
- 选择当前方案的原因
- 方案的优缺点分析
- 相关的技术约束和业务背景

**使用场景**：

```
Query: "这里为什么用 ConcurrentHashMap 而不是 synchronizedMap？"
Memory: "2026-05-15，张工在实现缓存层时决策：
  - 考虑过 synchronizedMap：简单但并发度低，高写入场景会成瓶颈
  - 考虑过读写锁：读多写少场景合适，但实现复杂度高
  - 最终选 ConcurrentHashMap：并发度高，实现简单，适合当前读多写少场景
  - 相关 issue：#456"
```

**质量标准**：

- 决策 reasoning 必须有据可查
- 能够关联到具体的代码变更
- 支持按时间线回溯决策演变

### 需求二：Bug 调试记忆 (Bug Debug Memory)

**定义**：记录 bug 症状、根因、修复方案的完整链条，支持跨 session 追踪。

**解决的问题**：开发者花了 2 天调试的 bug，修好后记忆随之消失，下次遇到类似问题要从零开始。

**记忆内容**：

- Bug 初次发现时间、环境、症状描述
- 根因分析过程（尝试过的排错方法、中途放弃的假设）
- 最终根因
- 修复方案及选择原因
- 相关的测试用例（防止 regression）
- 预防措施（如何避免类似 bug 再次发生）

**使用场景**：

```
Query: "有没有遇到过类似的 NullPointerException 在并发场景下？"
Memory: "2026-04-20 记录：
  - Bug ID: INC-2026-0420
  - 症状：偶发性 NPE，出现在高并发写入时
  - 排查路径：
    1. 排除空指针来源（检查了 12 个可能的 null 引用点）
    2. 加上详细日志后发现是 HashMap 并发扩容导致的死链
    3. 根因：putIfAbsent 和 computeIfAbsent 在并发时的交互问题
  - 修复：替换为 ConcurrentHashMap 的 computeIfAbsent
  - 测试：加了对应的 concurrency test，1000 并发 0 失败"
```

**质量标准**：

- Bug 调试过程可完整回放
- 根因到修复方案的逻辑链清晰
- 相关的测试用例可关联召回

### 需求三：API 契约记忆 (API Contract Memory)

**定义**：追踪接口定义的变更历史，知道"什么时候改了什么、为什么改"。

**解决的问题**：当 API 变更导致调用方出错时，能够快速定位"什么时候改的、谁改的、为什么改"。

**记忆内容**：

- API 签名（方法名、参数、返回值）
- 每个版本的变更内容
- 变更原因和背景
- 相关的 PR 或 issue
- 向下兼容或 breaking change 标记
- 调用方信息（谁在用这个 API）

**使用场景**：

```
Query: "UserService.createUser 的返回值什么时候从 User 改成了 Result<User>？"
Memory: "变更历史：
  - 2026-03-01 (v1.0)：初始版本，返回 User，抛异常表示错误
  - 2026-04-15 (v1.2)：改成返回 Result<User>，符合团队错误处理规范
    PR: #789，reviewer: 李工
    原因：团队 decision，所有 API 用 Result<T> 而非异常
  - 2026-05-10 (v1.5)：参数加了 optional locale 字段
    PR: #1234
    原因：国际化需求，支持指定locale创建用户"
```

**质量标准**：

- API 变更时间线完整
- 每个变更可追溯到具体的 PR
- Breaking change 必须有明确标记

### 需求四：团队偏好记忆 (Team Preference Memory)

**定义**：学习并应用团队的代码风格和 review 标准，Agent 生成的代码应符合团队规范。

**解决的问题**：Agent 每次生成的代码风格不一致，需要开发者反复纠正。

**记忆内容**：

- 代码风格偏好（命名约定、格式化规则）
- 错误处理策略（用 Result 还是异常、日志级别）
- 测试覆盖率要求（百分比、必须覆盖的场景）
- Review 标准（什么必须 review、什么可 waive）
- 文档要求（javadoc 必需程度、README 更新时机）
- 禁止的模式（不使用全局变量、不使用 Thread.sleep 等待）

**使用场景**：

```
Query: "生成一个新的 Service 类"
Memory (团队偏好):
  - 命名：UserService, OrderService（名词+Service）
  - 错误处理：Result<T> 而非 throw
  - 日志：用 SLF4J，ERROR 级别才打
  - 测试：必须有对应的 *ServiceTest 类
  - javadoc：每个 public 方法必须有 @param @return
  - 禁止：不允许用 System.out.println
Agent 输出：符合上述所有偏好，不需要手动调整
```

**质量标准**：

- 偏好更新后，新生成的代码立即反映新偏好
- 开发者可以通过对话更新偏好
- 偏好可以按项目、团队、个人多级别覆盖

### 需求五：构建/部署记忆 (Build/Deploy Memory)

**定义**：记录构建配置、部署流程、环境差异，支持可复现的构建和部署。

**解决的问题**：构建失败时不知道是配置问题还是代码问题；部署后出问题不知道是哪个环境的差异。

**记忆内容**：

- 构建配置（maven/gradle 版本、依赖版本、profile）
- 部署流程（步骤、审批、环境）
- 环境差异（dev/staging/prod 的配置差异）
- 部署历史（什么时候部署了什么版本、结果如何）
- 常见构建失败的原因和解决方案
- 环境变量的用途和当前值

**使用场景**：

```
Query: "为什么本地 build 成功但 CI 失败？"
Memory:
  - 本地 Java: 17.0.3, CI Java: 17.0.2（minor version 差异）
  - CI 多一步：mvn verify -P sonar（本地默认不跑）
  - 失败原因：sonar profile 要求 80% 覆盖率，本地覆盖率 65%
  - 解决方案：
    1. 本地跑 mvn verify -P sonar 本地验证
    2. 或者降低覆盖率要求到 70%（需 team lead 审批）"
```

**质量标准**：

- 构建配置可完整复现（任何机器上能跑出相同结果）
- 环境差异有明确文档
- 部署流程可自动化执行

## 6. 对记忆系统的技术要求

编码阶段对记忆系统的技术要求比设计阶段更高，主要体现在精确度、实时性和增量更新方面。

### 技术能力需求表

| 能力 | 要求等级 | 具体要求 | 原因 |
|------|---------|---------|------|
| **精确检索** | 必须 | "User 表有多少字段" 不能模糊回答；必须精确返回 15 | 编码是精确行为，模糊答案会导致 bug |
| **时序推理** | 高 | 需要知道 API 什么时候改的、为什么改、谁改的 | 代码变更需要了解上下文 |
| **多跳推理** | 中高 | "改了这个字段的地方有哪些？影响哪些测试？" | 影响分析是编码的日常需求 |
| **实时索引** | 高 | 刚改的代码要立即可查，不能有延迟 | 编码时不能等索引更新 |
| **冲突检测** | 高 | 同一事实不能矛盾，如 "主键是 ULID" 不能同时存在 "主键是自增ID" | 编码需要可信的事实来源 |
| **上下文窗口** | 大 | 编码涉及长文件、多模块、跨文件调用 | Agent 需要看到足够上下文才能正确修改 |
| **增量更新** | 必须 | 代码频繁变更，记忆需要同步更新，不能全量重建 | 实时性要求排除了定期全量索引 |
| **结构化提取** | 高 | 能够从代码、git log、CI/CD 输出中自动提取结构化事实 | 开发者不会手动维护记忆 |
| **语义理解** | 中 | 理解代码的语义而不只是语法 | "并发场景"和"多线程场景"是同一语义概念 |
| **可解释性** | 中 | 记忆召回时需要说明为什么召回这条记忆 | 开发者需要判断记忆的相关性 |

### 精确度 vs 召回率权衡

| 场景 | 精确度要求 | 召回率要求 | 说明 |
|------|-----------|-----------|------|
| API 签名查询 | 100% | 100% | 必须精确，不能遗漏 |
| Bug 调试记忆 | 90% | 80% | 可以召回相似 bug，但需要明确说明不是同一个 |
| 团队偏好 | 95% | 95% | 偏好必须一致，但可以有少量例外 |
| 代码决策 | 80% | 70% | 决策 reasoning 难以完全精确，模糊匹配可接受 |
| 构建配置 | 100% | 100% | 配置值必须完全精确 |

### 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 查询延迟 P50 | < 100ms | 常规查询 |
| 查询延迟 P99 | < 500ms | 复杂查询（多跳推理） |
| 索引更新延迟 | < 5s | 代码变更后 5 秒内可查 |
| 支持并发查询 | > 100 QPS | 多个开发者同时使用 |
| 记忆存储容量 | > 100GB | 支撑中大型项目多年积累 |

## 7. 编码阶段记忆工作流

编码阶段的记忆流转遵循以下工作流，从开发者提问到记忆沉淀形成闭环。

### 记忆流转图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   开发者提问                                                             │
│       ↓                                                                 │
│   Agent 检索记忆                                                        │
│       ↓                                                                 │
│   组装上下文（记忆 + 当前代码 + 项目配置）                                  │
│       ↓                                                                 │
│   生成 / 审查 / 调试代码                                                 │
│       ↓                                                                 │
│   决策沉淀                                                             │
│       ↓                                                                 │
│   提取结构化记忆（事实 / 事件 / 过程 / 偏好 / 规则）                          │
│       ↓                                                                 │
│   记忆持久化                                                            │
│       ↓                                                                 │
│   下次相关查询 ←──────── 记忆被召回                                      │
│       ↑                                                                 │
└─────────────────────────────────────────────────────────────────────────┘
```

### 各环节详细说明

#### 第一步：开发者提问

开发者通过自然语言向 Agent 提问。问题可能涉及：

- 精确查找（"UserService 在哪个文件"）
- 语义关联（"之前有没有遇到过类似问题"）
- 时序推理（"这个接口什么时候改的"）
- 代码生成（"帮我写一个处理订单创建的 Service"）

#### 第二步：Agent 检索记忆

Agent 根据问题类型选择检索策略：

| 问题类型 | 检索策略 |
|---------|---------|
| "UserService 在哪" | 精确查找：LSP + 记忆中的位置信息 |
| "之前类似 bug 怎么修的" | 语义关联：相似度召回 + 时间线过滤 |
| "这个接口为什么改成这样" | 时序推理：git history + 决策记忆 |
| "帮我写代码" | 代码生成 + 团队偏好记忆注入 |

#### 第三步：组装上下文

Agent 将检索到的记忆与当前状态组装成完整上下文：

- 记忆层：历史决策、调试经验、团队偏好
- 代码层：当前代码、依赖关系、测试文件
- 配置层：项目配置、环境变量、构建参数

#### 第四步：生成 / 审查 / 调试代码

Agent 在完整上下文中执行任务：

- **代码生成**：按照团队偏好生成符合规范的代码
- **代码审查**：按照团队 review 标准检查代码
- **Bug 调试**：结合历史调试记忆分析当前问题

#### 第五步：决策沉淀

任务完成后，Agent 从交互中提取值得沉淀的记忆：

- 明确的决策（"这里用 X 是因为 Y"）
- 调试过程中的发现（"根因是 Z"）
- 团队偏好（"这个场景应该用 Result<T>"）
- 规则违反（"这个修改违反了 @RateLimiter 规则"）

#### 第六步：记忆持久化

提取的记忆以结构化形式存储：

| 记忆类型 | 存储结构 |
|---------|---------|
| 事实 | { type: "fact", entity: "UserService", attribute: "primaryKey", value: "ULID", source: "code", updatedAt: "..." } |
| 事件 | { type: "event", action: "bug_fix", subject: "#123", detail: "...", timestamp: "..." } |
| 过程 | { type: "procedure", name: "debug_race_condition", steps: [...], applicableTo: "..." } |
| 偏好 | { type: "preference", dimension: "error_handling", value: "Result<T>", team: "backend", updatedBy: "..." } |
| 规则 | { type: "rule", content: "@RateLimiter required", enforcedAt: "public_api", severity: "must" } |

#### 第七步：下次相关查询

当有新的相关查询时，记忆被召回。好的记忆系统应该：

- 主动推荐相关记忆（不只是被动响应）
- 在生成代码前检查是否违反已有规则
- 在调试时主动展示相关的历史 bug 记忆

### 工作流优化点

| 环节 | 当前痛点 | 优化方向 |
|------|---------|---------|
| 开发者提问 | 问题表述模糊 | Agent 主动追问或建议 |
| 记忆检索 | 召回太多噪音 | 更好的排序和过滤 |
| 上下文组装 | 需要手动拼凑 | Agent 自动组装 |
| 决策沉淀 | 需要开发者确认 | Agent 主动沉淀，开发者可撤回 |
| 记忆召回 | 不知道有相关记忆 | Agent 主动推荐 |

## 8. AgentCenter 编码阶段记忆优先级建议

基于以上分析，建议分四个 Phase 实施编码阶段记忆系统，逐步构建完整能力。

### Phase 1：事实记忆 + 精确检索（API 契约、Schema）

**目标**：解决"这个 API 签名是什么、这个字段叫什么"这类最基础的记忆问题。

**核心功能**：

- API 签名索引（Controller、Service、Repository 的方法签名）
- Database schema 索引（表名、字段、类型、约束）
- 配置项索引（application.yml、application.properties 的 key-value）

**实现方案**：

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 索引 | LSP + AST parser | 从代码直接提取结构化信息 |
| 存储 | SQLite（与 Bridge 共用） | 复用现有数据库 |
| 检索 | 精确匹配 | 方法签名、字段名精确查找 |
| 更新 | 增量更新 | 代码变更后 5 秒内更新索引 |

**成功标准**：

- 开发者问"UserService 有哪些方法"，Agent 能准确列出方法名和签名
- 开发者问"User 表的主键是什么"，Agent 能准确回答
- 索引更新延迟 < 5 秒

**与现有系统集成**：

- 利用 GitNexus 的代码索引能力
- 与 Flyway migration 集成，自动追踪 schema 变更

### Phase 2：事件记忆 + 时序推理（变更历史、Bug 链条）

**目标**：解决"这个什么时候改的、为什么改"这类问题。

**核心功能**：

- Commit 记忆（关联 commit message、PR、reviewer）
- Bug 修复记忆（关联 issue、复现步骤、根因）
- API 变更记忆（关联 breaking change、调用方）
- 时序推理（按时间线展示变更）

**实现方案**：

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 来源 | Git hooks + CI/CD | 自动提取 commit 和部署事件 |
| 存储 | 图数据库 (Neo4j) | 适合变更链的多跳关系 |
| 检索 | 时序查询 | 支持按时间范围、关联性召回 |
| 关联 | 自动关联 | commit ↔ issue ↔ test ↔ API |

**成功标准**：

- 开发者问"这个接口什么时候加的参数"，Agent 能给出准确时间和 PR
- 开发者问"这个 bug 之前有没有类似问题"，Agent 能召回相关 bug
- 变更链可完整回放

**与现有系统集成**：

- 与 GitHub/GitLab API 集成，自动同步 PR 和 review 信息
- 与 JIRA/Linear 集成，自动关联 issue

### Phase 3：过程记忆 + 语义关联（调试经验、最佳实践）

**目标**：解决"之前类似问题怎么解决的"这类语义相似问题。

**核心功能**：

- 调试经验记忆（复现步骤、根因分析、修复方案）
- 最佳实践记忆（团队认可的解决方案）
- 语义检索（代码 embedding + 记忆 embedding）
- 跨模态关联（文字记忆 ↔ 代码片段 ↔ 文档）

**实现方案**：

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| Embedding | CodeBERT / GraphCodeBERT | 代码语义向量 |
| 存储 | 向量数据库 (Qdrant/Milvus) | 高维向量相似度检索 |
| 召回 | 混合检索 | 关键词 + 向量 + 知识图谱 |
| 更新 | 自动 + 手动 | CI/CD 自动沉淀 + 开发者标注 |

**成功标准**：

- 开发者问"有没有遇到过类似的 race condition"，Agent 能召回相关的调试记忆
- 召回的记忆中，80% 与当前问题相关
- 语义检索延迟 < 1 秒

**与现有系统集成**：

- 与 IDE 集成，在开发者调试时主动记录过程
- 与知识库（Confluence、Notion）集成

### Phase 4：偏好 / 规则记忆 + 自动应用（代码风格、Review 标准）

**目标**：让 Agent 生成的代码直接符合团队规范，不需要反复纠正。

**核心功能**：

- 团队偏好记忆（代码风格、错误处理策略、测试要求）
- 编码规则记忆（安全规范、合规要求）
- 自动应用（代码生成时自动注入偏好）
- 主动检查（生成代码时主动检查规则违反）

**实现方案**：

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 偏好存储 | 结构化 KV | { dimension: "error_handling", value: "Result<T>", team: "backend" } |
| 规则存储 | 规则引擎 (Drools/O Rule) | 支持复杂条件判断 |
| 应用层 | Pre-commit hook + Review bot | 生成前检查 + 生成后 review |
| 学习 | 反馈循环 | 开发者纠正时自动更新偏好 |

**成功标准**：

- Agent 生成的代码中，90% 符合团队偏好
- 规则违反能在代码写入前被拦截
- 开发者可以通过对话更新偏好，更新后立即生效

**与现有系统集成**：

- 与 CI/CD 集成，在构建阶段检查规则
- 与 Code Review 系统集成，自动检查 PR 违反规则
