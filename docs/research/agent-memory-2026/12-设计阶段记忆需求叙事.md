# 设计阶段记忆需求叙事

> AgentCenter 记忆系统调研 · 场景分析 #1：设计阶段
> 调研日期：2026-05-23
> 定位：从设计阶段的真实工作流出发，推导记忆系统需要什么能力

## 1. 设计阶段在做什么

企业软件的设计师（或架构师）在设计阶段本质上在做三件事：

1. **理解需求** — "业务方要什么？约束是什么？以前怎么做的？"
2. **做决策** — "选 A 方案还是 B 方案？为什么？上次类似场景选的什么？"
3. **表达设计** — "接口怎么定义？数据模型怎么画？和现有系统怎么对接？"

这三件事的每一件都需要不同类型的记忆支撑。

## 2. 三类核心问题

### 2.1 第一类："业务规则是什么？"

**典型提问**："采购审批流程超过 50 万的谁批？"

这类问题需要的是**业务领域知识**。特点：
- 事实会变 — 去年超过 50 万总监批，今年改成了 VP 批
- 需要溯源 — "这个规则是哪份制度文件定的？"
- 跨系统关联 — ERP 里的审批规则和 OA 里的可能不一样

**需要召回的东西**：
- 业务规则文档（制度、流程规范）
- 历史设计决策（上次为什么这么设计的）
- 相关人员的角色和职责

**对应评测维度**：Temporal Reasoning（时态推理）+ Knowledge Update（知识更新）

Zep/Graphiti 的 invalid_at 机制就是为这类问题设计的 — 系统必须知道"当前生效的规则是什么"，而不是把所有版本的规则混在一起。

### 2.2 第二类："现有系统长什么样？"

**典型提问**："订单模块现在谁在维护？它依赖哪些服务？上次改它是什么时候？"

这类问题需要的是**代码实时上下文**。特点：
- 需要结构 — 不是搜"订单"两个字就行，要知道 package → service → method 的层次
- 需要关联 — 订单 Service 调了库存 Service，库存 Service 又调了 ERP Adapter
- 需要新鲜度 — 上周刚重构过，记忆必须是新的

**需要召回的东西**：
- 代码结构（GitNexus 已有的能力）
- 最近的变更（谁改了什么、为什么改）
- 依赖关系和服务拓扑

**对应评测维度**：Information Extraction（信息提取）+ Multi-hop Reasoning（多跳推理）

OpenHuman 的树结构思路很对口 — 代码本身就是 package → class → method 的树，按树召回比按向量召回更精准。GitNexus 已经有 28205 个关系，但缺的是变更信号 — 谁最近改了这块代码，为什么改。

### 2.3 第三类："以前类似问题怎么解决的？"

**典型提问**："上次做支付对接的时候，选的是同步还是异步？踩了什么坑？"

这类问题需要的是**经验记忆**。特点：
- 跨会话 — 上次讨论可能在一个月前的另一个项目里
- 需要推理 — 不是直接找"支付"关键词就行，要把 5 次会议的结论串起来
- 需要洞察 — 不是返回 5 条记录，而是告诉设计师"综合来看，异步方案更合适，因为…"

**需要召回的东西**：
- 历史设计讨论的结论
- ADR（架构决策记录）
- Incident 复盘（出了什么问题、怎么修的）

**对应评测维度**：Multi-Session Reasoning（多会话推理）+ Preference Following（偏好跟随）

Hindsight 的 reflect 操作要解决的就是这件事 — 不只是检索，而是跨记忆推理出洞察。

## 3. 召回策略：应该怎么做？

设计阶段的召回不是"用户说一句话，系统返回 N 条相似记录"。应该是**渐进式的三步走**：

### 第一步：理解意图

设计师说"帮我设计一下退款接口" — 这句话背后至少包含三个意图：
- 查业务规则（退款的政策是什么？）
- 查代码现状（现在有没有退款相关代码？）
- 查历史经验（以前做过退款吗？踩过什么坑？）

系统需要同时命中三类记忆，不能只搜一类。

### 第二步：分层召回

| 召回层 | 召回什么 | 用什么策略 | 参考框架 |
|--------|---------|-----------|---------|
| 业务知识层 | 退款规则、审批流程、金额阈值 | 时态图谱查询（当前生效的规则） | Zep/Graphiti |
| 代码上下文层 | RefundService、依赖链、最近变更 | 树结构 drill-down（从模块到方法） | OpenHuman + GitNexus |
| 经验记忆层 | 上次退款重构的 ADR、incident 复盘 | 跨会话推理 + reflect | Hindsight |

三层并行召回，然后 cross-encoder 重排，最后 reflect 综合出洞察。这就是 Hindsight 91.4% vs Mem0 49% 证明的 — 检索策略比存储设计更重要。

### 第三步：渐进精化

设计师看了第一轮结果后说"退款金额超过 500 的需要总监审批，这个规则从什么时候开始的？"

这时系统不需要重新全量搜索，而是在已有结果上 **drill-down**：
- 从业务规则树往下钻，找到"退款金额阈值"的变更历史
- 关联到代码层，看有没有硬编码 500 这个数字
- 关联到经验层，看上次调这个阈值的时候出过什么问题

这就是 OpenHuman 的 6 把钥匙的设计思路 — 不是一把搜完，而是渐进式精化。

## 4. 评测能力映射

10 种记忆评测能力在设计阶段的优先级：

| 评测能力 | 设计阶段需要吗 | 具体场景 | 优先级 |
|----------|-------------|---------|--------|
| Information Extraction | 必须 | 从 PRD、制度文档中提取业务规则 | **P0** |
| Temporal Reasoning | 必须 | "当前的审批规则是什么？""什么时候改的？" | **P0** |
| Knowledge Update | 必须 | 制度变了，系统要跟上 | **P0** |
| Multi-hop Reasoning | 必须 | "退款规则 → 审批人 → 审批人当前角色"需要跳三跳 | **P0** |
| Multi-Session Reasoning | 必须 | 上次设计讨论的结论要能跨会话召回 | **P0** |
| Contradiction Resolution | 必须 | PRD 说 A，但代码里是 B，哪个是对的？ | P1 |
| Preference Following | 有用 | 设计师偏好的命名风格、分层方式 | P2 |
| Abstention | 有用 | 不确定的要说不确定，不能瞎编 | P2 |
| Summarization | 有用 | "帮我总结一下这个模块的设计现状" | P2 |
| Event Ordering | 有用 | "这个需求从提出到上线经历了哪些变更" | P3 |

**设计阶段的 5 个 P0 能力**：信息提取、时态推理、知识更新、多跳推理、多会话推理。

## 5. AgentCenter 应该怎么建设计阶段评测集

业界现有 benchmark 全是"个人助手"场景（聊天记录里夹着偏好、事实、指令）。**没有一个是面向企业设计场景的。**

应该基于 BEAM 的框架，建一套设计阶段专用评测集：

| 场景 | 输入（设计师问） | 期望输出 | 考察能力 |
|------|---------------|---------|---------|
| 业务规则查询 | "超过 50 万的采购单审批流程是什么？" | 当前生效的审批链 + 规则来源 + 生效日期 | TR + KU |
| 代码现状了解 | "订单模块的核心类有哪些？谁最近改过？" | 类结构树 + 最近 3 次 commit 信息 | IE + MR |
| 历史经验调用 | "上次对接支付网关选了哪家？为什么？" | ADR 内容 + 选型理由 + 踩坑记录 | MSR + CR |
| 跨层关联 | "退款接口涉及哪些业务规则？代码实现了吗？" | 规则列表 + 对应代码位置 + 未实现的差距 | MR + IE |
| 矛盾检测 | "PRD 说是异步，但代码是同步调用，哪个对？" | 矛盾点 + 两边依据 + 推荐解决方案 | CR + ABS |

每个场景 50-100 个测试用例，总共 250-500 个。

## 6. 核心结论

1. **三类问题**（业务规则、代码现状、历史经验）需要**三层记忆**分别服务
2. **检索策略**比存储设计重要 — 三层并行 + cross-encoder 重排 + reflect 推理
3. **渐进式精化** — 不是一把搜完，而是 drill-down
4. **5 个 P0 能力**：信息提取、时态推理、知识更新、多跳推理、多会话推理
5. **业界没有设计阶段专用评测集** — 这是 AgentCenter 可以建立的差异化评测资产
