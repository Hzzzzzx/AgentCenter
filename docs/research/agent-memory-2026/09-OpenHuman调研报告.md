# OpenHuman 调研报告

> AgentCenter 记忆系统调研 · 开源项目 #9
> 调研日期：2026-05-23

## 项目概览

OpenHuman 是一个于 2026 年 2 月创建的 local-first 个人 AI 超级智能系统，在 GitHub（tinyhumansai/openhuman）上已获得 25,884 星。该项目采用 Rust 作为核心语言，TypeScript/React 构建前端，通过 Tauri v2 实现桌面应用，遵循 GNU GPL3 开源协议，当前版本为 0.54.10。

与本系列之前调研的所有记忆框架不同，OpenHuman 的核心创新在于其 **Memory Tree** 架构——一种确定性的、基于 bucket-seal 的层级摘要树，而非业界常见的向量数据库方案。这种设计选择背后的理念值得深入探讨。

### Memory Tree 核心设计

Memory Tree 是 OpenHuman 区别于其他所有记忆框架的核心创新。传统记忆系统多基于向量数据库，通过 embedding 的余弦相似度进行语义检索。这种方案成熟、简单，但存在几个根本性问题：

1. **检索结果无结构**：返回的是平列表决，缺乏信息间的层级关系
2. **检索不可解释**：无法回答"这条记忆为什么被召回"
3. **全局上下文丢失**：向量相似度是局部的，无法呈现记忆的演进脉络

OpenHuman 的 Memory Tree 用层级摘要树替代向量索引。每个节点包含两部分：原始 chunk 的摘要，以及指向子节点的引用指针。树的根节点是全局摘要，叶子节点是原始的最小记忆单元。这种结构天然地保留了记忆之间的包含关系和层级结构。

### Bucket-Seal 管线详解

OpenHuman 的数据处理管线采用了 **bucket-seal** 架构，这是一种批处理与即时写入相结合的折中方案：

1. **Source Adapters**：对接各类数据源（聊天、邮件、文档等），将异构数据统一为标准化格式
2. **Canonicalizer**：将所有输入规范化为 Markdown 格式，附加 provenance 元数据（来源、时间戳、原始格式等）
3. **Chunker**：按确定性的 chunk ID 进行分块，每个 chunk 不超过 3k token。ID 采用 SHA256 内容寻址，确保相同内容不会重复存储
4. **Content Store**：原子性地将每个 chunk 写入磁盘的 .md 文件
5. **Store**：通过 SQLite 持久化元数据，包括 chunk 之间的关联关系
6. **Score**：结合信号、embedding 和实体抽取计算每个 chunk 的 admission 分数
7. **Tree**：构建三种作用域的摘要树（source/topic/global）
8. **Retrieval**：对外暴露 6 种可被 LLM 调用的检索工具

这条管线的精妙之处在于：**确定性**。从输入到树结构，每一步都是可重现的，不依赖任何随机因素。

### 三种 Tree Scope

OpenHuman 维护了三套独立的 Memory Tree，分别服务于不同的检索场景：

**Source Tree** 每一个数据源（ Gmail label、Slack channel、文档目录等）对应一棵独立的源树。这棵树完整地保留了某个来源的所有记忆，适合需要限定来源的检索场景。例如："根据上周的邮件总结一下我收到的需求变更"。

**Topic Tree** 每个高频实体（人名、项目名、关键词等）对应一棵主题树。这些树采用懒加载策略，只有当某实体的热度达到阈值时才会创建或更新。主题树解决了跨源关联的问题——关于同一个项目"AgentCenter"的信息可能散落在邮件、聊天和文档中，主题树把它们汇聚到一起。

**Global Tree** 每天生成一次全局摘要树，提供整个记忆空间的俯瞰视角。这棵树的根节点是全量记忆的全局摘要，适合需要全局概览的检索任务，例如年度回顾或长期趋势分析。

三种树作用域不同，互补而非冗余，构成了完整的记忆空间视图。

### Chunk 生命周期状态机

每个 chunk 在系统中经历以下状态流转：

```
pending_extraction ─┬─→ admitted ─→ buffered ─→ sealed
                    │
                    └─→ dropped
```

**pending_extraction** 是 chunk 进入系统后的初始状态，等待评分引擎的决定。

**admitted** 表示 chunk 通过了 admission 评分，被接受进入记忆系统，但尚未整合到树结构中。

**buffered** 是 admission 和 seal 之间的过渡状态。chunk 被暂时保存在缓冲区，等待与同批次或同主题的其他 chunk 一起处理。

**sealed** 是 chunk 的最终稳定态，已被整合进 Memory Tree 的某个节点中，不会再被修改或重新评分。

**dropped** 表示 chunk 被评分引擎判定为低价值内容，直接丢弃，不进入记忆系统。

这种状态机的设计确保了：进入系统的记忆都是经过筛选的，不会无限膨胀；已被整合的记忆是稳定的，不会因为后续数据的变化而被意外修改。

### Admission 评分机制

Admission 评分采用三段式设计，将 LLM 调用次数降到最低：

| 分数区间 | 决策行为 | 是否调用 LLM |
|---------|---------|-------------|
| ≤ 0.15 | 直接丢弃 | 否 |
| 0.15 ~ 0.85 | 由 LLM extractor 决定 | 是 |
| ≥ 0.85 | 直接 admit | 否 |

分数低于 0.15 的 chunk 被认为是低价值的，直接丢弃，避免不必要的资源消耗。分数高于 0.85 的高价值 chunk 直接 admit，不需要 LLM 的额外判断。只有落在中间区间的 chunk 才需要 LLM extractor 进行细致评估，决定是否 admit。

这种三段式设计的巧妙之处在于：**极低分和极高分都不需要 LLM 调用**。在实践中，大部分 chunk 的分数会落在两端，只有少部分需要真正的 LLM 判断。这大大降低了 token 消耗，同时保证了核心记忆不会被遗漏。

### 检索系统（6种工具）

OpenHuman 向 LLM 暴露了 6 种检索工具，分别是：

1. **memory_tree_query_source**：在指定 Source Tree 中检索，适合限定来源的记忆查询
2. **memory_tree_query_global**：在 Global Tree 中检索，适合全局概览类查询
3. **memory_tree_query_topic**：在指定 Topic Tree 中检索，适合围绕特定主题的深入查询
4. **memory_tree_search_entities**：搜索所有 Topic Tree，查找与某个实体相关的记忆
5. **memory_tree_drill_down**：在已知节点的基础上向下钻取，获取子节点的详细内容
6. **memory_tree_fetch_leaves**：获取指定树路径下的所有叶子节点，即最原始的记忆 chunk

这 6 种工具覆盖了从全局到局部、从概览到细节的各种检索需求。LLM 可以根据具体查询选择合适的工具，甚至组合使用多个工具。

### STM Recall 短时记忆

除了持久化的 Memory Tree，OpenHuman 还维护了一个 **Short-Term Memory (STM)** 召回机制，处理最近会话中的短期记忆。STM 采用双路召回策略：

**Arm 1**：基于 FTS5（SQLite 的全文搜索）检索近期episodic entries。这里的"其他会话"是关键——来自其他会话的相关记忆可能对当前任务有帮助。例如，用户在项目 A 的会话中学到的背景知识，可能对正在进行的项目 B 有参考价值。

**Arm 2**：暴力法 cosine nearest-neighbour，在 segment embeddings 上做向量相似度检索。这条 arm 捕捉的是语义上相似但文本上不完全匹配的短期记忆。

STM 的存在说明 OpenHuman 认识到了短时记忆和长时记忆的不同需求，两者采用了完全不同的技术方案。

## 核心设计思想

### 为什么用 Tree 而不是向量数据库？

向量数据库是当前 agent 记忆系统的主流选择，成熟、简单、效果不错。但 OpenHuman 认为向量检索存在根本缺陷：检索结果是无结构的平列表决，无法呈现记忆之间的层级关系和包含关系。Tree 结构天然地编码了这些关系——每个节点知道自己的父节点和子节点，检索结果本身就是可解释的层级结构。代价是构建和维护树结构的复杂度更高。

### 确定性 Chunk ID 的意义

OpenHuman 采用 SHA256 内容寻址生成 chunk ID，相同内容必然得到相同 ID。这个设计的核心价值是**去重和可重现**。在传统向量系统中，相同内容可能被重复 embedding，浪费存储和计算资源。确定性 ID 确保了系统永远不会存储两份相同的内容。更重要的是，整个处理管线是确定性的——给定相同输入，必然得到相同的树结构，这对调试和审计非常有价值。

### 三段式 Admission 的巧妙设计

三段式 admission 评分机制看似简单，实际上是一个非常精妙的工程优化：将 LLM 调用从"每个 chunk 必调"优化为"只在必要时调用"。考虑到 LLM 调用的 token 消耗和延迟，这个优化带来的收益是显著的。同时，边界分数（≤0.15 和 ≥0.85）的直接判决也符合直觉——极端低分和高分的内容本就不需要细致评估。

### Local-first 的价值

OpenHuman 选择了 local-first 架构，所有数据存储在本地（~/.openhuman/），不依赖任何云服务。这个选择有几个层面的价值：**隐私**——用户数据永远不会离开本地设备；**可靠性**——不依赖第三方服务的可用性；**所有权**——用户真正拥有自己的数据。对于一个个人 AI 超级智能系统来说，local-first 是最符合"个人"这一核心定位的选择。

### Obsidian 集成的理念

OpenHuman 可以生成一个 Markdown wiki/ 目录，用户直接在 Obsidian 中浏览自己的记忆系统。这个集成的理念是：**记忆系统不仅是 AI 的知识库，也是人类可读的个人的知识库**。当用户想要回顾、探索甚至编辑自己的记忆时，不需要任何特殊工具，只需要一个 Obsidian。这种"AI 与人类共享同一份记忆"的理念在所有调研的框架中是独一无二的。

## 优势分析

OpenHuman 的优势首先体现在**架构 sophistication**上。Memory Tree 的设计在所有调研的框架中独树一帜，不是简单地对标某个现有方案，而是从问题本质出发设计了全新的数据结构。三种 tree scope 的划分、短时记忆的双路召回、Bucket-Seal 管线——这些设计组合在一起，构成了一套完整且自洽的记忆系统架构。

**Rust 实现**带来了显著的性能优势和内存安全性。在处理大量 chunk 的评分和树构建时，Rust 的零成本抽象和并行计算能力是 Python 框架难以匹敌的。

**Obsidian 集成**是另一个差异化亮点。其他框架的关注点都在如何让 AI 更高效地使用记忆，OpenHuman 同时关注了如何让人类也能方便地访问记忆。这种双视角的设计思考是难得的。

**Local-first + GPL3** 的组合虽然限制了商业应用，但确实保证了用户的隐私和数据主权。对于个人用户而言，这是一个重要的价值主张。

**118+ 数据源连接器**和**每 20 分钟自动抓取**的机制，使得系统能够持续地、广泛地收集个人信息，真正成为一个living的记忆系统。

## 局限与不足

**GPL3 许可证**是最大的限制因素。GNU GPL3 要求任何基于 OpenHuman 开发的衍生作品也必须开源并采用相同许可证。对于希望在 AgentCenter 这样的商业产品中借鉴其设计理念的团队来说，这个许可证几乎封死了直接复用的可能性。Apache 2.0 或 MIT 会是更友好的选择。

**Desktop-only 的定位**意味着 OpenHuman 不是一个可以嵌入其他系统的 library 或 service。它是一个完整的个人 AI 桌面应用，设计初衷就不是作为组件被集成的。如果 AgentCenter 想要借鉴 Memory Tree 的设计，需要从头实现，不能直接引用 OpenHuman 的代码。

**Rust 语言**虽然性能优异，但也提高了贡献门槛。大多数 AI/ML 工程师更熟悉 Python，Rust 的学习曲线和生态丰富度都不如 Python。这意味着社区贡献的活跃度可能受限。

**缺乏公开的 benchmark 数据**，无法与其他框架在性能、召回率、精确度等维度上进行量化对比。对于技术选型来说，这是一个信息缺口。

**无 temporal reasoning 能力**。系统虽然维护了时间维度的信息（每天的 global digest），但似乎没有基于时间轴的推理能力。如果用户问"我在这半年里对某个项目的看法有什么变化"，系统可能无法给出时序对比。

**早期 beta 状态**（2026 年 2 月创建，当前 0.54.10）意味着 API 和数据格式可能不稳定，生产环境使用存在风险。

## 与 AgentCenter 的契合度

OpenHuman 的 Memory Tree 设计对 AgentCenter 有较高的参考价值，但直接复用的可能性很低。

**设计理念的借鉴**：三种 tree scope（source/topic/global）的划分、短时记忆与长时记忆的分离、确定性处理管线——这些设计思想可以直接应用到 AgentCenter 的记忆系统设计中，不需要复制代码。

**Rust 实现的选择**：AgentCenter 的 Bridge 是 Java/Spring Boot，短期内没有切换到 Rust 的计划。如果要借鉴 OpenHigh 的架构思路，需要在 Java 生态中重新实现一套 Memory Tree。这不是不可能，但工程量不小。

**Obsidian 集成值得参考**：AgentCenter 的前端工作台可以借鉴"人类可浏览的记忆视图"这一理念，在 Web UI 中提供类似的体验。

**Local-first 不适合 AgentCenter**：AgentCenter 是企业 agent 平台，local-first 与其多用户、协作、SaaS 部署的场景不匹配。

综合来看，OpenHuman 的 Memory Tree 架构是一个值得深入研究的参考设计，AgentCenter 可以借鉴其设计思想而非代码实现。三种 tree scope 的概念、STM 的双路召回机制、确定性处理的理念——这些都可以在 Java 生态中重新实现。
