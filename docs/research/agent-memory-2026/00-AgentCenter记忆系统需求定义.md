# AgentCenter 记忆系统需求定义

> 基于 2026 年开源 Agent 记忆框架调研
> 调研日期：2026-05-23
> 状态：需求草稿，待确认

## 1. 项目背景与系统定位

### 1.1 AgentCenter 是什么

AgentCenter 是企业智能编排平台，通过对话驱动企业内部工具流程。产品定位如下：

**核心价值**：为企业提供统一的工作台和对话界面，编排覆盖软件全生命周期的工具流程，包括设计、开发、运维、调试等环节。

**目标用户**：企业内部的开发者、运维人员、产品经理等，需要跨系统协作完成软件工程任务的角色。

**当前技术架构**：

- **前端**：Vue 3 + Pinia + Vite，运行于端口 5173
- **Java Bridge**：Spring Boot 3.4.5 + MyBatis + Flyway，运行于端口 8080，负责业务编排、权限、审计
- **OpenCode Runtime**：headless AI 引擎，通过 `opencode serve` API 连接，运行于端口 4097
- **通信协议**：REST API + SSE（Server-Sent Events），M1 决策不采用 WebSocket
- **数据库**：当前使用 SQLite，规划迁移至 PostgreSQL

**数据流向**：Vue 前端 → Java Bridge → OpenCode Runtime，三者形成完整的请求处理链路。

**为什么需要记忆系统**：当前的 AgentCenter 在单次会话中可以处理用户请求，但无法跨会话保持上下文。当用户第二天回来询问"上次你说那个接口有问题，后来怎么解决的"时，Agent 无法回答。这是因为现有系统只保存了对话历史，没有保存真正的业务理解、代码上下文和项目知识。记忆系统要解决的核心问题是：让 Agent 拥有持续学习、跨会话累积、随时间追踪变化的能力。

### 1.2 当前系统已有的记忆能力

AgentCenter 并非从零开始，以下是已有的记忆相关能力：

**AgentSession / AgentMessage**：Bridge 层已经保存了对话历史，每次用户与 Agent 的交互都被记录在 `agent_session` 和 `agent_message` 表中。这是记忆系统可利用的基础设施，不需要重新建设。

**GitNexus 代码知识图谱**：项目已索引 11358 个符号、28205 个关系、300 条执行流。GitNexus 提供了代码实体之间的关联查询能力，包括符号定义、调用关系、继承关系等。这是 Layer 3（代码实时上下文）的良好基础，但目前缺乏与业务上下文、项目动态的联动。

**docs/architecture/ 文档体系**：项目已有架构文档结构，包括 AGENT-RUNTIME-BRIDGE-DEVELOPMENT-BLUEPRINT、OPENCODE-BRIDGE-TARGET-STATE、AI-NATIVE-DEVELOPMENT、VERIFICATION-FRAMEWORK 等。这些文档是 Layer 4（文档-代码关联）的原始材料。

**WorkItem / Workflow / Confirmation 实体**：已有的流程上下文数据，反映了项目的工作项状态、审批流程和确认请求。这些是 Layer 2（业务领域知识）的信息来源。

### 1.3 当前系统的记忆缺口

尽管已有上述能力，系统在记忆方面存在明显缺口：

**跨会话语义理解缺失**：AgentSession 保存了对话文本，但 Agent 无法理解对话背后的语义。例如用户说"那个接口"，Agent 不知道指的是哪个接口、哪个项目的接口。

**业务知识无累积**：系统不知道"当前的审批流程是什么"，也无法回答"去年这个时候的采购规则是什么"。业务规则变化没有版本化追踪。

**代码上下文断层**：GitNexus 提供了静态代码结构，但无法回答"AuthService 谁在改""这个接口被哪些地方调用"等实时工程信号问题。

**文档与代码无关联**：docs/architecture/ 中的文档与代码实体之间没有建立双向链接，无法从文档跳转到代码，也无法从代码跳转到文档。

**多模态资料孤立**：架构图、流程图、会议录音等资料没有与代码或业务实体关联，无法通过自然语言检索。

## 2. 记忆系统五大需求层

### 2.1 第一层：对话记忆（Conversation Memory）

**需求描述**：Agent 应记住用户偏好、跨会话的项目背景、过往对话中达成的结论和action items。用户不应重复解释背景，Agent 应能从历史对话中推断当前需求。

**典型用户场景**：

- "上次你说那个接口有问题，后来怎么解决的？"
- "我们上次讨论的 AuthService 重构方案，现在进展到哪了？"
- "我习惯用中文交流，请用中文回复"

**功能需求**：

- 用户偏好持久化：语言偏好、响应风格、常用工具配置
- 对话摘要提取：从长对话中提取关键决策、action items、悬而未决的问题
- 项目上下文追踪：用户当前在哪个项目、哪个迭代、涉及哪些模块
- 跨会话引用解析：用户说"那个接口"时，Agent 能根据上下文确定指的是哪个接口

**非功能需求**：

- 对话历史查询延迟 < 200ms（热门数据）
- 摘要生成在后台异步完成，不阻塞主流程
- 隐私保护：用户可查看和删除自己的对话记忆

**对应开源方案参考**：

Mem0 提供了最简洁的对话记忆 API 模型，两行代码即可完成记忆读写，但其存储后端不擅长处理时间维度。Letta 的 3-tier OS 模型中，memories 层专门处理对话记忆，但 Letta 是全栈运行时框架，太重。Hermes 的 MEMORY.md 机制使用文件存储，适合个人助理场景但不适合企业级的权限管理。

**在本系统中的定位**：第一层是基础层，必须优先实现。它依赖现有的 AgentSession 和 AgentMessage 表进行扩充，添加语义索引和偏好提取逻辑。

### 2.2 第二层：业务领域知识（Business Domain Knowledge）

**需求描述**：Agent 应理解企业的业务规则、领域模型、实体关系和状态机。关键挑战是业务知识随时间变化，Agent 需要能回答"当前规则是什么"和"某时间点的规则是什么"这类双时态问题。

**典型用户场景**：

- "当前的审批流程是什么？"
- "去年这个时候的采购规则是什么？"
- "供应商关系最近有变化吗？"
- "这个实体的状态机是怎么定义的？"

**功能需求**：

- 领域实体建模：支持实体类型、属性、关系、状态机定义
- 双时态数据模型：记录每个业务事实的有效时间（valid time）和事务时间（transaction time）
- 历史版本查询：支持"某时间点是什么状态"的查询
- 多源数据接入：支持从 ERP、CRM、采购系统等外部数据源同步业务数据
- 业务规则版本化：每次业务规则变更生成新版本，保留变更历史

**非功能需求**：

- 时间旅行查询的平均延迟 < 500ms
- 支持至少 3 年的历史数据存储
- 外部数据源变更检测：增量同步而非全量同步

**对应开源方案参考**：

Zep/Graphiti 在双时态数据模型方面最为成熟，其 Temporal Knowledge Graph 设计专门解决了"某时间点的事实是什么"这类问题。Cognee 提供了 30+ 数据连接器，适合多源数据接入，但缺乏时态建模能力。Hindsight 的 reflect 机制可应用于业务规则的自动总结。

**在本系统中的定位**：第二层是最高优先级之一。它需要与 WorkItem/Workflow 表打通，并将业务实体关系建模为知识图谱的节点和边。关键设计决策是双时态模型是否内建于核心存储，还是作为视图层实现。

### 2.3 第三层：代码实时上下文（Real-time Code Context）

**需求描述**：Agent 应理解代码库结构、调用链、最近变更和工程信号。代码不是静态的，每次 commit、PR review、deploy、incident 都是工程信号，Agent 需要追踪这些信号并关联到代码实体。

**典型用户场景**：

- "AuthService 谁在改？最近有什么变化？"
- "这个接口被哪些地方调用？调用链是什么？"
- "上次部署后有没有相关报警？"
- "这个 PR 改了什么？Review 通过了吗？"

**功能需求**：

- 代码实体关联：将 GitNexus 的静态代码结构与运行时调用链关联
- 工程信号捕获：支持接入 commit、PR、deploy、incident 等信号源
- 变更追踪：追踪每个代码实体最近谁改过、改了什么、为什么改
- 调用链可视化：支持从任意代码实体向上下游展开调用链
- 代码搜索：基于语义和结构的代码检索，不是简单的文本搜索

**非功能需求**：

- GitNexus 索引更新延迟 < 5 分钟（对新增 commit）
- 工程信号接入延迟 < 1 分钟（通过 webhook 或 polling）
- 调用链查询延迟 < 300ms

**对应开源方案参考**：

GitNexus 本身已经是代码知识图谱，其 11358 符号和 28205 关系是极好的基础。九大框架中没有任何一个覆盖了工程信号捕获这个领域——这是一个行业空白。Graphify 使用 tree-sitter 解析代码结构，但其设计目标是生成 Wiki 文档，不是实时追踪工程信号。

**在本系统中的定位**：第三层同样最高优先级别。需要基于 GitNexus 建设，补充工程信号接入能力。这是 AgentCenter 与其他框架差异化的关键领域——其他框架都是 per-user 或 per-agent 记忆，而企业需要的是跨项目、跨团队的代码上下文共享。

### 2.4 第四层：文档-代码关联（Doc-Code Linking）

**需求描述**：PRDs、HLDs、LLDs、API 文档应与代码实体双向关联。从文档能跳转到实现代码，从代码能查到来历文档。

**典型用户场景**：

- "这个 PRD 对应的代码在哪里？"
- "AuthService 是哪个设计文档定义的？"
- "这个 API 文档和实际接口实现一致吗？"
- "上次需求变更对应的代码改动是什么？"

**功能需求**：

- 文档解析：支持 Markdown、PDF、Word 等常见文档格式
- 实体提取：从文档中识别代码实体（类名、函数名、API 路径）、业务实体、流程步骤
- 双向链接：文档 → 代码实体，以及代码实体 → 文档的双向可查询关联
- 文档版本管理：每次文档更新生成新版本，保留历史版本
- 一致性检查：定期比对文档描述与实际代码实现是否一致

**非功能需求**：

- 新增文档的索引延迟 < 10 分钟
- 文档与代码一致性检查可配置周期，默认每日一次
- 支持至少 10000 篇文档的规模

**对应开源方案参考**：

Cognee 在文档 ingestion 方面最为成熟，提供了多种文档解析器和连接器。Graphify 使用 Claude Vision 解析图片中的图表信息，适合多模态文档。LangMem 提供了文档记忆的存储抽象，可与多种后端配合。

**在本系统中的定位**：第四层是第二优先级。docs/architecture/ 是已知的文档来源，需优先建立该目录与代码实体的关联。

### 2.5 第五层：多模态资料关联（Multimodal Asset Association）

**需求描述**：架构图、流程图、白板照片、截图、会议录音等资料应可搜索、可关联。用户能用自然语言找到相关资料，资料也能关联到代码或业务实体。

**典型用户场景**：

- "上周白板上画的那个架构图，对应的代码实现了吗？"
- "找一下包含这个流程的所有截图"
- "这个架构决策当时的会议录音在哪里？"
- "这个截图里的错误信息，关联的 incident 是哪个？"

**功能需求**：

- 多模态内容解析：支持图片（架构图、流程图、截图）、PDF、音频、视频
- 语义搜索：用户用自然语言描述，Agent 找到相关的多模态资料
- 实体关联：多模态资料可关联到代码实体、业务实体、文档
- 知识卡片生成：自动为图片生成描述，音频生成转录本
- 版本管理：资料更新后保留历史版本

**非功能需求**：

- 图片描述生成延迟 < 30 秒
- 音频转录延迟 < 5 分钟（取决于时长）
- 多模态检索的召回率目标 > 80%

**对应开源方案参考**：

Graphify 在多模态知识图谱方面最为突出，使用 Claude Vision 解析图片内容。OpenHuman 提供了本地优先的多模态处理能力。ppocrv5（已有技能）可用于中文文档的 OCR 识别。

**在本系统中的定位**：第五层是长期目标，短期内不做重点。当前阶段专注于前三层。

## 3. 跨层共性需求

### 3.1 统一知识模型

**核心问题**：五层记忆使用不同的数据模型，如何统一表示并互通？

**实体统一表示策略**：所有层的实体都应建模为知识图谱的节点。具体如下：

- **对话记忆层**：节点类型为 ConversationSummary、UserPreference、ActionItem、ProjectContext
- **业务领域知识层**：节点类型为 BusinessEntity、BusinessRule、StateMachine、Relationship
- **代码实时上下文层**：节点类型为 CodeSymbol、CallChain、Commit、PR、Deploy、Incident（复用 GitNexus 已有模型）
- **文档-代码关联层**：节点类型为 Document、DocSection、CodeReference
- **多模态资料关联层**：节点类型为 Image、Audio、Video、Transcript、KnowledgeCard

**跨层实体合并策略**：当同一实体在不同层被表达时，以代码实体（Layer 3）为锚点进行合并。例如用户提到"AuthService"，对话记忆中的引用、业务知识中的服务实体、代码中的类名都应汇聚到同一个实体节点。

**属性规范**：每个节点应包含 created_at、updated_at、valid_from、valid_to（时态）、source（数据来源）、confidence（置信度）等通用属性。

### 3.2 统一查询接口

**Agent 查询路由策略**：Agent 发起的记忆查询应通过统一入口，内部路由到对应层：

```
QueryRequest {
  query: string
  layers: Layer[]  // 可选，限定查询范围
  timeRange: TimeRange  // 可选，时间范围
  projectScope: Project[]  // 可选，项目范围
}
```

**查询路由规则**：

- 用户问题中包含时间限定词（如"去年""当前"）优先路由到 Layer 2
- 用户问题中包含代码实体（类名、函数名）优先路由到 Layer 3
- 用户问题中包含文档特征词（如"哪个 PRD""设计文档"）优先路由到 Layer 4
- 用户问题中包含视觉特征词（如"那张图""截图"）优先路由到 Layer 5
- 无特征时默认查询 Layer 1

**跨层联合查询**：支持跨层查询，例如"AuthService 相关的所有记忆""最近的变更和涉及的文档"。

### 3.3 存储架构

**Polyglot Persistence 策略**：不同层的数据适合不同的存储介质：

| 数据类型 | 推荐存储 | 选型考虑 |
|---------|---------|---------|
| 对话历史、摘要 | PostgreSQL | 已有基础设施，事务支持好 |
| 业务实体、关系 | PostgreSQL + Graph Extension 或 Neo4j | 关系查询 + 图遍历 |
| 代码实体、调用链 | GitNexus (已有) + PostgreSQL | 复用现有索引 |
| 向量嵌入 | Qdrant 或 Milvus | 企业内网部署友好 |
| 时态业务数据 | TimescaleDB 或 Neo4j | 时序查询优化 |
| 多模态资料 | 对象存储 (MinIO/S3) + PostgreSQL 元数据 | 大文件存储 |

**与现有 SQLite 的关系**：当前 SQLite 用于轻量场景（开发、演示）。生产环境应迁移到 PostgreSQL 以支持向量搜索和图查询扩展。

**向量库选型**：Qdrant 推荐理由——Rust 实现，性能好，支持过滤查询，企业内网友好。Milvus 可作为备选。

**图数据库选型**：Neo4j 推荐理由——成熟度高，Cypher 查询语言强大，适合业务实体关系建模。TimescaleDB 可作为时态数据的备选。

### 3.4 权限与隔离

**多项目记忆隔离**：不同项目的记忆默认隔离，项目成员只能访问项目内记忆。跨项目查询需要显式授权。

**用户数据权限**：

- 用户只能读取自己参与的项目的记忆
- 对话记忆受用户隐私设置控制
- 管理员可审计所有记忆访问

**审计追踪**：所有记忆的写入、读取、删除操作都应记录审计日志，包括操作者、时间、操作类型、目标实体。

## 4. 与现有系统的集成约束

### 4.1 Java Bridge 集成

**记忆服务在 DDD 架构中的位置**：记忆服务属于基础设施层（Infrastructure Layer），为应用服务层（Application Service Layer）提供记忆查询和写入能力。它不直接暴露给前端，而是通过现有的 Controller-Service 链路间接调用。

**API 设计原则**：

- RESTful 风格，与现有 Bridge API 一致
- 记忆查询结果通过 SSE 推送给前端（与现有事件流一致）
- 记忆写入操作支持异步处理，不阻塞主流程

**SSE 事件中的记忆注入**：当 Agent 生成响应时，Bridge 应自动注入相关记忆作为上下文。记忆注入的触发时机和内容由 Agent 的查询结果决定。

**DDD 分层约束**：记忆服务不涉及 WorkItem、Workflow 等核心域对象的业务逻辑，它只提供记忆的存储和检索能力。核心域对象的业务逻辑仍在各自的 Domain Service 中。

### 4.2 OpenCode Runtime 集成

**记忆如何传递给 Agent**：Bridge 通过 `prompt_async` API 将记忆作为上下文注入发送给 OpenCode。记忆内容经过筛选和压缩，不传递完整的历史记录。

**Skill / MCP 层面的集成**：未来考虑将记忆查询封装为 OpenCode Skill 或 MCP Tool，使 Agent 能主动查询记忆。当前的设计是 Bridge 被动提供记忆，即在发送请求时主动注入。

### 4.3 前端集成

**记忆状态可视化**：用户应能看到 Agent 正在使用哪些记忆，并可选择启用或禁用特定层的记忆。

**用户记忆管理**：用户应能查看自己对话记忆的历史、删除不需要的记忆、管理偏好设置。

**各框架适合解决我们哪层需求**：

- **Mem0**：参考其对话记忆 API 设计的简洁性
- **Letta**：参考其 memories 层的抽象，但需避免全栈运行时
- **Zep/Graphiti**：直接借鉴其双时态知识图谱设计
- **Cognee**：直接使用其文档解析和接入能力
- **Graphify**：参考其多模态解析和代码结构解析
- **Hindsight**：参考其 reflect 机制用于自动摘要

### 5.2 行业空白与差异化机会

**工程信号捕获**：九大框架无一覆盖 commit、PR review、deploy、incident 等工程信号的捕获和关联。这是一个巨大的行业空白，也是 AgentCenter 的差异化机会。企业需要知道"代码最近谁改的、部署后有没有问题"，这直接影响 Agent 的建议质量。

**共享企业记忆**：所有框架都是 per-user 或 per-agent 记忆模型，没有一个是为企业级共享记忆设计的。企业需要跨项目、跨团队的共享知识库，而不是每个人或每个 Agent 各自独立的记忆。

**Java-first 集成**：所有框架都是 Python-first，缺乏 Java 生态的集成方案。AgentCenter 需要从零设计 Java 端的记忆服务，无法直接复用现有框架的代码，但可以借鉴其设计思想。

### 5.3 可借鉴的设计模式

**从 Zep/Graphiti 借鉴**：双时态数据模型设计。即使时不记录"事实是什么"，同时记录"某时间点的事实是什么"。这对业务规则变化追踪至关重要。

**从 Cognee 借鉴**：数据 pipeline 的抽象。将数据接入、转换、存储、解码分离为独立步骤，支持灵活的数据源组合。

**从 Hindsight 借鉴**：4-strategy retrieval + reflect 机制。检索策略多样性（语义、关键词、实体、层次）和周期性反思提升记忆质量。

**从 Graphify 借鉴**：多模态内容的结构化解析。图片不只是存储，还提取结构化信息存入知识图谱。

**从 Mem0 借鉴**：API 设计的简洁性。两行代码完成记忆读写的用户体验是记忆服务 API 设计的标杆。

## 6. 非功能性需求

### 6.1 性能

- 对话记忆查询延迟 P95 < 200ms（热门数据），P99 < 500ms（冷数据）
- 业务知识时间旅行查询延迟 P95 < 500ms
- 代码调用链查询延迟 P95 < 300ms
- 记忆写入吞吐量 支持 100 并发写入/秒
- 向量搜索延迟 P95 < 100ms（10000 维向量，100000 条记录）

### 6.2 可靠性

- 记忆服务可用性目标 99.9%（全年停机时间 < 8.7 小时）
- 数据持久性要求 99.999%（每年数据丢失 < 0.0001%）
- 记忆服务支持 graceful degradation，单层服务故障不影响其他层

### 6.3 安全与合规

- 所有记忆数据加密存储（AES-256）
- 传输加密（TLS 1.3）
- 记忆访问遵循最小权限原则
- 支持 GDPR 等数据合规要求的导出和删除
- 审计日志保留至少 1 年

### 6.4 可观测性

- 所有记忆操作暴露 Prometheus metrics
- 结构化日志（JSON 格式），包含 correlation_id 以追踪请求链路
- 健康检查端点（/health）暴露记忆服务状态
- 支持分布式追踪（OpenTelemetry）

## 7. 优先级建议

### Phase 1 (MVP)

**目标**：建立对话记忆和基础代码上下文，实现从零到有的突破。

**范围**：

- Layer 1 对话记忆：基于现有 AgentSession/AgentMessage，添加语义索引和偏好提取
- Layer 3 部分能力：复用 GitNexus，补充最近变更追踪（基于 git log）
- 统一查询接口原型：单一入口，支持 Layer 1 和 Layer 3 的联合查询
- 存储：使用现有 PostgreSQL（迁移 SQLite 后），添加向量搜索扩展

**验收标准**：

- Agent 能记住用户上次会话中提到的项目上下文
- 用户问"上次那个问题解决了吗"，Agent 能从记忆中找到相关对话
- 代码实体查询能返回最近 10 条相关 commit

**时间预估**：8-12 周

### Phase 2

**目标**：深化业务知识建模，建立文档关联，实现工程信号接入。

**范围**：

- Layer 2 业务领域知识：双时态数据模型，支持业务规则版本化
- Layer 3 工程信号：接入 commit、PR、deploy、incident 信号源
- Layer 4 文档关联：基于 docs/architecture/ 建立与代码的双向链接
- Polyglot 存储落地：引入 Neo4j 或 TimescaleDB

**验收标准**：

- 能回答"当前审批流程是什么"和"去年这个时候的流程是什么"
- 代码实体能显示最近修改人、PR 状态、deploy 状态
- 文档能关联到代码实体，代码实体能关联到文档

**时间预估**：12-16 周

### Phase 3

**目标**：多模态资料关联，完善企业级特性。

**范围**：

- Layer 5 多模态资料：接入架构图、流程图、截图、会议录音
- 企业级权限：多项目隔离、角色权限、审计追踪
- 性能优化：缓存层、读写分离
- 多语言文档支持

**验收标准**：

- 用户能用自然语言找到相关截图或录音
- 不同项目记忆严格隔离，跨项目查询需授权
- 所有记忆操作可审计

**时间预估**：16-24 周

## 8. 开放问题

以下问题需要用户确认后继续推进：

1. **时态模型设计**：业务知识是否真的需要双时态模型（valid time + transaction time）？还是只需要记录历史变更即可？如果需要，是内置在核心存储还是作为视图层实现？

2. **向量库选型**：Qdrant 和 Milvus 都有企业级特性，哪个更符合现有运维能力？是否考虑云服务（如 Pinecone）还是全自托管？

3. **工程信号接入范围**：Phase 1 是否需要接入真实的工程信号（GitHub/GitLab API、CI/CD 系统、监控告警）？还是先基于 git log 模拟？

4. **记忆所有权**：记忆数据的最终控制权在谁手里？企业（组织）还是个人用户？是否允许用户导出或删除自己的记忆？

5. **与 OpenCode 的边界**：记忆服务是 Bridge 的子模块，还是独立部署的服务？记忆的序列化和传递给 OpenCode 的格式是什么？

6. **多租户策略**：当前架构是否需要支持多租户？如果需要，记忆的隔离策略是什么？
