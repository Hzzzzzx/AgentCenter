企业级 Agent 工作流目标态

先从目标态说清楚：AgentCenter 不应该只是一个“多轮聊天入口”，也不应该只做一个固定流程工具。它要成为企业任务的 Agent 控制平面，把业务对象、流程实例、Agent 自主执行、人机协作、治理和证据链放在同一个可运营系统里。

八个目标态特性

01业务对象驱动

入口是企业任务、项目、变更请求或工单；聊天只是交互界面，不是流程事实源。

02目标与验收契约

每个 workflow 和节点都有目标、输入输出、完成标准、质量门禁和停止条件。

03可编排但不僵化

支持顺序、并行、barrier、router、loop、retry、fallback、nested task，但不把每个判断都画死。

04Agent 节点自治

Agent 在节点内自主选工具、拆子任务、判断是否需要提问；中心服务负责边界和验收。

05人机协作原生

澄清、审批、权限、产物评审、阻塞都要能追踪，可由 Agent 或策略触发。

06状态与证据持久

实例、节点、事件、操作、产物、决策理由进入运行账本，支持审计、回放和交接。

07企业治理内建

权限、预算、沙箱、敏感操作确认、组织角色、审计策略必须在工作流层可见。

08可运营可恢复

运行视图、取消、重试、接管、恢复、健康检查和 SLA 是平台能力；压缩只是异常恢复场景之一。

推荐形态：中心运行器负责派活和推进，Agent 负责把节点里的具体事情做好，交互区承接问人和审批，运行账本留下可审计证据。 这不是现状描述，而是 AgentCenter 面向企业级 Agent 工作台应当收敛到的目标模型。

控制：任务实例、规则、运行账本 执行：OpenCode / Codex / Claude 等执行器 协作：提问、审批、评审、接管 治理：权限、预算、审计、沙箱 恢复：重试、取消、继续、存档点

Agent 时代工作流的要素

目标态之后，再看通用要素。Agent workflow 不再只是传统流程工具或 prompt chain；它是一套把确定性控制、非确定性推理、人机协作和企业治理组合起来的运行模型。

01定义层

自然语言可以生成草案，但最终要沉淀成可审查、可版本化、可复跑的目标、契约和结构定义。

02编排层

流程引擎决定阶段、依赖、并行、等待、回滚和推进条件；不要把控制权散落在聊天上下文里。

03Agent 自主层

Agent 负责节点内探索、工具选择、子任务拆分、完成声明和风险暴露；平台负责边界和验收。

04人机协作层

澄清、审批、权限、产物评审、人工接管都要有负责人、截止时间、恢复语义和界面展示。

05过程留痕层

存档点、运行账本、事件记录和产物来源，让系统能应对断线、失败、重试和上下文压缩等异常。

06治理与运营层

权限、预算、沙箱、审计、observe、质量门禁、取消和重跑能力，决定它能否成为企业中心服务。

先按两个问题对齐

下面每个方案都先回答两个白话问题：任务到底按什么被定义出来？真正跑起来时，是谁在记状态、派活、等人、判断下一步？

两问总览

界面只是入口，真正要比的是“任务规则”和“运行机制”。

CC 任务来自一份脚本或步骤清单，里面写谁先做、谁并行做、最后怎么合并。 跑起来像项目经理：主流程派多个后台 Agent 干活，收结果，再决定下一步。 适合大任务拆分和并行检查。
AgentWeave 任务来自我们写好的工作流脚本，也可以包装成类似 CC 的任务/子 Agent 入口。 运行时自己记进度、等输入、重试、恢复；OpenCode 只是被它叫去干节点里的活。 最像可落地的执行核。
Dify 任务来自一组固定节点和变量规则；界面只是把这些规则展示出来。 运行时按节点顺序传变量、走分支、调模型或工具，适合流程稳定的场景。 适合借鉴规则表达，不适合全照搬。
AC 1026 任务来自一个会话里的阶段约定，比如 PRD、HLD、LLD、验证一步步走。 跑起来靠同一条会话继续推进，体验顺，但状态容易跟着聊天上下文漂。 保留体验，补硬状态。
LangGraph 任务来自一张状态图：每个点做什么、往哪走、什么时候暂停都写清楚。 运行时每走一步都存档，可以暂停、恢复、重放，很适合长任务。 适合借鉴可恢复机制。
OpenAI SDK 任务来自应用代码里定义的 Agent、工具、交接规则和护栏。 跑起来是多个 Agent 接力：一个 Agent 判断要不要交给另一个，应用负责记业务状态。 适合借鉴接力和护栏。

六种路线的“一句话识别”

CC 先有步骤清单，再派多个 Agent 并行干活。
Dify 先把固定流程和变量规则定好，再按规则跑。
AC 1026 一条会话带着阶段往前走，顺手但容易漂。
LangGraph 每一步都记账，可暂停、恢复、重放。
OpenAI SDK 多个 Agent 在应用里接力，护栏管风险。
AgentWeave 我们自己的脚本运行时，能等人、恢复、观察。

看运行时的三条主轴

谁定任务 用户白话、固定规则、脚本、状态图都可以是入口 真正要沉淀成可复用的任务定义 聊天上下文只能做界面
谁往前推 中心运行时：派活、等人、重试、验收 Agent：节点内探索、选工具、做判断 固定流程只适合稳定路径
谁记过程 运行时要记节点、操作、产物、人的回复 Agent 的判断要变成证据，而不是只留在消息里 异常恢复：断线、重试、压缩

优缺点一眼看

这里用 Thinking Compass 的取舍矩阵来评价：先定企业级 AgentCenter 的目标尺子，再看每种方案实际带来的效果提升、状态可靠性、交互能力和实现成本。结论不按偏好站队，只按场景取舍。

评价尺子

企业级场景最怕两件事：任务跑着跑着没人知道真实状态，或者 Agent 做了很多事但没有证据可接管。所以下面的优缺点不只看“能不能跑”，还看“能不能运营”。

  • 效果提升：是否让 Agent 更少被上下文污染，能否并行、复核、减少单次判断失误。
  • 状态可靠：中间结果、节点状态、等待项、产物是否在外部可查，而不是只在聊天里。
  • 企业协作：运行中能不能问人、审批、接管、取消、重试、追责。
  • 落地成本：是否需要整套复杂引擎，还是可以用简单表结构和运行账本先撑起来。

CC 工作流的真实优势

CC 最值得借鉴的不是“能派很多 Agent”本身，而是把主流程从聊天上下文里拿出来：计划由脚本持有，中间结果进入脚本变量和运行记录，主会话只拿必要结论。这会明显降低上下文污染，让大任务更容易做出稳定结果。

中间结果不挤爆主上下文 子 Agent 的发现先留在脚本变量里，最终只把筛过的结论交给主会话。好处是主 Agent 不必背着全部日志继续推理。
并行和复核提升结果质量 同一问题可以让多个 Agent 从不同角度查，再交叉验证。效果提升来自“多路独立证据”，不是单纯增加轮数。
计划可读、可保存、可复跑 流程写成脚本后，团队能看见它到底怎么拆任务、怎么汇总，也能把稳定流程保存成命令反复用。
但它不是企业工作流底座 它更像强执行核。运行中任意问人、跨会话长期恢复、企业审批和主数据归属，还需要 AgentCenter 自己补。

关于“变量不进入上下文是否更好”：通常是更好，但前提是变量结构化、可追溯，最后汇总时只提取必要事实。如果只是把细节藏起来却没有证据链接，反而会让审计和接管更难。

企业级场景可以先做简单版吗？

可以，而且应该先做。企业级不等于一开始就上复杂工作流引擎。只要把“任务、节点、变量、事件、等待项”这几个硬状态抓住，很多常见流程就能跑起来；复杂图引擎可以等需求真的长出来再引入。

1. Work Item记录业务任务、负责人、目标、验收标准和当前状态。
2. Run Instance每次运行有独立实例，知道开始、暂停、失败、完成和取消。
3. Node Record每个节点有输入、输出、状态、错误、重试次数和执行器归属。
4. Variable & Artifact中间结果放变量和产物，不塞满聊天上下文；最终只给 Agent 必要摘要。
5. Interaction问人、审批、评审都变成等待项，绑定到具体节点，可以恢复和追踪。

最小可行结论：AgentCenter 不需要一开始支持任意复杂图。先支持顺序、并行派活、等待用户、重试、取消、恢复、节点验收和运行账本，就能覆盖大多数企业 Agent 任务；后续再把常见模式沉淀成模板或脚本。

维度 相似之处 关键差异 对 AgentCenter 的含义
任务定义 都要把用户意图变成某种可执行规则，否则 Agent 只能靠聊天记忆猜下一步。 CC 和 AgentWeave 偏脚本；Dify 偏固定节点规则；1026 偏会话阶段；LangGraph 偏状态图；OpenAI SDK 偏 Agent 能力和交接规则。 我们需要保存“任务规则”，但入口可以是自然语言、模板或代码生成。

边界场景 都会遇到人机输入、权限确认、失败重试、取消、部分成功、产物不合格。 CC 运行中任意问人较弱；Dify 动态探索较弱；1026 状态容易漂；LangGraph 引擎化成本高；OpenAI SDK 业务状态要自建。 Interaction Dock、运行账本、节点验收和恢复材料必须成为一等能力。
Claude Code / 脚本派活适合大任务拆成很多后台 Agent 并行处理。
定义用户要求跑工作流，或选择已有命令；Claude 生成一份可读脚本。
批准用户先看计划和脚本，再决定是否运行、保存、取消。
派活脚本在后台调度多个子 Agent，中间结果留在脚本里。
收束主会话最终拿到汇总结果，用户可以看进度、暂停、恢复或停止。

怎么定义、怎么跑

本质是“把计划写成脚本”。脚本负责循环、分支、并行和汇总，Agent 负责子任务里的具体工作。

遇到上下文压缩

中间结果不主要塞进主聊天，而是留在脚本变量和运行记录里，所以主会话变短不会直接弄丢进度。

边界场景

  • 运行中任意问用户不是强项，更适合阶段间确认。
  • 子 Agent 失败可以重启，已完成结果可复用。
  • 如果本地会话退出，跨会话恢复能力有限。
AgentWeave / 我们的脚本运行时适合验证 AgentCenter 自己的派活、等待、恢复、观察能力。
定义任务写成工作流脚本，也可以包装成任务/子 Agent 入口。
运行运行时读脚本，调 OpenCode 子会话或其他执行器。
等待需要用户输入时进入等待状态，保存问题和当前进度。
恢复收到回答后继续，失败时可取消、重试、恢复或观察运行。

怎么定义、怎么跑

外部看起来是“任务、子 Agent、进度、等待、结果”；内部用脚本运行时保存状态、存档点和子任务输出。

遇到上下文压缩

父会话只拿简短结果，详细过程留在运行时存储里；恢复时用保存的运行状态，而不是要求 Agent 记住所有细节。

边界场景

  • 支持等待用户、取消、重试、恢复和观察。
  • 子任务输出必须结构化，否则运行时可以判失败并重试。
  • 它是执行核，不应该直接拥有企业主数据。
Dify / 固定流程运行时适合规则稳定、输入输出清楚、可以一步步传变量的任务。
定义先配置节点、输入输出、变量、分支和工具调用。
触发一次输入启动一次流程，或由对话流按轮次触发。
传递运行时按节点顺序传变量，满足条件就走对应分支。
结束到达结束节点后返回结果，日志里保留每个节点的运行记录。

怎么定义、怎么跑

它不是“界面方案”，而是固定规则运行方案。界面只是把节点规则、变量流转和条件分支展示出来。

遇到上下文压缩

状态主要在节点输入输出和运行日志里，模型上下文变短影响较小；但跨轮长期记忆仍要靠会话变量或外部存储。

边界场景

  • 流程稳定时很清楚，动态探索时比较僵。
  • 人机输入通常要预先设计好等待点。
  • 复杂分支太多会变成难维护的规则网。
AgentCenter 1026 / 单会话节点适合体验连贯的阶段式协作，但状态不能只靠会话记忆。

推进Agent 在同一条会话里继续下一阶段，用户随时能插话。

怎么定义、怎么跑

用户感觉像一直在同一条对话里推进任务；每个阶段像一个节点，但节点状态必须外置保存。

遇到上下文压缩

这是最容易出问题的地方。压缩摘要不是事实源；必须从数据库恢复当前节点、已做操作、用户确认和产物。

边界场景

  • 直接继续会绕过恢复材料，导致状态漂移。
  • 用户插话要绑定到具体节点，否则不知道回答的是哪个问题。
  • 工具调用和产物必须进账本，不能只在聊天里出现。
LangGraph / 可恢复状态机适合长任务、人工确认、失败恢复和重放调试。
定义任务变成状态图：每个节点做什么、条件满足后往哪走。
执行每一步读取状态，写回结果，再进入下一步。
存档运行时按 thread 保存每一步状态。
恢复可以暂停、人工更新状态、从某个存档点继续或重放。

怎么定义、怎么跑

它把“跑到哪一步”变成明确状态，不靠 Agent 自己回忆。Agent 可以在节点里做判断，但跳转规则清楚。

遇到上下文压缩

每步都有存档点，压缩或失败后可以从上一个成功状态继续；这也是它对长任务最有价值的地方。

边界场景

  • 人工审批可以通过暂停和恢复处理。
  • 失败后可从上个成功点重跑。
  • 引入整套引擎会增加语言栈、调试和集成成本。
OpenAI Agents SDK / Agent 接力适合应用内多个专门 Agent 按能力接力处理任务。
定义应用代码定义多个 Agent、工具、输出要求、交接规则和护栏。
判断当前 Agent 判断自己能否处理，或是否要交给别的 Agent。
接力交接像一次工具调用，任务进入另一个专门 Agent。
护栏输入输出可被规则检查，风险命中时阻断或转人工。

怎么定义、怎么跑

它更像一组应用组件：Agent 负责专业能力，应用负责把业务任务、状态和界面串起来。

遇到上下文压缩

SDK 有会话和追踪能力，但企业工作流状态仍要应用层保存；否则只知道 Agent 做过什么，不知道业务流程走到哪。

边界场景

  • 适合按专业能力接力，不天然等于完整工作流引擎。
  • 护栏能拦风险，但待办、审批、产物生命周期要自己建。
  • 跨 Agent 交接要控制上下文，否则容易丢业务约束。

关键决策

下面是面向 AgentCenter 的收敛建议。

D1

定义方式

用户可以用白话描述流程，但保存下来时必须变成一份可审查的任务规则：有哪些步骤、每步要什么输入、产出什么、怎么判断完成、什么时候需要问人。

D2

状态事实源

中心数据库是唯一权威。截图、摘要、运行时自己的记忆都只能辅助恢复，不能当最终事实。

D3

完成标准

Agent 可以说“我做完了”，但中心服务要检查结构、证据、质量、风险和停止条件，确认后才推进下一步。

D4

交互策略

任务规则先说明哪些情况可以问人、哪些必须审批；Agent 自己判断何时提问，所有待处理问题都进入统一交互区。

D5

运行时边界

OpenCode、Codex、Claude 这些执行器只负责干活，不保存企业主数据；AgentCenter 负责派活、记账和验收。

1026 弯路复盘

1026 的价值不是“方案失败”，而是暴露了 Agent workflow 的关键故障模式。

Lesson 1

单会话连续性很诱人

工作项里连续推进多个节点，用户体验自然；但如果没有外部状态闭环，连续性会变成对模型记忆的依赖。

Lesson 2

continue 不能裸发

任何“继续跑”都必须带上当前节点、上一步做了什么、正在等谁、完成标准是什么。

Lesson 3

prompt 协议不等于账本

Agent 汇报状态有用，但工具调用、产物、用户确认和运行事件都要进入运行账本。

Lesson 4

UI 应读状态，不读聊天

首页卡片、节点进度、待确认和产物面板必须读中心状态，而不是解析聊天消息。

推荐架构

这是建议的目标态切片：先把中心服务的派活、记账、验收做稳，再逐步增加自然语言生成、并行评审、循环重试和多执行器。

读法：左边把自然语言变成可审查任务规则，中间中心服务负责派活和验收，右边执行器只干活，状态和界面都从数据库派生。

落地路线

不要一上来做完整流程编辑器，也不要直接把所有逻辑交给 Agent。先做最小闭环，再扩展表达力。

M1

状态闭环

统一所有“继续跑”的入口；每次工具调用和状态变化都记账；用 AgentWeave 验证派活、等待、恢复的执行核。

M2

固定任务规则

先支持一份可保存的任务规则，写清完成标准和什么时候需要问人，不急着做复杂流程编辑器。

M3

自然语言生成

让 Agent 从用户描述生成任务规则草稿；用户在审查页确认后保存，形成可复用工作流。

M4

高级控制节点

再加入并行、汇合、评审团、循环直到通过、重试兜底、子流程和多执行器。
