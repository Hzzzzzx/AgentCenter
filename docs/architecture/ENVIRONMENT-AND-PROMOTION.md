## 二、各层晋升策略

### 开发 → 测试

```
├── ✓ 代码扫描通过 (无高危/严重漏洞)
├── ✓ 单元测试覆盖率达标
├── ✓ 集成测试通过
└── 自动触发，无需审批
```

### 测试 → 预生产
```
├── ✓ 全量功能测试通过
├── ✓ 性能基线达标
├── ✓ 安全扫描无高危
├── ✓ 至少 1 名评审人确认
└── 需人工审批
```

### 预生产 → 生产
```
├── ✓ 预生产环境验证 >= 规定时长 (无异常)
├── ✓ 回滚方案已确认
├── ✓ 监控告警就绪
├── ✓ 变更窗口内 (如需)
├── ✓ 至少 2 名审批人 (含运维负责人)
└── 需多人审批 + 变更窗口
```

## 三、晋升流程

```
┌──────────────────────────────────────────────────────────────────────────┐
│  1. 开发者提交晋升申请                                                 │
│     │                                                                  │
│  2. 系统执行预检 (自动校验 PromotionPolicy 中定义的检查项)                   │
│     │                                                                  │
│  3. 预检通过 → 进入审批 (根据层级决定是否需要审批)                     │
│     │                                                                  │
│  4. 审批通过 → 创建 Deployment                                   │
│     │                                                                  │
│  5. 部署引擎执行 (Rolling/Blue-Green/Canary)                            │
│     │                                                                  │
│  6. 部署完成后 → 触发验证任务                             │
│     │                                                                  │
│  7. 验证通过 → 更新环境状态 → 通知相关人                           │
│     │                                                                  │
│  8. 验证失败 → 触发回滚 → 回到步骤 2                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

---

## 四、关键约束

| 约束 | 说明 |
|------|------|
| 晋升单向 | 只能从低 tier 到高 tier，不可反向 |
| 不可跨级 | 不允许跳级晋升 (如不可直接从开发到生产) |
| 制品不可变 | 晋升的是同一个制品，不允许修改 |
| 环境隔离 | 不同环境的配置/密钥完全隔离 |
| 审计追踪 | 每次晋升必须记录完整的审批链和验证结果 |
