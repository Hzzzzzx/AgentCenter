Abstract Model

第一版先把 AgentCenter 理解成“任务列表里的多个独立工作项空间”。

用户不是在理解目录，也不是在理解内部编号。用户先看到一个任务列表，列表里有多个工作项；每个工作项都有自己的独立作业空间。Agent 在某个工作项空间里按阶段推进，过程留在这个空间里，产物也沉淀回这个空间。

任务列表 承载多个工作项。
工作项 最小作业对象。
作业空间 每个工作项独立拥有。
工作流阶段 只在所属空间推进。
02 MVP Picture

MVP 图：一个需求列表，多个工作项，每个工作项一个独立作业空间。

第一版先讲清楚映射关系：任务列表不是作业空间，列表里的每个工作项才对应一个独立作业空间。用户选中某个工作项后，这个工作项的工作流阶段在它自己的作业空间里连续推进。

03 Experience Model

第一版用户应该理解什么。

MVP 的空间设计不是让用户选择目录，而是让用户知道“我正在处理列表里的某个工作项，这个工作项有自己的作业空间”。

用户看到

  • 一个需求列表，里面有多个工作项。
  • 点开某个工作项后，进入这个工作项自己的作业空间。
  • 一条持续会话，展示这个工作项每个阶段的进度。
  • 需要确认的关键决策。
  • 最终沉淀到这个工作项空间里的文档、变更、验证结果。

系统保证

  • 不同工作项的上下文不会互相混淆。
  • 同一工作项的上下文不会在阶段之间断掉。
  • 每个阶段产生的内容能追溯到所属工作项。
  • 产物属于当前工作项作业空间，而不是散落在对话外。
  • 后续可以把个人、团队、企业边界加进来。
04 Target Architecture

目标态：把同一套映射关系放进更大的边界里。

后续不是推翻 MVP，而是在它外面一层层套边界。最里面仍然是“任务列表包含多个工作项，每个工作项有独立作业空间”；外面再加用户视角、工作空间/项目边界、企业/租户边界。

MVP

列表到工作项空间

先跑通最核心的映射：列表包含工作项，工作项拥有独立作业空间。

下一层

加入用户视角

用户不改变工作项归属，只影响可见范围、个人上下文和操作身份。

再下一层

加入工作空间和租户

工作空间管理项目资料和共享规则，租户管理组织级隔离和审计。

05 Workflow Evolution

工作流拆分也要后置。

MVP 中，工作流阶段只是同一工作项作业空间里的过程。后续只有在并行、回滚、权限差异、长任务恢复变成真实需求时，才把阶段拆成更独立的运行空间。

第一版

阶段在同一空间

理解、设计、执行、验证都在同一个工作项作业空间里展开，用户不切空间。

增强版

阶段有独立证据

每个阶段有自己的过程记录和产物来源，但仍聚合给用户看。

完整态

阶段可独立运行

当需要并行、回滚或隔离时，阶段才拥有更独立的运行边界。
