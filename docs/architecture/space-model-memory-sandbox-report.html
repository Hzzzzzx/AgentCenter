Executive Verdict

空间模型按终局设计，能力实现按 MVP 分期。

AgentCenter 第一阶段不要做完整企业权限平台，但必须从第一天固定正确的空间语义：每次 Agent 工作都绑定用户、项目、任务和运行；所有沙盒、产物、记忆和召回都在这个身份边界内发生。

Recommended MVP
单租户默认工作区，多项目、多用户、每次运行独立沙盒。

MVP 支持 run、private、project 三类记忆可见性，并用 candidate -> confirmed -> archived / superseded 管住记忆生命周期。

1026 Branch Alignment

基于 1026 分支继续，不另起一套状态模型。

空间、沙盒和记忆设计应该接到 1026 的 workflow state、runtime_operation、artifact、confirmation 和 notification 闭环上。它是下层作用域和证据底座，不是替代 1026 的新主线。

1026 主线与空间/记忆落点
Bridge DB 权威状态 -> WorkflowStateSnapshotService -> RuntimeOperation(skill.run) -> Runtime Event Listener -> Workflow Engine -> Notification / Confirmation / Artifact Projection Space / Sandbox / Memory -> 为以上对象提供 user/project/run scope、文件边界和 evidence 规则
1026 现有对象 本设计承接方式 实施含义
workflow_node_instance 属于 Project + WorkItem + Run 下的执行状态 不新建另一套工作流状态源。
runtime_operation MVP 的 run 派发账本 skill.run 应带 sandbox 和 memory query context。
runtime_event raw evidence 只作为候选记忆来源，不直接进入长期记忆。
artifact source_ref 和报告证据 confirmed memory 必须能追溯到 artifact、message、event 或文档。
confirmation_request 确认、提升、审批动作真源 candidate memory 和 artifact promotion 都通过确认闭环。
Space Taxonomy

四类空间先分清，不要让一个 Workspace 承担所有含义。

协作空间、运行空间、记忆空间和治理空间分别回答不同问题。MVP 可以简单，但语义不能混。

Collaboration

协作空间

定义企业、团队、项目、任务和用户关系。它是业务逻辑边界，不是磁盘目录。

  • Tenant 企业边界
  • Workspace 团队或业务域
  • Project 第一隔离单元
Runtime

运行空间

定义 Agent 在哪里读写文件、跑命令和生成产物。所有路径必须由服务端解析。

  • 项目共享区
  • 用户项目区
  • 单次运行沙盒
Memory

记忆空间

定义记忆属于谁、谁能看、何时召回。它不是单个向量库或单个 collection。

  • run 工作记忆
  • private 私人记忆
  • project 项目记忆
Governance

治理空间

负责确认、淘汰、复审、审计和自我改进，避免记忆越多上下文越脏。

  • 候选到确认
  • 过期和替代
  • 召回审计
Target Skeleton

终局骨架现在固定，复杂能力以后再开。

第一期可以只有默认 tenant 和 workspace，但字段、路径、事件和记忆 scope 都要保留它们。

长期对象层级
Tenant -> Workspace -> Project -> WorkItem -> Run -> RunSandbox User -> UserProjectWorkspace -> UserPrivateMemory
1

Agent 每次工作必须绑定 user_id + project_id + work_item_id + run_id，不存在游离运行。

2

前端只传业务 ID，服务端通过 Resolver 解析路径、权限和运行上下文。

3

Agent 默认只写 RunSandbox，共享产物和项目记忆必须经过提升或确认。

4

外部记忆框架只能作为 backend adapter，不能拥有权限、生命周期和确认语义。

Evolution

MVP 到企业完整态：先把骨架跑通，再逐步增强治理。

这条路线避免两种风险：MVP 变成临时玩具，或者一开始就被完整企业权限系统拖死。

MVP

默认 tenant/workspace，多项目、多用户、每次 run 独立沙盒，三类记忆可见性。

V1 项目协作

加入项目角色、RecallTrace、用户反馈、复审队列和 supersede 关系。

V2 组织知识

加入 workspace memory、tenant memory、跨项目受控引用和保留策略。

V3 企业治理

加入 RBAC/ABAC、DLP、query-as-user、secret scope 和治理看板。

多后端编排

按记忆类型路由到 Cognee、Graphiti、pgvector、全文检索或内部事实库。

Sandbox Contract

沙盒是运行边界：默认隔离，显式提升。

OpenCode 或未来 Runtime 都不能直接拿前端路径执行，必须走统一的 SandboxResolver。

runtime-workspace/
  tenants/{tenant_id}/
    workspaces/{workspace_id}/
      projects/{project_id}/
        shared/
        users/{user_id}/
        runs/{run_id}/
适配要求 设计含义 MVP 策略
Server-side resolve 路径只由服务端解析，前端不传裸绝对路径。 project_id / work_item_id / run_id 入参。
Default write isolation Agent 默认只能写本次运行沙盒。 writable_roots = [run_root]。
Explicit promotion 进入用户空间或项目共享空间必须有确认事件。 生成 ArtifactCandidate，用户确认后提升。
Path containment 所有文件操作必须留在 runtime workspace 根下。 路径规范化、前缀检查、符号链接防逃逸。
Artifact snapshot run 结束后扫描新增或修改文件，变成证据。 记录 artifact、source_ref、run_id。
Cleanup policy 成功、失败、取消需要不同保留策略。 成功可清理，失败保留到 TTL，取消保留日志。
Memory Contract

记忆空间是 AgentCenter 自己拥有的合同，不是外部框架的 schema。

记忆后端负责索引和检索，AgentCenter 负责权限、生命周期、确认、淘汰、审计和召回解释。

run

Run Working Memory

当前 run 的临时发现、草稿、工具结果。默认只影响本次运行，结束后可提炼为候选记忆。

private

User Private Memory

用户偏好、沟通习惯、个人工作上下文。可影响 Agent 协作方式，但不能泄露给其他用户。

project

Project Shared Memory

项目架构决策、约束、历史坑和共识。必须从 candidate 经确认后进入共享层。

对象 关键字段 为什么必须有
MemoryItem visibility、kind、status、source_ref 决定谁能看、是什么类型、是否可信、证据在哪里。
MemoryQueryContext tenant_id、workspace_id、project_id、user_id、run_id 召回前必须知道当前 Agent 是以谁的身份、在哪个项目工作。
MemoryHit backend_name、score、reason、access_decision 让前端和审计知道命中来自哪里，为什么命中，是否被过滤。
RecallTrace 命中记忆、过滤原因、注入上下文、使用结果 企业场景下必须能解释 Agent 看了什么、为什么用了这些上下文。
Memory Governance

没有淘汰和复审，记忆越多，回答越脏。

共享需要审批只是第一步。真正能长期运行的记忆系统还需要候选、确认、过期、替代、反馈和召回排序。

MVP 生命周期
raw_evidence -> run_working -> candidate -> confirmed -> archived -> superseded candidate -> rejected

召回排序不只看相似度

最终 ContextRanker 应综合相关性、权限匹配、来源质量、确认状态、新鲜度、使用反馈、过期状态和 superseded 状态。这样旧记忆仍可审计，但不会默认污染上下文。

治理规则 系统动作 价值
没有 source_ref 不能升级为 confirmed 防止无证据推测进入正式上下文。
candidate 超过 TTL 无人确认 自动 archived 控制候选记忆堆积。
新记忆覆盖旧记忆 旧记忆 superseded 保留历史，同时默认不召回旧结论。
用户反馈不准确 增加负反馈并进入复审 让系统学习哪些记忆会误导。
到达 review_due_at 进入项目复审队列 项目约束和规则不会永久僵化。
Harness Memory

Agent 也要记住怎么把自己做得更好，但不能自动全局生效。

Harness Memory 记录的是做事方法、工具经验、失败模式和协作偏好。它和项目事实、用户私人信息、企业规则必须隔离治理。

What it stores

自我改进记忆存什么

  • 什么任务适合先问澄清问题。
  • 什么报告结构用户更容易理解。
  • 哪些工具组合在某项目里有效。
  • 某个 Runtime Adapter 的常见失败模式。
  • 用户不认可 benchmark 时，先解释测试场景和期望召回。
Hard boundary

不能自动全局生效

  • 影响 prompt、skill、workflow 前必须有确认或评测证据。
  • 用户私人协作偏好不能泄露为项目共享 harness。
  • 项目级 harness 只在该项目或 workspace 下影响 Agent 行为。
  • Harness 记忆不是项目事实，也不是企业规范。
Decision Record

本轮应确认的设计决策

问题 建议决策
是否直接做最终完整版 否。最终骨架现在固定，复杂能力按 MVP、V1、V2、V3 演进。
MVP tenant/workspace 怎么处理 固定默认值，但所有核心字段、路径和查询上下文保留。
Agent 默认写哪里 只写 RunSandbox，进入用户或项目空间必须提升。
项目共享记忆怎么产生 必须从 candidate 经过确认进入，不能自动把推测写成团队共识。
外部记忆框架的位置 只能是 backend adapter，不能拥有 AgentCenter 的权限、生命周期和治理语义。
