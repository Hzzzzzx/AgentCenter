## 一、现状诊断

### 当前架构痛点

```
各模块独立建设 → 无统一对象模型 → 各系统"说不同语言"

具体表现:
❌ 跨系统关联困难 (工单 ↔ 代码变更 ↔ 构建 ↔ 部署)
❌ 环境 promotion 需要手动同步/转换
❌ 重复建设，能力无法复用
❌ 数据孤岛，无法形成完整视图
```

## 三、适配器职责定义

### 3.1 需求管理适配器

```yaml
能力:
  - 工单的增删改查 → 统一 WorkItem
  - 状态变更事件监听 → 事件总线
  - 迭代/里程碑信息同步
入向映射:
  外部工单状态 → WorkItem.status
  外部工单类型 → WorkItem.type
  外部优先级 → WorkItem.priority
出向映射:
  WorkItem.status → 外部工单状态
  WorkItem.comment → 外部评论
```

### 3.2 代码托管适配器

```yaml
能力:
  - 仓库/分支/提交信息 → 统一 CodeRepo
  - 合并请求事件监听 → 事件总线
  - 代码评审状态同步
入向映射:
  外部合并请求 → CodeChange
  外部提交 → Commit
  合并请求状态 → CodeChange.status
出向映射:
  CodeChange.reviewComment → 外部评审意见
  合并操作 → 外部合并请求合并
```

### 3.3 持续集成适配器

```yaml
能力:
  - 构建任务状态 → 统一 BuildJob
  - 制品元数据 → 统一 Artifact
  - 构建日志/结果同步
入向映射:
  外部构建任务 → BuildJob
  外部制品 → Artifact
  构建日志 → BuildJob.logs
出向映射:
  触发构建请求 → 外部构建系统
```

### 3.4 部署编排适配器

```yaml
能力:
  - 部署记录 → 统一 Deployment
  - 部署状态实时同步
  - 回滚操作转发
入向映射:
  外部部署记录 → Deployment
  部署状态 → Deployment.status
  部署阶段 → DeploymentStage
出向映射:
  Deployment指令 → 外部部署系统
  回滚指令 → 外部部署系统
```

### 3.5 容器编排适配器

```yaml
能力:
  - 工作负载状态 → Service 运行时信息
  - 资源使用率采集
  - 伸缩操作
入向映射:
  外部实例组 → Service.runtime
  资源指标 → ServiceEnvironmentStatus
出向映射:
  伸缩指令 → 外部容器平台
  重启指令 → 外部容器平台
```

### 3.6 指标监控适配器

```yaml
能力:
  - 系统指标采集 (CPU/内存/网络/磁盘)
  - 应用指标采集 (QPS/延迟/错误率)
  - 告警规则管理
入向映射:
  外部指标数据 → Metric
  外部告警 → Alert → Incident
出向映射:
  告警规则 → 外部告警系统
```

### 3.7 日志采集适配器

```yaml
能力:
  - 应用日志汇聚
  - 日志搜索/分析
  - 日志归档
入向映射:
  外部日志 → LogEntry
出向映射:
  (通常只读，无出向)
```

### 3.8 通知渠道适配器

```yaml
能力:
  - 消息发送 (即时通讯/邮件/短信)
  - 消息模板管理
  - 投递状态追踪
入向映射:
  外部消息状态 → Notification.status
出向映射:
  Notification → 外部通知渠道
```

## 四、事件驱动机制

### 统一事件格式

```json
{
  "eventId": "evt-20260401-001",
  "eventType": "deployment.status_changed",
  "source": "部署编排适配器",
  "timestamp": "2026-04-01T10:30:00Z",
  "correlationId": "flow-xxx",
  "payload": {
    "objectId": "DEP-001",
    "objectType": "Deployment",
    "change": {
      "field": "status",
      "oldValue": "IN_PROGRESS",
      "newValue": "SUCCESS"
    },
    "context": {
      "artifact": "ART-sha256xxx",
      "environment": "staging",
      "service": "user-service",
      "triggeredBy": "zhangsan@company.com"
    }
  }
}
```

### Topic 划分

| Topic | 用途 |
|-------|------|
| `workitem.events` | 工单相关事件 |
| `codechange.events` | 代码变更事件 |
| `artifact.events` | 制品相关事件 |
| `deployment.events` | 部署相关事件 |
| `environment.events` | 环境相关事件 |
| `incident.events` | 事件/告警 |

## 五、分阶段路线图

### Phase 1: 模型统一 & 数据打通 (8~12 周)

```
目标: 建立统一语言，连接存量系统

1. 定义统一对象模型
   • 8 个核心对象 + 3 个 AI 原生对象
   • ID 规范 + 外部引用映射
   • 关系定义

2. 建设核心服务
   • 对象注册中心 (CRUD + 查询)
   • 关系服务 (链路追踪)
   • 事件总线 (消息队列集群)
   • 适配器框架 (标准化接入协议)

3. 接入第一批适配器
   • 需求管理适配器 (工单 ↔ WorkItem)
   • 代码托管适配器 (仓库 ↔ CodeRepo)
   • 持续集成适配器 (构建 ↔ BuildJob/Artifact)

4. 存量数据初始化
   • 活跃工单导入
   • 代码仓库注册
   • 近期制品录入

验收标准:
✓ 统一模型可 CRUD
✓ 存量系统数据实时同步
✓ 工单可关联到代码变更
```

### Phase 2: 环境管理 & 晋升闭环 (8~12 周)

```
目标: 打通环境流转，实现端到端链路

1. 环境服务
   • 环境注册 (dev/test/staging/production)
   • 环境状态聚合
   • 服务 × 环境矩阵视图

2. 晋升引擎
   • 晋升策略 DSL
   • 质量门禁校验
   • 审批工作流
   • 变更窗口管理

3. 接入第二批适配器
   • 部署编排适配器
   • 容器编排适配器
   • 制品仓库适配器

4. 链路追踪增强
   • WorkItem → CodeChange → Artifact → Deployment 全链路
   • 反向追溯: Deployment → Artifact → CodeChange → WorkItem

验收标准:
✓ 环境间晋升可执行
✓ 质量门禁自动校验
✓ 审批流程在线化
✓ 端到端链路可视化
```

### Phase 3: 可观测 & 智能告警 (6~8 周)

```
目标: 统一监控，主动告警

1. 接入观测类适配器
   • 指标监控适配器
   • 日志采集适配器
   • 链路追踪适配器

2. 统一告警
   • 告警聚合/去重/分级
   • 告警 ↔ Service ↔ Deployment 关联
   • 告警 → Incident 转化

3. 气泡系统
   • 气泡引擎 (数据聚合 + 实时推送)
   • 内置气泡模板 (需求/构建/部署/监控)
   • 自定义气泡能力

验收标准:
✓ 监控数据统一展示
✓ 告警关联到具体部署/服务
✓ 气泡实时展示关键指标
```

### Phase 4: AI 对话 & 智能化 (10~16 周)

```
目标: AI 能力注入，对话式交互

1. 对话引擎
   • 意图识别 + 实体提取
   • 上下文管理 (多轮对话)
   • 对话 ↔ 统一对象模型映射

2. Agent 编排
   • 自然语言 → 流程定义
   • 多 Agent 协作调度
   • 执行状态实时反馈

3. 智能运维
   • 异常自动检测
   • 根因分析
   • 自动修复/修复建议

4. 云端 IDE (可选)
   • Web 编辑器集成
   • 与平台对象模型打通
   • AI 代码辅助

验收标准:
✓ 对话可执行 DevOps 全流程操作
✓ 晋升/部署可通过对话完成
✓ 故障可自动诊断并给出修复建议
```

## 六、整合原则

| 原则 | 说明 |
|------|------|
| 能力抽象 | 用功能能力名称代替具体产品名 |
| 模型统一 | 核心对象共用统一模型，全局唯一 ID |
| 适配隔离 | 存量系统通过适配器接入，隔离差异性 |
| 事件驱动 | 异步解耦，适配器将外部事件转换为统一事件格式 |
| 环境强隔离 | 多层环境严格分离，晋升单向不可逆 |
| 渐进迁移 | 不推翻存量系统，先接入后逐步替代 |
| AI 叠加 | 在统一模型基础上叠加 AI 能力 |
