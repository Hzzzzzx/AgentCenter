> 本文件为 SDD (Specification-Driven Development) 框架的全局 Agent 行为合同。
> 任何项目均可将本文件内容复制到其根目录 `AGENTS.md` 作为起点。
> 标有 `<!-- 项目级配置 -->` 的部分需根据具体项目补充。

## 1. 核心流程（8 步）

```
需求澄清 → 方案对比 → 设计审批 → 计划 → 执行 → 审查 → CI/CD → 归档
                                  ↑
                          【Design Gate 硬性门控】
```

| 阶段 | 说明 |
|------|------|
| 需求澄清 | 澄清真正的需求边界和约束条件 |
| 方案对比 | 对比多种实现方案的优劣 |
| 设计审批 | 重大功能需人工签署设计文档后方可执行 |
| 计划 | 创建结构化任务计划 |
| 执行 | 按计划实施开发 |
| 审查 | 代码审查与质量检查 |
| CI/CD | 自动化测试与构建 |
| 归档 | 文档归档与经验沉淀 |

**关键机制：**
- **Design Gate**：计划阶段设置硬性门控，引用 `task-planner` skill 进行设计审批
- **CI/CD 触发**：审查通过后自动触发，引用 `cicd` skill

## 2. Constitution 原则（4 条强制）

| 原则 | 说明 |
|------|------|
| **Test-First** | 先写测试，失败后再写实现 |
| **Simplicity** | 优先简单方案，避免过度设计 |
| **Integration-First** | 优先真实环境测试 |
| **Lint-First** | 写完代码后立即运行 lint |

## 3. 分场景流程

不同复杂度的任务走不同仪式感，不搞一刀切：

| 级别 | 场景 | 核心要求 |
|:---:|------|---------|
| L0 | typo/琐碎 | 改完 → 安全扫描 → 审查卡片，5 分钟 |
| L1 | 小 bug | L0 + 回归测试 |
| L2 | 小功能 | brainstorming → L1 + TDD 单测 |
| L3 | 中型功能 | L2 + Plan + Blueprint 验证 |
| L4 | 大型功能 | L3 + Final Wave 四维验收 |
| 紧急 | 线上故障 | 先修，24h 内补审查卡片 |

> 详细分级矩阵请参考：[SDD 全流程分级矩阵](docs/reference/sdd-improvement-plan.md)

## 4. 行为约束（全局 Agent 必遵守）

| 约束 | 说明 |
|------|------|
| **[SCOPE]** | 只改 task scope 内文件，范围蔓延立刻报告 |
| **[READ]** | 改文件前先读，不了解代码就拒绝执行 |
| **[VERIFY]** | 改完跑 lint + 测试，失败不报告完成 |
| **[SIMPLE]** | 不引入新抽象/工具函数，除非 task 明确要求 |

**底线禁止项：**
- `as any`、`@ts-ignore`、空 catch 块
- 删除失败测试
- 提交 secrets

## 5. 需求唤起模式

收到功能请求时，**不要直接写代码**。按以下步骤：

```
1. [后退] 先搞清楚真正要解决什么问题
2. [追问] 2-3 个问题澄清边界：做什么？不做什么？有约束吗？
3. [分段] 把设计分成可消化的小段，每段确认后再往下
4. [总结] 用一句话 + 边界清单让用户确认
5. [不做] 用户确认前不写代码、不画架构、不给方案
```

**输出格式：**
```
> 我的理解: [一句话]
> 边界: ✅做 / ❌不做
> 需要你确认: [最多 1-2 个问题]
```

L0/L1 可跳过追问环节。L3/L4 追加：对话自动提炼 Plan + Blueprint Target，**必须人工签署后执行**。

## 6. 代码理解工具优先级

找代码时，按此优先级选择工具：

| 优先级 | 工具 | 适用场景 |
|:---:|------|---------|
| 1 | `semantic_search_nodes` | 语义搜索，了解函数/类所在位置 |
| 2 | `get_impact_radius` | 改函数前查影响面 |
| 3 | `query_graph` (callers_of) | 追溯调用链 |
| 4 | `glob` | 精确匹配文件名 |
| 5 | `grep` | 最后手段，搜索文本内容 |

**原则**：不要用 grep/read 去发现已经可以通过图工具一次拿到的信息。

## 8. 项目级状态管理

### `.sisyphus/` 目录结构

```
~/.sisyphus/
├── plans/           # 活跃计划
├── archive/         # 已完成归档
├── boulder.json     # 活跃计划追踪
└── improvements.md  # 改进建议
```

### 新会话启动检查

```bash
cat ~/.sisyphus/boulder.json
ls ~/.sisyphus/plans/
ls -lt ~/.sisyphus/archive/ | head 5
```

```json
{
  "summary": "做了什么（一句话）",
  "decisions": [{"what": "选了 X 而非 Y", "why": "一行理由"}],
  "changes": [{"file": "path", "desc": "改了什么"}],
  "verification": {"lint": "pass", "build": "pass", "tests": "n/n", "blueprint": "n deviations"}
}
```

L0/L1 简化版只需 summary + changes。
