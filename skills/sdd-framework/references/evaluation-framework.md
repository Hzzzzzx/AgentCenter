# SDD 开发体系评价框架 — 面向 AI 编程时代的全流程评估

> 创建日期：2026-05-03  
> 前提假设：大模型能力持续增强，AI 承担编码，人类承担需求确认 + 关键审查  
> 核心命题：六个 SDD 相关项目/体系在"需求澄清→任务执行→快速验证"上的真实表现

---

## 一、评价框架设计

### 1.1 核心理念

```
                   ┌─────────────────────────────┐
                   │     人：需求确认 + 关键审查     │
                   │      AI：编码 + 测试 + 自查    │
                   └──────────────┬──────────────┘
                                  │
    ┌─────────────────────────────┼─────────────────────────────┐
    │                             │                             │
    ▼                             ▼                             ▼
┌────────┐                  ┌──────────┐                ┌──────────┐
│ 需求澄清 │ ──────────────→ │  任务执行  │ ──────────────→ │   验证   │
│ (25%)  │                  │   (30%)   │                │  (20%)   │
└────────┘                  └──────────┘                └──────────┘
                                  │
                                  ▼
                          ┌──────────────┐
                          │  任务拆解      │
                          │   (25%)       │
                          └──────────────┘
```

### 1.2 四大评价阶段

| 阶段 | 权重 | 核心问题 |
|------|:---:|---------|
| **A. 需求澄清** | 20% | 人说不清想要什么，怎么聊透、定边界？ |
| **B. 任务拆解** | 20% | 大需求怎么拆成可靠的、细粒度的、可并行的子任务？ |
| **C. 任务执行** | 25% | 小改动能不能精准完成？大项目能不能持续保持上下文？ |
| **D. 验证回归** | 15% | 怎么快速、低成本地验证"改对了"？偏离设计怎么发现？ |
| **E. 人类审查友好度** | 20% | 人怎么快速判断 AI 做对了没有？信息不爆炸，一眼看懂 |

## 二、详细评分维度

### A. 需求澄清（25%，4 维度）

| 维度 | 权重 | 评分标准 |
|------|:---:|---------|
| **A1. 模糊需求处理** | 30% | 0=完全依赖人想清楚；5=能提问引导；10=多轮对话渐进式逼近完整 spec |
| **A2. 边界确认** | 25% | 0=无边界定义；5=有 scope 说明；10=有形式化的 scope 合同，越界自动告警 |
| **A3. 需求冻结机制** | 20% | 0=随时改，无代价；5=有"签字确认"环节；10=需求锁定后改动需走变更流程 |
| **A4. 非技术人友好** | 25% | 0=需要工程师才能理解和使用；5=产品经理可用；10=业务方能直接参与 |

| 维度 | 权重 | 评分标准 |
|------|:---:|---------|
| **B1. 拆解粒度** | 30% | 0=粗粒度，单任务可执行 >30 分钟；5=中等粒度，10-15 分钟；10=细粒度，2-5 分钟可完成 |
| **B2. 依赖管理** | 25% | 0=无依赖分析；5=能识别先后顺序；10=完整的 DAG 依赖图，自动检测循环 |
| **B3. 并行潜力** | 20% | 0=全串行；5=人工识别并行；10=自动识别并推荐并行执行 |
| **B4. 拆分可靠性** | 25% | 0=经常遗漏子任务；5=大部分覆盖；10=结构化保证覆盖（如蓝图合同验证） |

### C. 任务执行（30%，5 维度）

| 维度 | 权重 | 评分标准 |
|------|:---:|---------|
| **C1. 小改动精准度** | 25% | 0=常常顺手重构；5=大部分精准；10=严格"只改要改的，不碰其他代码" |
| **C2. 大项目上下文保持** | 25% | 0=改到一半忘记前面；5=有 compaction 但粗糙；10=结构化的上下文交棒，不丢进度 |
| **C3. Token/成本效率** | 20% | 0=所有任务用同级别模型；5=有模型选择建议；10=自动化模型路由 + 成本追踪 |
| **C4. 错误恢复** | 15% | 0=失败从头来；5=能从 breakpoint 继续；10=有 rollback + 断点恢复 + 自动重试 |
| **C5. 范围约束** | 15% | 0=Agent 随意扩展 scope；5=有 scope 文档但依赖自觉；10=自动检测 scope creep 并拦截 |

### D. 验证回归（20%，4 维度）

| 维度 | 权重 | 评分标准 |
|------|:---:|---------|
| **D1. 验证速度** | 30% | 0=需要全套 CI/CD 才能验证；5=有快速 lint/test；10=秒级增量验证 |
| **D2. 设计保真** | 30% | 0=无；5=人工对比 spec 和代码；10=**自动化检测代码是否偏离原始设计** |
| **D3. Bug 定位/回归** | 25% | 0=全靠人工 debug；5=有测试覆盖能缩小范围；10=有 mutation tracking/rollback 精确定位 |
| **D4. 验证成本** | 15% | 0=每次验证消耗大量 token；5=中等；10=几乎零额外成本 |

### 3.5 人类审查友好度 & 可视化（E，20%，6 维度）

**核心理念**：未来的 AI 编程范式是「人参与需求 + 人收尾验收」，中间全部自动。但自动不等于人可以盲信——需要一套机制让人**快速判断 AI 做对了没有**，而不是被大段文字淹没。

| 维度 | 权重 | 评分标准 |
|------|:---:|---------|
| **E1. 信息密度** | 20% | 0=每页有效信息极少；5=有一定密度；10=一句话结论+三句话摘要，附加详情可折叠 |
| **E2. 差异可视化** | 20% | 0=全量对比，人肉找变化；5=有 diff；10=**语义级差异高亮**（ADDED/MODIFIED/REMOVED 一眼可见） |
| **E3. 决策追溯** | 15% | 0=改了什么不知道为什么；5=有记录但分散；10=代码改动→task→plan→spec 逐级向上追溯 |
| **E4. 分层消费** | 15% | 0=要么全看要么不看；5=有摘要+详情；10=审查卡片→可视化→全量文档，按需深入 |
| **E5. 可视化输出** | 20% | 0=纯文字；5=有简单图表；10=前端自动出 mockup/wireframe，后端自动出 flowchart/diagram |
| **E6. 人参与度** | 10% | 0=全程需人紧盯；5=关键门控需人；10=**人只需需求澄清+最终验收，中间全自动** |

#### 3.5.2 E 维度分析

**OpenSpec 领先（6.7）**——天然的信息结构化优势：
- Delta Spec 的 `ADDED/MODIFIED/REMOVED` 格式是天然的语义差异可视化
- Schema 校验输出是结构化的
- 从 Change→Delta→Spec 的可追溯链清晰

**TianYuan（5.8）**——亮点与痛点并存：
- ✅ E2 差异可视化满分：Blueprint Graph diff report 精确定位
- ✅ E3 决策追溯完整：设计决策记录表提供人可读的"为什么"
- ❌ E1 信息密度低：Plan 文档是长文，扫读困难
- ❌ E4 分层消费弱：没有审查卡片机制，要么全看要么不看
- ❌ E5 可视化空白：没有前端 mockup、后端 flowchart 自动生成

**Chrys 信息密度最高（E1=9）**——不是因为工具好，是因为 Agent prompt 规定：
- "After completing work, skip everything obvious. Only mention what is surprising."
- "Lead with the answer. Do not narrate your search process."
- 把「不废话」写进了 Agent 的行为指令

**ECC 自动化程度最高（E6=9）**——48 agents + hooks 体系让中间流程几乎全自动。但信息爆炸问题最严重（763 个文档）。

**全项目共同的盲区：E5 可视化输出**——没有一个项目能在需求阶段自动生成前端 mockup 或后端 flowchart。这是 TianYuan 可以差异化突破的方向。

## 七、核心洞察

1. **没有单一赢家**。Superpowers 总分最高(6.5)，但执行和可视化都非最强。六个项目在不同阶段各有优势。

2. **Chrys 揭示了关键范式分裂**：流程驱动（定义环节）vs 行为驱动（定义 agent 怎么做）vs 信息驱动（让人能快速判断）。好的开发体系需要三者结合。

3. **TianYuan 的双轨优势独一无二**：Blueprint Graph（代码层）+ UI Control Plane（UI 层）的自动化验证，六个项目中只有 TianYuan 具备。

4. **审查友好度是所有项目的共同盲区，也是最容易被低估的维度**：
   - 所有项目的文档都偏长、偏文字，扫读困难
   - **可视化输出（E5）是最大的真空地带**——没人能在需求阶段自动生成前端 mockup 或后端 flowchart
   - Chrys 的信息密度（E1=9）揭示了一个简单而强大的方法：把"不废话"写进 Agent prompt

5. **小型改动的精准度（C1）只有 Chrys 真正做对了**——"Do exactly what was asked" + "Don't refactor" 应该是每个 Agent 的基础指令

6. **E5 可视化输出是 TianYuan 的下一个差异化突破点**：利用 UI Control Plane 的能力，在需求阶段自动生成低保真原型，让人类在进入编码前就能确认"这是我要的界面吗？"

### 8.4 TianYuan 成本优化方向

```
当前成本结构:
  启动: 中（Plan + Gates 人工）
  执行: 中（Agent 无路由，全部同等级）
  维护: 中（设计文档 + 决策记录表人工维护）
  
优化方向:
  启动 ↓ → 中小任务简化 Gates（Chrys 模式：直接开干 + review）
  执行 ↓ → 引入任务复杂度路由（GSD 模式：简单→轻量 agent）
  维护 → 保持（设计文档是长期资产，值得维护）
  
关键取舍:
  TianYuan 不能走 Chrys 的「零流程」路线
  ——因为 TianYuan 的核心价值就是设计保真
  但可以走「分层流程」：Quick 任务轻量，Large 任务完整
```
